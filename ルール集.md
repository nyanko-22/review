# ソフトウェアレビューガイド

## 1. レビューの種類

| レビュー | 目的 |
|---|---|
| 詳細設計レビュー | 要件・基本設計を満たし、実装・テストが迷いなく行える設計になっていることを確認する |
| 実装（コード）レビュー | 設計どおりに実装され、品質・保守性・性能・セキュリティを満たしていることを確認する |
| テストレビュー | 要件・設計を十分に検証できるテストとなっており、不具合を検出できることを確認する |

## 2. レビューごとの役割

| 項目 | 詳細設計レビュー | 実装（コード）レビュー | テストレビュー |
|---|---|---|---|
| **レビュアーの役割** | - レビュー観点に沿って設計品質を確認する<br>- 仕様漏れ・不整合・曖昧さを発見する<br>- 保守性・性能・セキュリティ・テスト容易性の観点から改善提案する | - コード品質を確認する<br>- 設計との整合性を確認する<br>- バグや性能問題、保守性の課題を発見する<br>- 実装方針の妥当性を確認する | - テストケースの網羅性を確認する<br>- 正常系・異常系・境界値の不足を確認する<br>- 期待結果の妥当性を確認する<br>- テストで品質を保証できるか確認する |
| **レビューイーの役割** | - 設計意図を説明する<br>- 設計根拠を示す<br>- 指摘内容を整理し修正する | - 実装方針を説明する<br>- コードの意図を説明する<br>- 指摘内容を修正する | - テスト設計の考え方を説明する<br>- テストケース作成根拠を説明する<br>- 不足しているテストを追加する |

## 3. やってはいけないこと

| 項目 | 詳細設計レビュー | 実装（コード）レビュー | テストレビュー |
|---|---|---|---|
| **レビュアー** | - 好みだけで設計変更を要求する<br>- 成果物ではなく作成者を評価する<br>- 根拠のない指摘をする<br>- 実装レベルの細部ばかり議論する | - コーディングスタイルだけを指摘する<br>- 設計を無視して実装方法だけを議論する<br>- 書き方の好みを押し付ける<br>- 人格を否定するような発言をする | - テスト実施結果だけを議論する<br>- テストケースを読まず承認する<br>- 実装者視点だけでレビュ
ーする<br>- 「動けばよい」という考えで網羅性を軽視する |
| **レビューイー** | - 指摘を感情的に否定する<br>- 設計意図を説明しない<br>- 曖昧なまま承認を求める | - コードだけ見てほしいと設計意図を説明しない<br>- 指摘を放置する<br>- 動作することだけを理由にする | - 正常系しか作成しない<br>- 根拠なくテストケースを省略する<br>- 指摘された不足を修正しない |

## 4. 各レビューの着眼点

| レビュー | 主な確認対象 | キーワード |
|---|---|---|
| 詳細設計レビュー | どう作るか | 設計品質、責務、例外処理、データ設計、保守性 |
| 実装レビュー | 設計どおりに作られているか | 可読性、保守性、性能、セキュリティ、バグ |
| テストレビュー | 品質を十分に確認できるか | 網羅性、正常系、異常系、境界値、期待結果 |

## 5. 共通ルール

### レビュアー

- 成果物をレビューし、作成者を評価しない。
- レビュー観点と根拠に基づいて指摘する。
- 問題の指摘だけでなく改善案も提案する。
- 品質向上を目的とした建設的な議論を行う。

### レビューイー

- 設計・実装・テストの意図を説明できるよう準備する。
- 指摘を品質向上の機会として受け止める。
- 指摘事項を適切に反映し、必要に応じて再レビューを依頼する。
- 修正内容を明確に共有する。
