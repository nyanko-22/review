# ソフトウェアレビュー標準

## 目的

本資料は、ソフトウェア開発におけるレビュー品質を標準化するための資料です。

対象とするレビューは以下の3つです。

- 詳細設計レビュー
- 実装レビュー
- テストレビュー

本資料は、以下の3部構成です。

1. レビューガイドライン
2. レビュー観点チェックリスト
3. レビュー記録テンプレート

---

# 1. レビューガイドライン

## 1.1 レビュー全体の目的

レビューは、成果物の品質をチームで高めるために実施するものです。個人を評価する場ではありません。

仕様書のミスは後工程での大きな手戻りにつながります。レビューは、要件漏れ、設計不備、実装ミス、テスト不足などを早期に発見し、こうした手戻りを防ぐための重要な活動です。一方で、時間がないことを理由に省略したり、逆に時間をかけ過ぎたりすると、形だけのレビューとなり本来の役割を果たせなくなるため注意が必要です。

レビューの目的はミスの検出だけでなく、以下のように多岐にわたります。

- 進捗状況の確認
- 成果物の改善、問題箇所の排除
- 関係者間の合意形成
- 設計思想のチーム間共有、認識合わせ
- 参加者の教育

例えば進捗管理の観点では、成果物が正しい方向に進んでいるか、どの程度進んでいるかをレビューを通じて把握できます。教育の観点では、自分の成果物を改善するだけでなく、他者の成果物を評価する過程で気づきや学びが得られ、チーム全体の成果物が洗練されていきます。

レビューの目的を理解しているかどうかは、レビューに対する意識やその効果を大きく左右します。

- 目的まとめ
「設計・コードの質を上げ、欠陥を減らすため」だけではなく、「チーム全員に同じ知識を共有させること、また設計・コーディングにおいて全員が守るべきガイドラインを確立すること」
そして「 レビューを楽しいものにすること」がおそらく最も有効。


コードレビューを楽しい場にするためにも、レビュアー／レビュイー が同じ方向性を向くことが重要である。同時に、人によってコメントなどの表現が揺れ、それにより認識齟齬があると楽しむ以前の問題であるため、守るべきルールも存在するはずである。このガイドラインは推奨する行動と、守るべきルールの両方を定義し、コードレビューをより有意義で学びが多く、生産性と品質を高める場とすることを目指す。

## 1.2 レビューのグラウンドルール

グラウンドルールは、「成果物の品質向上」と「建設的で円滑なコミュニケーション」を両立させるために、参加者全員が共有する基本マナーと前提条件です。レビュアー・レビューイーが同じ方向を向くために、以下を守ってください。

### 基本姿勢：レビューは共同作業である

- レビューはコミュニケーションであり、作成者を詰問する場ではない。より良いものをチーム一丸となって作るという意識を持つ。
- 目的は「問題を見つけること」ではなく「品質を上げること」と認識を合わせる。
- 「正しさ」にこだわりすぎず、お互いの意図を理解する対話に努める。意見を一方的に押し付けない。
- 常に友好的・建設的な態度を心がけ、辛辣な批判は避ける。

### 敬意と心理的安全性：伝え方への配慮

- コメント時は相手への敬意を忘れず、人格を否定しない。特にレビュアーは指摘を多く行う役割のため、上下関係を作らないやりとりが一層求められる。
- 指摘の前に、まず作成者の意図を確認する（例:「〜の意図は何ですか？」）。
- 指摘は一般化せず「私は〜だと感じた」とI(私)メッセージで伝え、改善のための提案をセットにする。言葉づかいや記号（プレフィックスなど）に配慮し、機械的・感情的な指示は避ける。
- 心理的安全性を保つため、良い点はオーバーなくらい褒め、ポジティブな絵文字（🚀✨💯）や適度なユーモアを取り入れる。特に新規参画者は、論理的・建設的な指摘であっても数が多いと精神的に辛くなりやすいため、辛い場面こそジョークで和ませる配慮をする。

  例:
  - 「このチューニングが入ると操作性がチート並に爆速になって、お客様も喜んでくれそうですね🐆💨」
  - 「さすがに年齢設定の上限が1000になっています。鶴は1000年生きると言いますが、ここは150までにしておきましょう🧝」

### 進め方のルール

- 事前に資料を配布し、スケジュールを調整する。レビュアーも忙しいため、時間を有効活用できるよう準備する（難しい場合はレビュー技法を活用して調整する）。
- レビュー中は問題点の指摘にとどめ、修正方法や修正内容までは議論しない。修正方法の議論は時間を要するため、司会者が毎回冒頭に宣言するとよい。
- 指摘内容は議事録に残し、いつでも再確認できるようにする。目の前の問題解決だけでなく、同じ問題を繰り返さない意識を持つ。
- 指摘された箇所はすべて対応する。対応要否を自己判断で決めない（自分では気づけなかった箇所のため）。

## 1.3 レビューごとの目的（完了の基本基準）

各レビューは、以下の状態に達していることを目的とし、これを完了の基本基準とする。

| レビュー | 達成すべき状態 |
|---|---|
| 詳細設計レビュー | 要件・基本設計を満たし、実装者が迷わず実装でき、テスト担当者がテストケース（機能テスト・結合テスト）を作成できる状態。 |
| 実装レビュー | 設計どおりに実装され、保守性・性能・セキュリティを含めて重大な問題がない状態。 |
| テストレビュー | 要件・設計を十分に確認でき、品質を説明できるテストケースがある状態（性能テストもできる範囲で行う）。 |

## 1.4 レビューの進め方と関連ドキュメント

### 課題：レビュー時間の肥大化

質の低い成果物をそのままチームレビューに出すと指摘が多くなり、以下のサイクルが繰り返されてレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

特に「ルールとして明確に決まっているのに守られていない」類の指摘は、チームで議論する価値がない。これらを提出前に提出者本人がつぶし、チームレビューを本当に議論すべき設計判断に集中させることで、レビュー全体を効率化する。

### 基本方針：観点を仕分けてからレビューする

各レビュー（詳細設計／コード／テスト）は、観点を以下の3つの軸で仕分けてから実施する。各軸の判断基準と仕分け結果は、後述の関連ドキュメントで定義する。

| 軸 | 問い（何を分けるか） | 分類 |
|---|---|---|
| 軸0: 確認主体（誰が確認するか） | 答えが一意か／本人だけで確認できるか | 事前自己チェック / チームレビュー |
| 軸1: 実施方式（どう確認するか） | リアルタイムの双方向対話が必要か | 同期（会議/モブ） / パスアラウンド（非同期回覧） |
| 軸2: 指摘の重大度（指摘をどう扱うか） | マージをブロックすべきか | Must / Should / Nits |

3軸は「誰が（WHO）→ どう（HOW）→ 指摘の重み」の順に重ねて使う。まず軸0で自己チェックかチームレビューかを分け、チームレビューになった項目に軸1（同期かパスアラウンドか）を、出てきた指摘に軸2（重大度）を適用する。

### 関連ドキュメント

レビューの詳細な観点・仕分け・進め方は、工程ごとに以下のドキュメントへ分けて定義している。本標準はその入口であり、実際のレビューでは該当工程のドキュメントを参照する。

| 工程 | 観点仕分け方針（何を自己チェック／チームレビューに分けるか） | レビューガイドライン（観点の詳細・完了基準・チェックリスト） |
|---|---|---|
| 詳細設計 | [詳細設計レビュー観点仕分け方針.md](詳細設計レビュー観点仕分け方針.md) | [詳細設計レビューガイドライン.md](詳細設計レビューガイドライン.md) |
| コード | [コードレビュー観点仕分け方針.md](コードレビュー観点仕分け方針.md) | [コードレビューガイドライン.md](コードレビューガイドライン.md) |
| テスト | [テストレビュー観点仕分け方針.md](テストレビュー観点仕分け方針.md) | [テストレビューガイドライン.md](テストレビューガイドライン.md) |

加えて、3工程に共通する横断的な分類軸（軸1: 同期／パスアラウンド、軸2: Must／Should／Nits）は次のドキュメントで定義する。

- [レビュー分類方針総覧.md](レビュー分類方針総覧.md): 軸1・軸2の判断基準と、チームで決めるべき論点（回覧人数・SLA・Must/Shouldの線引きなど）をまとめた討議のたたき台。

各ドキュメントの役割は次のとおり。

- **観点仕分け方針**: 「この観点は提出者が一人で確認できる（＝事前自己チェック）か、チームの判断が必要（＝チームレビュー）か」を観点ごとに仕分けた結果と、その理由（軸0）。
- **レビューガイドライン**: 各工程のレビュー観点（A〜E）の詳細、指摘の重要度分類、完了基準（Exit Criteria）、および「事前自己チェックリスト」「チームレビューチェックリスト」を収録した実務ドキュメント。レビュアー・レビューイーが同じ観点でレビューに臨むために用いる。
- **レビュー分類方針総覧**: 上記を貫く分類軸の全体像と、同期／パスアラウンド・重大度ラベルの運用方針（軸1・軸2）。

### レビューの進め方（標準フロー）

1. **提出前（レビューイー）**: 該当工程のガイドラインの「事前自己チェックリスト」で全項目を確認する。**全項目をクリアしていない成果物はチームレビューに提出しない。**
2. **実施方式の決定**: チームレビュー対象の項目を、総覧の軸1に従って振り分ける。双方向の議論や認識合わせが必要なものは同期（会議／モブ）、一方向のフィードバックで足りるものはパスアラウンド（非同期回覧）とする。
3. **チームレビュー**: ガイドラインの「チームレビューチェックリスト」をもとに、設計判断・トレードオフなど議論が必要な観点に集中する。
4. **指摘の付与**: レビュアーは各指摘に重大度（Must／Should／Nits。ガイドラインでは [必須]／[推奨]／[提案] として定義）を必ず明示する。重大度が明示されていない指摘への対応は不要とする。
5. **完了**: ガイドラインの完了基準（Must相当がすべて解消、Should相当が修正または見送り理由つき、承認記録あり）を満たした時点でレビュー完了とする。

> ツールや静的解析（Lint・型チェック・CI/CD）で自動検出できるものは、自己チェック項目にも含めず自動化を優先する。工程別の詳細な観点・チェックリストは上記の関連ドキュメントを参照する。

## 1.5 レビュアーの役割

### 共通の役割

- 成果物を客観的に確認する。
- レビュー観点に基づいて確認する。
- 問題点や抜け漏れを指摘する。
- 指摘の根拠を明確にする。
- 必要に応じて改善案を提示する。
- 品質向上を目的とした建設的な議論を行う。

### 詳細設計レビューにおける役割

- 要件・基本設計との整合性を確認する。
- 仕様漏れ、不整合、曖昧な記述を発見する。
- 処理フロー、例外処理、データ設計、責務分担を確認する。
- 実装者が迷わず実装できる粒度になっているか確認する。
- テストケースを作成できる程度に仕様が明確か確認する。

### 実装レビューにおける役割

- 詳細設計どおりに実装されているか確認する。
- コードの可読性、保守性、拡張性を確認する。
- バグ、性能問題、セキュリティ問題の可能性を確認する。
- 例外処理、ログ出力、トランザクション制御が適切か確認する。
- 不要な重複や複雑すぎる実装がないか確認する。

### テストレビューにおける役割

- テスト観点に抜け漏れがないか確認する。
- 正常系、異常系、境界値が網羅されているか確認する。
- 期待結果が明確で、判定可能な内容になっているか確認する。
- 要件・設計の重要な仕様がテストで確認できるか確認する。
- テストデータ、前提条件、確認手順が明確か確認する。

## 1.6 レビューイーの役割

### 共通の役割

- レビュー対象の成果物を事前に準備する。
- 設計・実装・テストの意図を説明する。
- 前提条件や制約事項を説明する。
- 指摘内容を正しく理解する。
- 指摘事項を修正し、修正内容を共有する。
- 必要に応じて再レビューを依頼する。

### 詳細設計レビューにおける役割

- 設計方針、設計意図、設計根拠を説明する。
- 要件・基本設計からどのように詳細設計へ落とし込んだか説明する。
- 未決事項、前提条件、制約事項を明確にする。
- 指摘された仕様漏れや曖昧な記述を修正する。

### 実装レビューにおける役割

- 実装方針やコードの意図を説明する。
- 設計との差分がある場合は理由を説明する。
- 工夫した点、懸念点、重点的に見てほしい点を共有する。
- 指摘されたコードを修正し、必要に応じてテスト結果を提示する。

### テストレビューにおける役割

- テスト設計の考え方を説明する。
- テストケース作成の根拠を説明する。
- テスト対象、前提条件、テストデータ、期待結果を説明する。
- 指摘された不足観点や不足ケースを追加する。

## 1.7 レビュアーがやってはいけないこと

### 共通

- 成果物ではなく作成者を評価する。
- 人格を否定する。
- 根拠のない指摘をする。
- 好みだけで変更を要求する。
- 代替案や理由を示さずに否定する。
- レビュー観点から外れた議論を続ける。
- 指摘の粒度が細かすぎて本質的な問題を見落とす。

### 詳細設計レビュー

- 実装方法の好みだけで設計変更を要求する。
- 要件や基本設計を確認せずに否定する。
- 曖昧な記述を見逃す。
- 例外処理や境界条件を確認しない。
- テスト可能性を考慮しない。

### 実装レビュー

- コーディングスタイルだけを指摘する。
- 設計との整合性を確認しない。
- 動作確認済みであることだけを理由に承認する。
- 性能、セキュリティ、保守性の観点を確認しない。
- 書き方の好みを押し付ける。

### テストレビュー

- テストケースを読まずに承認する。
- テスト実施結果だけを確認して、テスト内容を確認しない。
- 正常系だけで十分と判断する。
- 期待結果の曖昧さを見逃す。
- 「動けばよい」という考えで網羅性を軽視する。

## 1.8 レビューイーがやってはいけないこと

### 共通

- 指摘を感情的に否定する。
- 設計意図や実装意図を説明しない。
- 指摘を放置する。
- 修正内容を共有しない。
- 曖昧なまま承認を求める。
- 指摘の意図を理解しないまま表面的に修正する。

### 詳細設計レビュー

- 設計根拠を説明できない状態でレビューに出す。
- 未決事項を明記しない。
- 例外処理や境界条件を省略する。
- 実装者が判断すればよいという前提で曖昧に書く。

### 実装レビュー

- 動作することだけを理由に指摘を退ける。
- 設計との差分を説明しない。
- 複雑な処理の意図を説明しない。
- 指摘された修正に対して確認テストを行わない。

### テストレビュー

- 正常系しか用意しない。
- テストケースの根拠を説明しない。
- 期待結果を曖昧にする。
- テスト対象外の範囲を明記しない。
- 指摘された不足ケースを追加しない。

# 2. レビュー観点チェックリスト

## 2.1 詳細設計レビュー チェックリスト

### 要件・基本設計との整合性

- [ ] 要件定義の内容と矛盾していない。
- [ ] 基本設計の内容と矛盾していない。
- [ ] 画面仕様、API仕様、DB設計と整合している。
- [ ] 業務ルールが詳細設計に反映されている。
- [ ] 未決事項や前提条件が明記されている。

### 処理仕様

- [ ] 正常系の処理フローが明確である。
- [ ] 異常系の処理フローが明確である。
- [ ] 分岐条件が明確である。
- [ ] 入力値、出力値、戻り値が明確である。
- [ ] 外部システム連携の仕様が明確である。
- [ ] バッチ処理や非同期処理の開始条件、終了条件が明確である。

### 責務分担

- [ ] クラス、関数、モジュールの責務が明確である。
- [ ] 1つの部品に複数の責務が詰め込まれていない。
- [ ] 共通化すべき処理と個別処理が整理されている。
- [ ] 呼び出し関係が分かりやすい。
- [ ] 将来変更が想定される箇所が局所化されている。

### データ設計

- [ ] 使用するテーブル、カラムが明確である。
- [ ] 型、桁数、NULL可否、初期値が明確である。
- [ ] コード値やステータス値の意味が明確である。
- [ ] 登録、更新、削除、参照の条件が明確である。
- [ ] データの整合性を保つ仕組みが考慮されている。

### 例外処理・エラー処理

- [ ] 入力エラー時の処理が明確である。
- [ ] 外部システムエラー時の処理が明確である。
- [ ] DBエラー時の処理が明確である。
- [ ] タイムアウト時の処理が明確である。
- [ ] エラー時の戻り値、画面表示、ログ出力が明確である。
- [ ] リトライやロールバックの方針が明確である。

### 境界条件

- [ ] 0件の場合の動作が定義されている。
- [ ] NULL、空文字、未設定の場合の動作が定義されている。
- [ ] 最大件数、最大文字数、最大桁数の場合の動作が定義されている。

## 3.4 指摘分類の定義

回覧レビュー時のみでよいかな？対面のレビューでは指摘分類の意図は記載しなくても伝わる。
| 指摘分類 | 内容 |
|---|---|
| 仕様漏れ | 必要な仕様、条件、処理、テストケースが不足している。 |
| 不整合 | 要件、設計、実装、テスト間で内容が一致していない。 |
| 改善 | 品質、保守性、可読性、性能をより良くするための提案。 |
| 質問 | 内容確認や前提確認が必要な事項。 |

## 3.5 重要度の定義

| 重要度 | 内容 |
|---|---|
| 高 | リリース前に必ず修正が必要。要件未達、不具合、重大な品質問題につながる。 |
| 中 | 修正が望ましい。保守性、テスト性、運用性に影響する可能性がある。 |
| 低 | 軽微な改善。修正しなくても仕様を満たせる。 |
