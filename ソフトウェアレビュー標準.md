# ソフトウェアレビュー標準

## 目的

本資料は、ソフトウェア開発におけるレビュー品質を標準化するための資料です。

対象とするレビューは以下の3つです。

- 詳細設計レビュー
- コードレビュー
- テストレビュー

本資料は、レビュー全体に共通する**ガイドライン（§1）**を定義する入口資料です。
詳細な観点チェックリストとレビュー記録の様式は、肥大化を避けるため別資料に分離し、本資料からは参照します。

| 資料 | 役割 |
|---|---|
| 本資料（§1 レビューガイドライン） | レビューの目的・グラウンドルール・進め方・役割・重大度ラベルなど、全工程に共通する標準 |
| [レビュー観点チェックリスト](共通/レビュー観点チェックリスト.md) | 工程別（詳細設計／コード／テスト）の観点チェックリスト |
| [レビュー記録テンプレート](共通/レビュー記録テンプレート.md) | レビュー記録（議事）の様式 |

工程ごとの詳細な観点・仕分け・実務ガイドラインは、§1.4 の関連ドキュメントを参照してください。

# 1. レビューガイドライン

## 1.1 レビュー全体の目的

レビューは、成果物の品質をチームで高めるために実施するものです。個人を評価する場ではありません。

仕様書のミスは後工程での大きな手戻りにつながります。レビューは、要件漏れ、設計不備、実装ミス、テスト不足などを早期に発見し、こうした手戻りを防ぐための重要な活動です。一方で、時間がないことを理由に省略したり、逆に時間をかけ過ぎたりすると、形だけのレビューとなり本来の役割を果たせなくなるため注意が必要です。

レビューの目的はミスの検出だけでなく、以下のように多岐にわたります。

- 進捗状況の確認
- 成果物の改善、問題箇所の排除
- 関係者間の合意形成
- 設計思想のチーム間共有、認識合わせ
- 参加者の教育

例えば進捗管理の観点では、成果物が正しい方向に進んでいるか、どの程度進んでいるかをレビューを通じて把握できます。教育の観点では、自分の成果物を改善するだけでなく、他者の成果物を評価する過程で気づきや学びが得られ、チーム全体の成果物が洗練されていきます。

レビューの目的を理解しているかどうかは、レビューに対する意識やその効果を大きく左右します。

**目的のまとめ**: レビューの本質は「設計・コードの質を上げ、欠陥を減らすこと」だけではない。「チーム全員で同じ知識を共有し、全員が守るべきガイドラインを確立すること」、そして「レビューを楽しいものにすること」が最も有効である。

レビューを楽しい場にするには、レビュアー／レビューイーが同じ方向を向くことが重要となる。一方で、人によってコメントの表現が揺れ、認識齟齬が生じると楽しむ以前の問題になるため、守るべきルールも必要となる。本ガイドラインは「推奨する行動」と「守るべきルール」の両方を定義し、レビューをより有意義で学びが多く、生産性と品質を高める場とすることを目指す（参考文献は[末尾](#参考)）。

## 1.2 レビューのグラウンドルール

グラウンドルールは、「成果物の品質向上」と「建設的で円滑なコミュニケーション」を両立させるために、参加者全員が共有する基本マナーと前提条件です。レビュアー・レビューイーが同じ方向を向くために、以下を守ってください。

### 基本姿勢：レビューは共同作業である

- レビューはコミュニケーションであり、作成者を詰問する場ではない。より良いものをチーム一丸となって作るという意識を持つ。
- 目的は「問題を見つけること」ではなく「品質を上げること」と認識を合わせる。
- 「正しさ」にこだわりすぎず、お互いの意図を理解する対話に努める。意見を一方的に押し付けない。
- 常に友好的・建設的な態度を心がけ、辛辣な批判は避ける。

### 敬意と心理的安全性：伝え方への配慮

- コメント時は相手への敬意を忘れず、人格を否定しない。特にレビュアーは指摘を多く行う役割のため、上下関係を作らないやりとりが一層求められる。
- 指摘の前に、まず作成者の意図を確認する（例:「〜の意図は何ですか？」）。
- 指摘は一般化せず「私は〜だと感じた」とI(私)メッセージで伝え、改善のための提案をセットにする。言葉づかいや記号（プレフィックスなど）に配慮し、機械的・感情的な指示は避ける。
- 心理的安全性を保つため、良い点はオーバーなくらい褒め、ポジティブな絵文字（🚀✨💯）や適度なユーモアを取り入れる。特に新規参画者は、論理的・建設的な指摘であっても数が多いと精神的に辛くなりやすいため、辛い場面こそジョークで和ませる配慮をする。

  例:
  - 「このチューニングが入ると操作性がチート並に爆速になって、お客様も喜んでくれそうですね🐆💨」
  - 「さすがに年齢設定の上限が1000になっています。鶴は1000年生きると言いますが、ここは150までにしておきましょう🧝」

### 進め方のルール

- 事前に資料を配布し、スケジュールを調整する。レビュアーも忙しいため、時間を有効活用できるよう準備する（難しい場合はレビュー技法を活用して調整する）。
- レビュー中は問題点の指摘にとどめ、修正方法や修正内容までは議論しない。修正方法の議論は時間を要するため、司会者が毎回冒頭に宣言するとよい。
- 指摘内容は議事録に残し、いつでも再確認できるようにする。目の前の問題解決だけでなく、同じ問題を繰り返さない意識を持つ。指摘は必ず根拠とセットで記録する。
- 指摘された箇所はすべて対応する。対応要否を自己判断で決めない（自分では気づけなかった箇所のため）。
- その場で結論が出ない事項は保留事項として記録し、理由と対応時期を明確にする。

## 1.3 レビューごとの目的（完了の基本基準）

各レビューは、以下の状態に達していることを目的とし、これを完了の基本基準とする。あわせて、各レビューの主な確認対象と着眼点キーワードを示す。

| レビュー | 主な確認対象 | 達成すべき状態（完了の基本基準） | 着眼点キーワード |
|---|---|---|---|
| 詳細設計レビュー | どう作るか | 要件・基本設計を満たし、実装者が迷わず実装でき、テスト担当者がテストケース（機能テスト・結合テスト）を作成できる状態。 | 設計品質、責務、例外処理、データ設計、保守性 |
| コードレビュー | 設計どおりに作られているか | 設計どおりに実装され、保守性・性能・セキュリティを含めて重大な問題がない状態。 | 可読性、保守性、性能、セキュリティ、バグ |
| テストレビュー | 品質を十分に確認できるか | 要件・設計を十分に確認でき、品質を説明できるテストケースがある状態（性能テストもできる範囲で行う）。 | 網羅性、正常系、異常系、境界値、期待結果 |

## 1.4 レビューの進め方と関連ドキュメント

### 課題：レビュー時間の肥大化

質の低い成果物をそのままチームレビューに出すと指摘が多くなり、以下のサイクルが繰り返されてレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

特に「ルールとして明確に決まっているのに守られていない」類の指摘は、チームで議論する価値がない。これらを提出前に提出者本人がつぶし、チームレビューを本当に議論すべき設計判断に集中させることで、レビュー全体を効率化する。

### 基本方針：観点を仕分けてからレビューする

各レビュー（詳細設計／コード／テスト）は、観点を以下の3つの軸で仕分けてから実施する。各軸の判断基準と仕分け結果は、後述の関連ドキュメントで定義する。

| 軸 | 問い（何を分けるか） | 分類 |
|---|---|---|
| 軸0: 確認主体（誰が・何が確認するか） | 機械的に検出できるか／答えが一意か／本人確認や合意が要るか | ①ツール自動化 / ②生成AIレビュー / ③人間・事前自己チェック / ④人間・チームレビュー |
| 軸1: 実施方式（どう確認するか） | リアルタイムの双方向対話が必要か | 同期（会議/モブ） / パスアラウンド（非同期回覧） |
| 軸2: 指摘の重大度（指摘をどう扱うか） | マージをブロックすべきか | [必須] / [推奨] / [提案]（Must / Should / Nits） |

軸0は、確認の担い手をコストの低い順に4層へ分ける。**ねらいは「人間が見る項目（③＋④）を絞り、機械的・定型的な確認をツール（①）と生成AI（②）へ寄せること」** である。

```
観点1項目
 ├ ツールで決定論的に検出できる？             → ①ツール自動化（CI/静的解析）
 ├ 不可だが答えが概ね一意で成果物だけで判定可？ → ②生成AIレビュー（提出前の自動一次レビュー）
 ├ 一意寄りだがAIが不安定／本人確認が必須？     → ③人間・事前自己チェック
 └ 一意でない・トレードオフ・合意が要る？       → ④人間・チームレビュー（AIは論点出しの補助）
```

3軸は「誰が（WHO）→ どう（HOW）→ 指摘の重み」の順に重ねて使う。まず軸0で担い手（①〜④）を分け、④人間・チームレビューになった項目に軸1（同期かパスアラウンドか）を、出てきた指摘に軸2（重大度）を適用する。軸0の4層の判断基準と運用は [レビュー分類方針総覧](共通/レビュー分類方針総覧.md) と [生成AIレビュー活用方針](共通/生成AIレビュー活用方針.md) で定義する。

- [レビュー観点チェックリスト.md](共通/レビュー観点チェックリスト.md): 工程別（詳細設計／コード／テスト）の汎用観点チェックリスト。

各ドキュメントの役割は次のとおり。

- **観点仕分け方針**: 「この観点は提出者が一人で確認できる（＝事前自己チェック）か、チームの判断が必要（＝チームレビュー）か」を観点ごとに仕分けた結果と、その理由（軸0）。
- **レビューガイドライン**: 各工程のレビュー観点（A〜E）の詳細、指摘の重大度分類、完了基準（Exit Criteria）、および「事前自己チェックリスト」「チームレビューチェックリスト」を収録した実務ドキュメント。レビュアー・レビューイーが同じ観点でレビューに臨むために用いる。
- **レビュー分類方針総覧**: 上記を貫く分類軸の全体像と、同期／パスアラウンド・重大度ラベルの運用方針（軸1・軸2）。

### レビューの進め方（標準フロー）

1. **提出前（レビューイー）**: ①ツール自動化（Lint・型チェック・テスト・静的解析）を通したうえで、②生成AIレビュー（提出前の自動一次レビュー）と③本人の事前自己チェックにより、ルール違反・定型の指摘をつぶす。該当工程のガイドラインの「事前自己チェックリスト」で全項目を確認する。**これらをクリアしていない成果物はチームレビューに提出しない。**
2. **実施方式の決定**: ④人間・チームレビュー対象の項目を、総覧の軸1に従って振り分ける。双方向の議論や認識合わせが必要なものは同期（会議／モブ）、一方向のフィードバックで足りるものはパスアラウンド（非同期回覧）とする。
3. **チームレビュー**: ガイドラインの「チームレビューチェックリスト」をもとに、設計判断・トレードオフなど議論が必要な観点に集中する。生成AIは論点出し・設計書との差分指摘の補助として用い、最終判断は人間が行う。
4. **指摘の付与**: レビュアーは各指摘に重大度ラベル（[必須]／[推奨]／[提案]。定義は §1.7 を参照）を必ず明示する。重大度が明示されていない指摘への対応は不要とする。
5. **完了**: 完了基準（[必須] がすべて解消、[推奨] が修正または見送り理由つき、承認記録あり）を満たした時点でレビュー完了とする。工程固有の完了基準は各ガイドラインの「完了基準（Exit Criteria）」を参照する。

> 軸0の方針に従い、ツール（①）で自動検出できるものは自己チェック項目にも含めず自動化を優先し、生成AI（②）で一次レビューできるものは提出前に通す。人間（③④）は判断・合意の要る項目に集中する。生成AIレビューの運用は [生成AIレビュー活用方針](共通/生成AIレビュー活用方針.md) を、工程別の詳細な観点・チェックリストは上記の関連ドキュメントを参照する。

## 1.5 レビューにおける役割

### 共通の役割

**レビュアー**

- 成果物をレビューし、作成者を評価しない。
- レビュー観点と根拠に基づいて指摘する。
- 問題の指摘だけでなく改善案も提示する。
- 品質向上を目的とした建設的な議論を行う。

**レビューイー**

- レビュー対象の成果物を事前に準備・共有し、設計・実装・テストの意図や前提・制約を説明できるようにする。
- 重点的に確認してほしい箇所を事前に明記する。
- 指摘を品質向上の機会として正しく理解する。
- 指摘事項を修正して内容を共有し、必要に応じて再レビューを依頼する。

### 工程ごとの役割

| 役割 | 詳細設計レビュー | コードレビュー | テストレビュー |
|---|---|---|---|
| **レビュアー** | ・要件・基本設計との整合性を確認する<br>・仕様漏れ／不整合／曖昧な記述を発見する<br>・処理フロー・例外処理・データ設計・責務分担を確認する<br>・実装者が迷わない粒度か、テストケースを作成できる明確さかを確認する | ・詳細設計どおりに実装されているか確認する<br>・可読性・保守性・拡張性を確認する<br>・バグ・性能・セキュリティ問題の可能性を確認する<br>・例外処理・ログ・トランザクション制御、不要な重複や過度な複雑さを確認する | ・テスト観点の抜け漏れを確認する<br>・正常系／異常系／境界値の網羅性を確認する<br>・期待結果が明確で判定可能か確認する<br>・重要仕様をテストで確認できるか、テストデータ・前提・手順が明確かを確認する |
| **レビューイー** | ・設計方針・意図・根拠を説明する<br>・要件・基本設計からの落とし込みを説明する<br>・未決事項／前提／制約を明確にする<br>・指摘された仕様漏れや曖昧な記述を修正する | ・実装方針やコードの意図を説明する<br>・設計との差分があれば理由を説明する<br>・工夫点・懸念点・重点的に見てほしい点を共有する<br>・指摘されたコードを修正し、必要に応じてテスト結果を提示する | ・テスト設計の考え方と作成根拠を説明する<br>・テスト対象／前提条件／テストデータ／期待結果を説明する<br>・指摘された不足観点・不足ケースを追加する |

## 1.6 レビューでやってはいけないこと

### 共通

**レビュアー**

- 成果物ではなく作成者を評価する、人格を否定する。
- 根拠のない指摘や、好みだけの変更要求をする。
- 代替案や理由を示さずに否定する。
- レビュー観点から外れた議論を続ける、細部に偏って本質的な問題を見落とす。

**レビューイー**

- 指摘を感情的に否定する、放置する。
- 設計意図や実装意図を説明しない。
- 修正内容を共有しない、曖昧なまま承認を求める。
- 指摘の意図を理解しないまま表面的に修正する。

### 工程ごとに避けること

| 対象 | 詳細設計レビュー | コードレビュー | テストレビュー |
|---|---|---|---|
| **レビュアー** | ・実装方法の好みだけで設計変更を要求する<br>・要件や基本設計を確認せずに否定する<br>・曖昧な記述や例外処理・境界条件を見逃す<br>・テスト可能性を考慮しない | ・コーディングスタイルだけを指摘する／書き方の好みを押し付ける<br>・設計との整合性を確認しない<br>・動作確認済みであることだけを理由に承認する<br>・性能・セキュリティ・保守性の観点を確認しない | ・テストケースを読まずに承認する<br>・テスト実施結果だけを確認し内容を見ない<br>・正常系だけで十分と判断する<br>・期待結果の曖昧さを見逃す／「動けばよい」で網羅性を軽視する |
| **レビューイー** | ・設計根拠を説明できない状態でレビューに出す<br>・未決事項を明記しない<br>・例外処理や境界条件を省略する<br>・実装者が判断すればよいという前提で曖昧に書く | ・動作することだけを理由に指摘を退ける<br>・設計との差分や複雑な処理の意図を説明しない<br>・指摘された修正に対して確認テストを行わない | ・正常系しか用意しない<br>・テストケースの根拠を説明しない／根拠なく省略する<br>・期待結果を曖昧にする／テスト対象外の範囲を明記しない<br>・指摘された不足ケースを追加しない |

## 1.7 指摘の重大度ラベル

レビュアーは各指摘に、以下のいずれかの重大度ラベルを必ず付与する。本表を全工程・全資料で共通の定義とし、各ガイドライン・[レビュー記録テンプレート](共通/レビュー記録テンプレート.md)・[レビュー分類方針総覧](共通/レビュー分類方針総覧.md) はこれを参照する。

| ラベル | 英語（総覧の軸2） | 定義 | レビューイーの対応 |
|---|---|---|---|
| **[必須]** | Must | ルール違反・バグ・要件未達・整合性破壊・セキュリティ問題など、放置するとマージできない欠陥 | 必ず修正してから次工程へ進む |
| **[推奨]** | Should | 品質・保守性・テスタビリティの向上に有効。放置にはリスクが伴う | 修正する、または見送り理由を明示して合意を得る |
| **[提案]** | Nits | 好み・スタイルの範囲で、品質への実害がない任意の改善 | 対応は任意（返答不要） |

> 重大度の付け方（マージをブロックすべきか等の判断基準）は [レビュー分類方針総覧](共通/レビュー分類方針総覧.md) の軸2を参照する。
