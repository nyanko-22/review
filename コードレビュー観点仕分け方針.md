# コードレビュー 観点仕分け方針

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

コードレビューでは特に「ルールが明確なのに守られていない」類の指摘（空のcatchブロック、domain層へのORM import等）が多発しやすい。これらはチームレビューで議論する必要がなく、提出前に本人が自己確認することで排除できる。

また、コードレビュー固有の事情として、**CI/CDで自動検出できるもの**（Lintエラー、型エラー、テスト失敗）は事前自己チェックリストにも含めず、パイプラインで強制する。チェックリストはCIが検出できない人間の判断が必要な項目のみを扱う。

**目標状態**:  
チームレビューは「設計意図との整合性・命名の妥当性・設計判断のトレードオフ」にのみ使われ、ルール違反の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を以下2つの問いで判定する（詳細設計レビューと同じ基準）。

### 問い①: 答えが一意に決まるか？

> 「このチェックをした人が10人いたとき、全員が同じ判断をするか？」

- **Yes（答えが一意）** → 事前自己チェック候補
- **No（人によって判断が変わる）** → チームレビュー候補

### 問い②: 提出者本人が単独で確認できるか？

> 「コードとDDDルールだけを見れば、他者の意見なしに確認できるか？」

- **Yes（単独確認可能）** → 事前自己チェック
- **No（他者視点や議論が必要）** → チームレビュー

### コードレビュー特有の追加原則

> **CIで自動検出できるものはチェックリストに含めない**

Lintエラー・型エラー・テスト失敗はCI/CDパイプラインで強制する。チェックリストはCIが検出できない「人間の判断が必要な項目」のみを対象とする。

| 分類 | 性質 | 例 |
|---|---|---|
| **CI/CDで自動化** | ツールで機械的に検出可能 | Lintエラー、型エラー、テスト失敗 |
| **事前自己チェック** | ルールベース・客観的・答えが一意 | domain層のimport、空のcatch、機密情報のログ出力 |
| **チームレビュー** | 設計判断・解釈・トレードオフあり | 命名の妥当性、ビジネスロジックの配置、エラー伝播の設計 |

## 3. 仕分け結果サマリー

### 観点A: DDDパターンの実装正確性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| VOにreadonly/Object.freezeで不変性が保証されている | 事前自己チェック | キーワードの有無はコードから客観的に確認できる |
| VOのファクトリメソッドでバリデーションを行っている | 事前自己チェック | バリデーション処理の有無はコードから確認できる |
| equals()が属性比較で実装されている | 事前自己チェック | 実装内容はコードから客観的に確認できる |
| EntityのIDがValue Objectとして実装されている | 事前自己チェック | 型の確認はコードから客観的に確認できる |
| Aggregate Root以外が外部から直接publicアクセスされていない | 事前自己チェック | アクセス修飾子の確認はコードから客観的にできる |
| Domain Serviceがprivateフィールドを持たない | 事前自己チェック | フィールドの有無はコードから客観的に確認できる |
| Repositoryがドメイン層のinterfaceを実装している | 事前自己チェック | implements宣言の確認はコードから客観的にできる |
| Repositoryの返却値がドメインオブジェクト | 事前自己チェック | 返却型の確認はコードから客観的にできる |
| Aggregate境界の実装が設計意図と合っているか | チームレビュー | 設計書との解釈の差が出やすい。設計者の意図の確認が必要 |
| Application ServiceとDomain層のロジック配置が適切か | チームレビュー | 何が「ビジネスロジック」かの解釈が人によって変わりやすい |

### 観点B: レイヤー構造・依存方向

| チェック項目 | 分類 | 理由 |
|---|---|---|
| domain層にORM・Webフレームワークのimportがない | 事前自己チェック | importの確認はコードから客観的にできる。ESLintで自動化も可能 |
| application層がinfrastructure具体クラスをimportしていない | 事前自己チェック | importの確認はコードから客観的にできる |
| presentationがDTOに変換してレスポンスを返している | 事前自己チェック | 変換処理の有無はコードから確認できる |

※ 観点Bは全項目がルールベースで客観的に確認できるため、全て事前自己チェック。

### 観点C: コードの可読性・保守性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| マジックナンバー・ストリングが定数/Enumに定義されている | 事前自己チェック | リテラルの直書きはコードから客観的に検出できる |
| 命名がユビキタス言語として適切か | チームレビュー | 命名の妥当性は主観が入る。チームで合意することで統一感が生まれる |
| 関数・クラスの粒度が適切か | チームレビュー | 何が「適切な粒度」かは設計判断。過剰な分割も問題になる |

### 観点D: エラーハンドリング

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 空のcatchブロック（握り潰し）がない | 事前自己チェック | 空のcatchブロックはコードから客観的に検出できる |
| エラーハンドリング戦略が適切か | チームレビュー | どのエラーをどの層で捕まえてどう伝播させるかはトレードオフがある |

### 観点E: セキュリティ基本確認

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 機密情報がログに出力されていない | 事前自己チェック | console.log等でのpassword/token出力はコードから確認できる |
| バリデーションの範囲・内容が十分か | チームレビュー | 何を検証すれば「十分か」はビジネスの判断が必要 |

## 4. CI/CDで自動化すべき項目

以下は事前自己チェックリストにも含めず、CI/CDで強制する。PRマージ条件として設定することで、人間のレビュー工数ゼロで品質を担保できる。

| 項目 | 自動化手段 |
|---|---|
| Lintエラーなし | ESLint（`--max-warnings 0`） |
| 型エラーなし | TypeScript（`tsc --noEmit`） |
| 全テストがpass | テストランナー（Jest等） |
| import依存方向の違反 | ESLintの `eslint-plugin-boundaries` 等 |
| コードフォーマット | Prettier |
