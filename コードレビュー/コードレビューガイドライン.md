  # コードレビューガイドライン

## 1. このガイドラインについて

### 目的
コードレビューは、**詳細設計で決定した設計意図がコードに正しく実装されているかを確認し、コーディング段階で混入したバグ・設計逸脱を検出する**ことを目的とする。  
詳細設計の内容自体はすでに設計レビューで合意済みであり、このレビューでは設計の見直しは行わない。

### 対象工程
- **対象**: コーディング完了後・マージ前
- **対象読者**: レビュアー（コードを評価する側）・レビューイー（コードを提出する側）の両方

### アーキテクチャ前提
本プロジェクトは **ドメイン駆動設計（DDD）** を採用している。  
コードレビューでは、詳細設計で定義したVO/Entity/Aggregate/Domain Service/Repository/Application ServiceがDDDの原則通りに実装されているかを重点的に確認する。

---

## 2. レビュースコープ

「何を見るか」と同時に「何を見ないか」を明確にする。スコープ外の指摘はレビュー時間を無駄に消費する。

| 対象（見る） | 対象外（見ない） |
|---|---|
| DDDパターンの実装正確性（VO/Entity/Aggregate等） | 詳細設計そのものの見直し（設計レビュー済み） |
| レイヤー構造・依存方向の実装 | コードフォーマット・インデント（Linterで自動化） |
| コードの可読性・保守性 | LintエラーBuild）・型エラー（CI/CDで自動検出） |
| エラーハンドリングの実装 | パフォーマンス最適化（プロファイリングなしの主観判断） |
| セキュリティ基本確認 | インフラ構成・CI/CD設定 |
| ユビキタス言語の命名 | テストコードの品質（テストレビューで確認） |

---

## 3. 役割と責任

レビュアー・レビューイーの役割は [ソフトウェアレビュー標準 §1.5](../ソフトウェアレビュー標準.md) の定義に従う。本工程（コードレビュー）での要点は次のとおり。

| 役割 | 本工程での要点 |
|---|---|
| **レビューイー** | コードを提出し、指摘に対して「修正」または「見送り理由」を回答する |
| **レビュアー** | コード観点でフィードバックを行い、指摘には必ず重大度ラベル（[必須]/[推奨]/[提案]）を明示する |

> **重要**: レビュアーは重大度ラベルを明示せずに指摘してはいけない。レビューイーは重大度が明示されていない指摘への対応は不要とする（[標準 §1.7](../ソフトウェアレビュー標準.md)）。

## 4. レビュー観点

### 観点A: DDDパターンの実装正確性

**なぜこれを見るか**  
設計段階で正しく定義されたVO/Entity/Aggregate等も、実装で誤るとDDDの価値が消える。設計の意図がコードに忠実に反映されているかが、このレビューの核心となる。

**チェック内容**

#### Value Object
- `readonly` キーワードや `Object.freeze()` 等で不変性（Immutable）が保証されているか
- `equals()` メソッドが属性の比較で実装されているか（IDでなく値で比較）
- コンストラクタまたは静的ファクトリメソッド（`create()`等）内でバリデーションを行い、不正な値のインスタンスが生成されないようになっているか

#### Entity
- IDがプリミティブ型（string/number）でなく、Value Objectとして実装されているか
- 状態変更メソッドがビジネスルールを表現しているか（例: `changeEmail()` はメールの形式検証も行う）
- 同一性の比較がIDで行われているか

#### Aggregate
- Aggregate Root以外の内部オブジェクトが外部から直接 `public` アクセスされていないか
- Aggregate内の変更はRoot経由のメソッドのみで行われているか
- Rootのメソッドがトランザクション整合性を保証しているか

#### Domain Service
- privateなフィールドを持っていないか（ステートレスであるか）
- Application層・Infrastructure層に依存していないか

#### Repository
- ドメイン層で定義したインターフェースを実装しているか
- Aggregate単位での `save()` / `findById()` が実装されているか
- ORMのモデルをそのまま返さず、ドメインオブジェクトに変換して返しているか

#### Application Service
- ドメインオブジェクトを呼び出して調整しているだけか（ビジネスロジックを含んでいないか）
- トランザクション管理を担当しているか

---

### 観点B: レイヤー構造・依存方向

**なぜこれを見るか**  
設計で正しい依存方向が定義されていても、実装の `import` 一つで依存方向が逆転する。逆転した依存はドメイン層の独立性を失わせ、フレームワーク変更時の影響範囲を広げる。

```
[依存の方向（正しい方向）]
Presentation → Application → Domain ← Infrastructure
```

**チェック内容**
- domain層のファイルに ORM・Webフレームワーク等のimportが含まれていないか
- application層のファイルが infrastructure の具体実装クラスを直接importしていないか（interfaceを経由しているか）
- presentation層がドメインオブジェクトをそのままレスポンスに使っていないか（DTOに変換しているか）

---

### 観点C: コードの可読性・保守性

**なぜこれを見るか**  
可読性の低いコードはバグの温床となり、将来の変更コストを高くする。特に命名はユビキタス言語との一致が重要で、ドメインの知識がコードから読み取れるかどうかに直結する。

**チェック内容**
- 関数・メソッドが単一の責任を持っているか（一つのことだけをするか）
- 変数・関数・クラスの命名がドメインのユビキタス言語を使っているか
- マジックナンバー・マジックストリングが定数またはEnumに定義されているか（例: `7` → `MAX_ORDER_ITEMS`）
- ネストが深すぎないか（3段階以上は早期returnへのリファクタリングを検討）

---

### 観点D: エラーハンドリング

**なぜこれを見るか**  
エラーハンドリングが不適切だと、バグの原因特定が困難になり、不整合データが静かに蓄積する。ドメインのエラーはドメイン層で定義・発生させることが重要。

**チェック内容**
- ドメインルール違反を表すカスタムエラークラスがドメイン層に定義されているか（例: `InsufficientStockError`）
- `catch` ブロックで例外を握り潰していないか（何もしない空のcatchブロックがないか）
- `null` / `undefined` を返してエラーを曖昧にしていないか（Result型・例外で明示しているか）
- エラーメッセージがデバッグに役立つ情報を含んでいるか

---

### 観点E: セキュリティ基本確認

**なぜこれを見るか**  
セキュリティの問題は後から発見されると修正コストが特に高く、データ漏洩・不正アクセスのリスクがある。Value Objectのバリデーションがセキュリティの第一の防衛線になる。

**チェック内容**
- 入力値のバリデーションがValue ObjectやPresentation→Application境界で行われているか
- パスワード・トークン・個人情報等の機密情報がログ（`console.log` 等）に出力されていないか
- SQLクエリにユーザー入力を文字列連結で埋め込んでいないか（ORMのパラメータバインドを使用しているか）

---

## 5. 指摘の重大度分類

レビュアーは全ての指摘に、標準で定義する重大度ラベル（[ソフトウェアレビュー標準 §1.7](../ソフトウェアレビュー標準.md)）を必ず付ける。本工程での具体的な目安は次のとおり。

| ラベル | 英語 | 本工程での目安 | レビューイーの対応 |
|---|---|---|---|
| **[必須]** | Must | DDD原則違反、バグ・データ不整合リスク、セキュリティ問題 | **必ずFix** してから次工程へ進む |
| **[推奨]** | Should | 可読性・保守性・テスタビリティの向上に有効。対応しないリスクがある | Fixするか、**見送り理由を明示**して合意を得る |
| **[提案]** | Nits | よりよい実装の可能性がある任意の改善案 | 対応は任意。返答不要 |

**記載例**
```
[必須] Domain層でExpress.Requestをimportしています。依存方向が逆転しているため修正が必要です。
[推奨] `7` というマジックナンバーを MAX_RETRY_COUNT のような定数に定義してください。
[提案] この処理はAggregateのメソッドに移動するとより意図が明確になります。
```

## 6. 完了基準（Exit Criteria）

完了の基本的な考え方は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)（レビューの進め方）に準拠する。本工程では以下の条件を全て満たした時点でレビュー完了とする。

- [ ] **[必須]** の指摘が全て解消されている
- [ ] **[推奨]** の指摘が全て「修正済み」または「見送り理由あり」の状態になっている
- [ ] レビュアーが承認コメントを記録している（例: `LGTM` / `承認`）

> **[推奨] の見送り理由の例**
> - 「今フェーズのスコープ外のため次スプリントのタスクに追加しました（チケット#XXX）」
> - 「コスト対効果を検討した結果、現時点では見送ります。〇〇の理由で今後の拡張時に対応します」

## 7. レビュー実施方式

レビュー指摘数を減らしてレビュー時間を圧縮するため、チェック項目を軸0の4層（[レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)）に分けて実施する。**ねらいは人間（③④）が見る項目を絞り、機械的・定型的な確認をツール（①）と生成AI（②）に寄せること。** コードレビューは成果物がコードのため、①ツール自動化の比率が他工程より高い。

| 層 | 実施者 | タイミング | 対象項目の性質 |
|---|---|---|---|
| **①ツール自動化** | CI（Lint・型・テスト・import境界） | PRオープン前（CI） | 機械的に検出できる（チェックリストにも載せない。§4参照） |
| **②生成AIレビュー** | 生成AI（提出前の自動一次レビュー） | PRオープン前 | 答えが概ね一意で、コードからAIが判定できる（§7-1の多く） |
| **③人間・事前自己チェック** | レビューイー本人 | PRオープン前 | AIに任せきれない／本人の最終確認が要る |
| **④人間・チームレビュー** | チーム全員 | PRレビュー時 | 設計判断・トレードオフ・合意が必要（§7-2。AIは論点出しの補助） |

> **運用ルール**: 提出前チェック（①〜③）が全て済んでいないコードはPRとして提出しない。  
> ④チームレビューは「判断が必要な議論」に集中することで、1件あたりの所要時間を削減する。生成AIの運用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。

## 7-1. 提出前チェックリスト（②生成AIレビュー → ③本人確認）

PRオープン前に確認する項目。多くは **②生成AIレビュー（提出前の自動一次レビュー）** に渡し、AIの指摘を解消したうえでレビューイー本人が最終確認する（③）。ツールで強制する項目（①）は §4 を参照。**全項目が ✅ になってからPRを提出する。**

### A. DDDパターンの実装正確性

**Value Object**
- [ ] `readonly` または `Object.freeze()` 等で不変性が保証されている
- [ ] コンストラクタ/ファクトリメソッドでバリデーションを行っている
- [ ] `equals()` が属性比較で実装されている

**Entity**
- [ ] IDがValue Objectとして実装されている（プリミティブ型ではない）

**Aggregate**
- [ ] Root以外の内部オブジェクトが外部から直接 `public` アクセスされていない

**Domain Service**
- [ ] privateフィールドがない（ステートレス）

**Repository**
- [ ] ドメイン層のインターフェースを実装している
- [ ] 返却値がドメインオブジェクト（ORMモデルのまま返していない）

### B. レイヤー構造・依存方向

- [ ] domain層のファイルにORM・Webフレームワークのimportがない
- [ ] application層が infrastructure の具体クラスをimportしていない（interfaceを経由）
- [ ] presentation層がドメインオブジェクトをそのままレスポンスに使っていない（DTOに変換）

### C. コードの可読性・保守性

- [ ] マジックナンバー・マジックストリングが定数/Enumに定義されている

### D. エラーハンドリング

- [ ] 空のcatchブロック（例外を握り潰している箇所）がない

### E. セキュリティ基本確認

- [ ] 機密情報（password, token, 個人情報等）がconsole.log/loggerに出力されていない

## 7-2. チームレビューチェックリスト（チームで議論して確認）

④人間・チームレビューの項目。判断が分かれる可能性があるため、**チームで合意を形成する**。生成AIは設計書との差分・観点の抜けを提示する一次レビューとして併用できるが、**最終判断は人間が行う**。

### A. DDDパターンの実装正確性

- [ ] Aggregate境界の実装が設計意図と合っているか（RootがAggregateの整合性を保証しているか）  
  *（設計と実装の解釈の差が出やすい。設計書との照合が必要）*
- [ ] Application ServiceとDomain層のロジック配置が適切か  
  *（何が「ビジネスロジック」かの解釈が人によって変わりやすい）*

### B. レイヤー構造・依存方向

（事前自己チェックでルールベースの確認が完了するため、チームレビューの観点なし）

### C. コードの可読性・保守性

- [ ] 命名がユビキタス言語として適切か（ドメインの概念を正しく表現しているか）  
  *（命名の妥当性は主観が入る。チームで合意することで統一感が生まれる）*
- [ ] 関数・クラスの粒度が適切か（分割/統合の判断）  
  *（何が「適切な粒度」かは設計判断。過剰な分割も問題）*

### D. エラーハンドリング

- [ ] エラーハンドリング戦略（どのエラーをどの層で捕まえてどう伝播させるか）が適切か  
  *（エラーの伝播設計はトレードオフがある。チームの合意が必要）*

### E. セキュリティ基本確認

- [ ] バリデーションの範囲・内容がビジネスルール上十分か  
  *（何を検証すれば「十分か」はビジネスの判断が必要）*

## 付録: DDDパターンとコードの対応クイックリファレンス

| DDDパターン | コードレベルの確認ポイント |
|---|---|
| **Value Object** | `readonly`/`freeze` で不変。`equals()` が属性比較。ファクトリメソッドでバリデーション |
| **Entity** | IDがVO型。状態変更メソッドがビジネスルールを含む |
| **Aggregate** | Root以外の内部オブジェクトはprivateまたはprotected |
| **Domain Service** | privateフィールドなし。domain層のimportのみ |
| **Repository** | domain層のinterfaceを実装。Aggregate単位でsave/find |
| **Application Service** | ドメインオブジェクトを呼び出すのみ。ビジネスロジックなし |
