# コードレビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体・確認手段（4層: ①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）」を、コードレビューの各チェック項目へ適用した仕分け結果である。4層の判断基準は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)、生成AIの活用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。仕分け対象の観点A〜Eの詳細は [コードレビューガイドライン](コードレビューガイドライン.md) を参照。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

コードレビューでは特に「ルールが明確なのに守られていない」類の指摘（空のcatchブロック、domain層へのORM import等）が多発しやすい。これらはチームレビューで議論する必要がなく、提出前に本人が自己確認することで排除できる。

また、コードレビュー固有の事情として、**CI/CDで自動検出できるもの**（Lintエラー、型エラー、テスト失敗）は事前自己チェックリストにも含めず、パイプラインで強制する。チェックリストはCIが検出できない人間の判断が必要な項目のみを扱う。

**目標状態**:  
チームレビューは「設計意図との整合性・命名の妥当性・設計判断のトレードオフ」にのみ使われ、ルール違反の指摘に時間を取られない状態。

## 2. 仕分けの考え方（判断基準）

各チェック項目を、軸0の4層（[レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)）へ、上の層から順に当てはめて仕分ける。判断の出発点は「問い①: 答えが一意に決まるか」「問い②: 本人が単独で確認できるか」。

| 層 | 判定 | この工程での例 |
|---|---|---|
| **①ツール自動化** | ツールで決定論的に検出できる（CIで強制し、チェックリストにも載せない） | Lint/型エラー、import依存方向違反、空のcatch、フォーマット |
| **②生成AIレビュー** | ①不可だが、答えが概ね一意で、コードだけで判定できる | VOの不変性・equals()実装、DTO変換の有無、機密情報のログ出力 |
| **③人間・事前自己チェック** | AIに任せきれない／本人の最終確認が要る | AIの誤検知の取捨、設計意図の最終確認 |
| **④人間・チームレビュー** | 答えが一意でない・トレードオフ・合意が要る | 命名の妥当性、ビジネスロジックの配置、エラー伝播の設計 |

コードレビューは成果物がコードであるため、**①ツール自動化** の比率が他工程より高い。従来「事前自己チェック」としていたルールベース項目のうち、ツールで表現できるものは①へ、コードの意味理解が要るものは**②生成AIレビュー**（提出前のAI一次レビュー）へ移す。これにより人間が見る項目（③＋④）を絞る。

> **CIで自動検出できるもの（①）はチェックリストに含めない**（従来原則を踏襲）。生成AIは誤検知・幻覚を起こすため、最終責任は人間にある（[生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md)）。

## 3. 仕分け結果サマリー

分類欄は軸0の4層（①ツール自動化／②生成AIレビュー／③人間・自己チェック／④人間・チームレビュー）を表す。「②→④」は、生成AIで一次評価したうえで人間が最終判断することを示す。

### 観点A: DDDパターンの実装正確性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| VOにreadonly/Object.freezeで不変性が保証されている | ②生成AI | キーワードの有無をAIが確認できる（lintルール化は手間） |
| VOのファクトリメソッドでバリデーションを行っている | ②生成AI | バリデーション処理の有無をAIが確認できる |
| equals()が属性比較で実装されている | ②生成AI | 実装内容をAIが意味的に確認できる |
| EntityのIDがValue Objectとして実装されている | ②生成AI | 型の確認をAIが行える |
| Aggregate Root以外が外部から直接publicアクセスされていない | ②生成AI | アクセス修飾子をAIが確認できる |
| Domain Serviceがprivateフィールドを持たない | ②生成AI | フィールドの有無をAIが確認できる |
| Repositoryがドメイン層のinterfaceを実装している | ②生成AI | implements宣言をAIが確認できる |
| Repositoryの返却値がドメインオブジェクト | ②生成AI | 返却型をAIが確認できる |
| Aggregate境界の実装が設計意図と合っているか | ④チーム | 設計書との解釈の差が出やすい。設計者の意図の確認が必要 |
| Application ServiceとDomain層のロジック配置が適切か | ④チーム | 何が「ビジネスロジック」かの解釈が人によって変わりやすい |

### 観点B: レイヤー構造・依存方向

| チェック項目 | 分類 | 理由 |
|---|---|---|
| domain層にORM・Webフレームワークのimportがない | ①ツール | import違反は eslint-plugin-boundaries 等で機械検出できる |
| application層がinfrastructure具体クラスをimportしていない | ①ツール | 同上。CIで強制する |
| presentationがDTOに変換してレスポンスを返している | ②生成AI | 変換処理の有無は意味理解が要るためAIが確認 |

### 観点C: コードの可読性・保守性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| マジックナンバー・ストリングが定数/Enumに定義されている | ①ツール／②生成AI | 単純なリテラル直書きはlintで、文脈依存のものはAIが検出 |
| 命名がユビキタス言語として適切か | ②→④ | AIが用語集と照合して一次評価 → 妥当性はチームで最終合意 |
| 関数・クラスの粒度が適切か | ④チーム | 何が「適切な粒度」かは設計判断。過剰な分割も問題になる |

### 観点D: エラーハンドリング

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 空のcatchブロック（握り潰し）がない | ①ツール | ESLint（`no-empty` 等）で機械検出できる |
| エラーハンドリング戦略が適切か | ④チーム | どのエラーをどの層で捕まえてどう伝播させるかはトレードオフがある |

### 観点E: セキュリティ基本確認

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 機密情報がログに出力されていない | ①ツール／②生成AI | 既知パターンは静的解析で、文脈依存のものはAIが検出 |
| バリデーションの範囲・内容が十分か | ④チーム | 何を検証すれば「十分か」はビジネスの判断が必要 |

## 4. ①ツール自動化で強制する項目

以下はチェックリストにも含めず、CI/CDで強制する。PRマージ条件として設定することで、人間のレビュー工数ゼロで品質を担保できる。

| 項目 | 自動化手段 |
|---|---|
| Lintエラーなし | ESLint（`--max-warnings 0`） |
| 型エラーなし | TypeScript（`tsc --noEmit`） |
| 全テストがpass | テストランナー（Jest等） |
| import依存方向の違反 | ESLintの `eslint-plugin-boundaries` 等 |
| 空のcatchブロック | ESLint（`no-empty`） |
| コードフォーマット | Prettier |
