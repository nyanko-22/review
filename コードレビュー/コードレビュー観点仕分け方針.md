# コードレビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体・確認手段（4層: ①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）」を、コードレビューの各チェック項目へ適用した仕分け結果である。4層の判断基準は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)、生成AIの活用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。仕分け対象の観点の詳細は [共通観点／コード](../共通観点/コード.md) と、該当する [設計別観点](../設計別観点/)（DDD／インフラ・IaC／HLF）を参照。本§3は **設計種別ごと** にグルーピングして分類を示す。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー時間が肥大化する課題と、観点を仕分けて機械的・定型的な確認をツール・生成AIへ寄せるねらいは [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) を参照。

コードレビューでは特に「ルールが明確なのに守られていない」類の指摘（空のcatchブロック、domain層へのORM import等）が多発しやすい。コードレビュー固有の事情として、**CI/CDで自動検出できるもの**（Lintエラー、型エラー、テスト失敗）は事前自己チェックリストにも含めず、パイプラインで強制する（①ツール自動化）。チェックリストはCIが検出できない人間の判断が必要な項目のみを扱う。

**目標状態**:  
チームレビューは「設計意図との整合性・命名の妥当性・設計判断のトレードオフ」にのみ使われ、ルール違反の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を軸0の4層へ上の層から順に当てはめて仕分ける。判断フロー（ツールで検出可か→答えが一意か→本人確認が要るか）は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md) に従う。本工程での各層の例は次のとおり。

| 層 | この工程での例 |
|---|---|
| **①ツール自動化**（CIで強制し、チェックリストにも載せない） | Lint/型エラー、import依存方向違反、空のcatch、フォーマット |
| **②生成AIレビュー** | VOの不変性・equals()実装、DTO変換の有無、機密情報のログ出力 |
| **③人間・事前自己チェック** | AIの誤検知の取捨、設計意図の最終確認 |
| **④人間・チームレビュー** | 命名の妥当性、ビジネスロジックの配置、エラー伝播の設計 |

コードレビューは成果物がコードのため **①ツール自動化の比率が他工程より高い**。ルールベース項目のうちツールで表現できるものは①へ（CIで強制し、チェックリストに含めない）、コードの意味理解が要るものは **②生成AIレビュー**（提出前一次レビュー）へ移す。生成AIの限界と責任分界（最終責任は人間）は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。

---

## 3. 仕分け結果サマリー

分類欄は軸0の4層（①ツール自動化／②生成AIレビュー／③人間・自己チェック／④人間・チームレビュー）を表す。「②→④」は、生成AIで一次評価したうえで人間が最終判断することを示す。項目は **設計種別ごと**（共通／DDD／インフラ・IaC／HLF）にグルーピングした。①ツールで強制する項目の一覧は §4 を参照。

### 3.1 共通観点（全設計種別で適用）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 詳細設計どおりに実装され、処理フロー・入出力・DB操作が設計と一致している | ②生成AI | 設計書とコードの処理整合をAIが意味的に確認できる |
| 正常系・異常系・分岐・境界値・型変換の正確性 | ②生成AI | コードの意味理解でAIが一次確認できる |
| マジックナンバー・ストリングが定数/Enumに定義されている | ①ツール／②生成AI | 単純なリテラル直書きはlintで、文脈依存のものはAIが検出 |
| 関数・クラスの粒度・重複・責務分割が適切か | ④チーム | 何が「適切な粒度」かは設計判断。過剰な分割も問題になる |
| 空のcatchブロック（握り潰し）がない | ①ツール | ESLint（`no-empty` 等）で機械検出できる |
| エラーハンドリング戦略が適切か | ④チーム | どのエラーをどの層で捕まえてどう伝播させるかはトレードオフがある |
| トランザクション範囲・整合性（部分更新/同時実行/再実行）が適切か | ④チーム | 範囲の妥当性は設計判断 |
| 性能（ループ内DB・N+1・大量データ）が考慮されているか | ④チーム | 実データ・負荷前提の判断が要る |
| 機密情報がログに出力されていない | ①ツール／②生成AI | 既知パターンは静的解析で、文脈依存のものはAIが検出 |
| SQLインジェクション対策・秘密情報のソース直書きなし | ①ツール／②生成AI | 既知パターンは静的解析、文脈依存はAIが検出 |
| バリデーション・認証/認可の範囲・内容が十分か | ④チーム | 何を検証すれば「十分か」はビジネスの判断が必要 |

### 3.2 DDD観点（DDDを採用する場合）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| VOにreadonly/Object.freezeで不変性が保証されている | ②生成AI | キーワードの有無をAIが確認できる（lintルール化は手間） |
| VOのファクトリメソッドでバリデーションを行っている | ②生成AI | バリデーション処理の有無をAIが確認できる |
| equals()が属性比較で実装されている | ②生成AI | 実装内容をAIが意味的に確認できる |
| EntityのIDがValue Objectとして実装されている | ②生成AI | 型の確認をAIが行える |
| Aggregate Root以外が外部から直接publicアクセスされていない | ②生成AI | アクセス修飾子をAIが確認できる |
| Domain Serviceがprivateフィールドを持たない | ②生成AI | フィールドの有無をAIが確認できる |
| Repositoryがドメイン層のinterfaceを実装している | ②生成AI | implements宣言をAIが確認できる |
| Repositoryの返却値がドメインオブジェクト | ②生成AI | 返却型をAIが確認できる |
| domain層にORM・Webフレームワークのimportがない | ①ツール | import違反は eslint-plugin-boundaries 等で機械検出できる |
| application層がinfrastructure具体クラスをimportしていない | ①ツール | 同上。CIで強制する |
| presentationがDTOに変換してレスポンスを返している | ②生成AI | 変換処理の有無は意味理解が要るためAIが確認 |
| Aggregate境界の実装が設計意図と合っているか | ④チーム | 設計書との解釈の差が出やすい。設計者の意図の確認が必要 |
| Application ServiceとDomain層のロジック配置が適切か | ④チーム | 何が「ビジネスロジック」かの解釈が人によって変わりやすい |
| 命名がユビキタス言語として適切か | ②→④ | AIが用語集と照合して一次評価 → 妥当性はチームで最終合意 |

### 3.3 インフラ・IaC観点（AWS CDK 等のコードを含む場合）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| cdk-nag（または相当の静的解析）の指摘がゼロで通過している | ①ツール | 静的解析で機械検出。CIで強制する |
| 環境依存値が context/props/環境変数で外出しされハードコードされていない | ②生成AI | ハードコードの有無をAIが検出できる |
| 削除保護・暗号化・公開設定が設計どおりコードに明示設定されている | ②生成AI | デフォルト任せかどうかをAIが確認できる |
| 機密情報がコードに直接記載されていない（Secrets Manager/SSM参照） | ①ツール／②生成AI | 既知パターンは静的解析、文脈依存はAIが検出 |
| スタック/Construct の分割粒度・抽象度が適切か | ④チーム | 責務・ライフサイクル境界の設計判断 |
| 生成リソースの命名・タグ付けがコード全体で一貫しているか | ①ツール／②生成AI | 規約はルール化でき、ツールまたはAIで照合できる |

### 3.4 HLF観点（Hyperledger Fabric のチェーンコードを含む場合）

> HLF観点の軸0分類は、現時点では [HLF観点モジュール](../設計別観点/HLF観点.md) の各チェックリストに付した暫定区分（提出前＝②③／チーム＝④）を正とする。プロジェクトのネットワーク要件が固まった段階で本§3.4へ確定分類を転記する。

## 4. ①ツール自動化で強制する項目

以下はチェックリストにも含めず、CI/CDで強制する。PRマージ条件として設定することで、人間のレビュー工数ゼロで品質を担保できる。

| 項目 | 自動化手段 |
|---|---|
| Lintエラーなし | ESLint（`--max-warnings 0`） |
| 型エラーなし | TypeScript（`tsc --noEmit`） |
| 全テストがpass | テストランナー（Jest等） |
| import依存方向の違反 | ESLintの `eslint-plugin-boundaries` 等 |
| 空のcatchブロック | ESLint（`no-empty`） |
| コードフォーマット | Prettier |
