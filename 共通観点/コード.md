# 共通観点：コードレビュー

本モジュールは **設計種別を問わず、すべてのコード成果物で必ず適用する**汎用観点とチェックリストである。DDD／インフラ・IaC／HLF のコードでは、本モジュールに加えて該当する [設計別観点](../設計別観点/) を重ねて参照する。

- 入口・全体像・3軸（軸0/1/2）: [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)
- 各チェック項目の軸0分類と理由: [コードレビュー観点仕分け方針 §3](../コードレビュー/コードレビュー観点仕分け方針.md)
- **①ツール自動化で強制する項目**（Lint・型エラー・テスト・フォーマット・空catch・import依存方向 等）はチェックリストに載せず、CI/CDで強制する。一覧は [同 §4](../コードレビュー/コードレビュー観点仕分け方針.md)。

> チェックリストは §A（提出前＝②生成AIレビュー→③本人確認）と §B（チームレビュー＝④人間）の2区分。**提出前チェックが済んでいないコードはPRとして提出しない。**

---

## 1. 観点（なぜ見るか）

### A. 設計との整合性
コードレビューの核心は「詳細設計どおりに実装されているか」。処理フロー・入出力・DB操作/外部連携が設計と一致し、差分があれば理由が説明されているかを確認する。

### B. 正確性
正常系・異常系・分岐・境界値・NULL/空文字/0件・型変換/桁/丸めで誤動作しないか。ロジックの誤りはこの段で潰す。

### C. 可読性
可読性の低いコードはバグの温床で変更コストを高める。処理の流れ、変数/関数/クラス名の分かりやすさ、過度に複雑な条件分岐、コメントの過不足、マジックナンバーを確認する。※命名がユビキタス言語として適切かは [DDD観点](../設計別観点/DDD観点.md) 側で見る。

### D. 保守性
重複コード、変更の局所化、責務の分割、共通処理と個別処理の整理、将来の仕様変更への追従しやすさを確認する。

### E. 例外処理・ログ
例外の握り潰し、エラー時の戻り値/レスポンスの適切さ、必要なログの出力、**機密情報をログに出さないこと**、障害調査に必要な情報が含まれるかを確認する。

### F. トランザクション・整合性
トランザクション範囲、エラー時のロールバック、部分更新による不整合、同時実行・再実行時の二重処理を確認する。

### G. 性能
ループ内の不要なDBアクセス、N+1問題、大量データ処理、不要なメモリ使用、タイムアウト/リトライを確認する（プロファイリングなしの主観的最適化は対象外）。

### H. セキュリティ
入力値チェック、認証・認可、SQLインジェクション（パラメータバインド）、XSS等、秘密情報のソース直書きを確認する。既知パターンは静的解析（例: Coverity）で自動化を優先する。

## A. 提出前チェックリスト（②生成AIレビュー → ③本人確認）

多くは **②生成AIレビュー** に渡し、指摘を解消のうえ本人が最終確認（③）する。**全項目が ✅ になってからPRを提出する。**

### 設計との整合性
- [ ] 詳細設計どおりに実装され、処理フロー・入出力・DB操作/外部連携が設計と一致している。
- [ ] 設計との差分がある場合、理由が説明されている。

### 正確性
- [ ] 正常系・異常系が正しく動作し、分岐条件に誤りがない。
- [ ] 境界値・NULL/空文字/0件で誤動作しない。
- [ ] 型変換、桁数、丸め処理に誤りがない。

### 可読性
- [ ] マジックナンバーや意味不明な定数がない（定数/Enum化されている）。
- [ ] 過度に複雑な条件分岐がなく、不要な/不足するコメントがない。

### 例外処理・ログ
- [ ] エラー時の戻り値やレスポンスが適切である。
- [ ] 必要なログが出力され、機密情報（password/token/個人情報等）をログに出力していない。

### セキュリティ
- [ ] SQLインジェクション対策（パラメータバインド）がされている。
- [ ] XSS等の脆弱性対策がされている。
- [ ] 秘密情報をソースコードに直接記載していない。

## B. チームレビューチェックリスト（④人間・チームで議論して確認）

判断が分かれる可能性があるため **チームで合意を形成する**。生成AIは一次レビューとして併用できるが **最終判断は人間が行う**。分類理由は [仕分け方針 §3](../コードレビュー/コードレビュー観点仕分け方針.md) を参照。

### 可読性・保守性
- [ ] 変数/関数/クラス名が分かりやすいか。
- [ ] 重複コードが少なく、変更が局所化され、責務が適切に分割されているか（過剰な分割も含めて粒度が妥当か）。
- [ ] 共通処理と個別処理が整理され、将来の仕様変更に対応しやすいか。

### 例外処理・ログ
- [ ] エラーハンドリング戦略（どのエラーをどの層で捕まえてどう伝播させるか）が適切か。
- [ ] 障害調査に必要な情報がログに含まれているか。

### トランザクション・整合性
- [ ] トランザクション範囲が適切で、エラー時にロールバックされるか。
- [ ] 部分更新・同時実行・再実行による不整合や二重処理が発生しないか。

### 性能
- [ ] ループ内の不要なDBアクセス・N+1問題がないか。
- [ ] 大量データ処理時の性能・メモリ使用・タイムアウト/リトライが考慮されているか。

### セキュリティ
- [ ] 入力値チェック・認証/認可の範囲と内容がビジネスルール上十分か。
