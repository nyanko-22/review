# 共通観点：詳細設計レビュー

本モジュールは **設計種別を問わず、すべての詳細設計成果物で必ず適用する**汎用観点とチェックリストである。DDD／インフラ・IaC／HLF などの成果物では、本モジュールに加えて該当する [設計別観点](../設計別観点/) を重ねて参照する。

- 入口・全体像・3軸（軸0/1/2）: [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)
- 各チェック項目の軸0分類（誰が確認するか）と理由: [詳細設計レビュー観点仕分け方針 §3](../詳細設計レビュー/詳細設計レビュー観点仕分け方針.md)
- 提出前ゲート→実施方式→ラベル→完了の運用: [レビュー分類方針総覧](../共通/レビュー分類方針総覧.md)

> チェックリストは §A（提出前＝②生成AIレビュー→③本人確認）と §B（チームレビュー＝④人間）の2区分。**提出前チェックが済んでいない設計書はチームレビューに提出しない。**

---

## 1. 観点（なぜ見るか）

### A. 要件・基本設計との整合性
設計がビジネス要件・基本設計を正しく反映していなければ、完璧に実装しても誤ったものができる。設計段階での発見が最も低コスト。要件・基本設計・画面/API/DB仕様との矛盾、業務ルールの反映、未決事項・前提の明記を確認する。

### B. 処理仕様
正常系・異常系のフロー、分岐条件、入出力・戻り値、外部連携、バッチ/非同期の開始・終了条件が明確でないと、実装者が迷い「仕様と違う」手戻りを生む。

### C. 責務分担
1部品に複数責務が詰め込まれると変更コストが跳ね上がる。クラス/関数/モジュールの責務、共通化すべき処理と個別処理の整理、呼び出し関係の分かりやすさ、変更が想定される箇所の局所化を確認する。

### D. インターフェース設計
不明確なインターフェース定義は実装者の誤解を生む。引数・戻り値の型、副作用（状態変更）の有無、エラー型・例外が明確かを確認する。※メソッド名のユビキタス言語適合・ドメインイベントは [DDD観点](../設計別観点/DDD観点.md) 側で見る。

### E. データ設計
使用テーブル/カラム、型・桁・NULL可否・初期値、コード値/ステータスの意味、CRUD条件、整合性を保つ仕組みが曖昧だと、実装・運用で不整合データを生む。

### F. 例外処理・エラー処理
入力・外部システム・DB・タイムアウト時の処理、戻り値/画面表示/ログ出力、リトライ/ロールバック方針が定義されていないと、障害時に不整合が静かに蓄積する。

### G. 境界条件
0件・NULL/空文字/未設定・最大件数/桁・日付境界（月末/年度末）・重複データの動作は見落としやすく、本番障害の温床になる。

### H. 状態管理・排他制御
ステータス遷移、二重実行、同時更新の排他、再実行時・処理途中失敗時の状態が未定義だと、整合性バグを招く。

### I. 非機能要件
性能・セキュリティ・ログ・運用監視・保守性は後から作り込むほど高コスト。設計段階で満たせる構造かを確認する。

### J. テスト容易性
テストしにくい設計はバグの早期発見とリファクタリングの安全性を損なう。単体/結合テストケースを作成できる粒度か、正常系/異常系/境界値の観点が読み取れるか、DI・外部依存の抽象化ができる構造かを確認する。

### K. 新規技術の実現可否検証 ※新規ライブラリ/FW/外部サービスを採用する場合のみ
未検証の新規技術をそのまま設計に組み込むと、実装フェーズで「実現できない」「ライセンス・性能・互換性に抵触」が判明し大きな手戻りになる。採用根拠を設計段階で示す。

## A. 提出前チェックリスト（②生成AIレビュー → ③本人確認）

提出前に確認する。多くは **②生成AIレビュー**（提出前の自動一次レビュー）に渡し、指摘を解消のうえ本人が最終確認（③）する。**全項目が ✅ になってからチームレビューに提出する。**

### 要件・基本設計との整合性
- [ ] 要件定義・基本設計の内容と矛盾していない。
- [ ] 画面仕様、API仕様、DB設計と整合している。
- [ ] 業務ルールが詳細設計に反映されている。
- [ ] 未決事項や前提条件が明記されている。

### 処理仕様
- [ ] 正常系・異常系の処理フローが明確である。
- [ ] 分岐条件が明確である。
- [ ] 入力値、出力値、戻り値が明確である。
- [ ] 外部システム連携の仕様が明確である。
- [ ] バッチ処理や非同期処理の開始条件、終了条件が明確である。

### インターフェース設計
- [ ] 引数・戻り値の型が明確に定義されている。
- [ ] 副作用（状態変更）の有無が明確になっている。
- [ ] エラー型・例外が定義されている。

### データ設計
- [ ] 使用するテーブル、カラムが明確である。
- [ ] 型、桁数、NULL可否、初期値が明確である。
- [ ] コード値やステータス値の意味が明確である。
- [ ] 登録、更新、削除、参照の条件が明確である。

### 例外処理・エラー処理
- [ ] 入力エラー・外部システムエラー・DBエラー・タイムアウト時の処理が明確である。
- [ ] エラー時の戻り値、画面表示、ログ出力が明確である。
- [ ] リトライやロールバックの方針が明確である。

### 非機能要件
- [ ] ログ出力方針が明確で、運用監視に必要な情報が出力される。

### テスト容易性
- [ ] 詳細設計から単体・結合テストケースを作成できる粒度で記載されている。
- [ ] 正常系、異常系、境界値のテスト観点が読み取れる。

### 新規技術の実現可否 ※新規技術を採用する場合
- [ ] ライセンス・保守状況・性能・互換性を整理している（②AIで一次整理）。

## B. チームレビューチェックリスト（④人間・チームで議論して確認）

判断が分かれる可能性があるため **チームで合意を形成する**。生成AIは「設計書との差分」「観点の抜け」を出す一次レビューとして併用できるが、**最終判断は人間が行う**。分類理由は [仕分け方針 §3](../詳細設計レビュー/詳細設計レビュー観点仕分け方針.md) を参照。

### 要件・基本設計との整合性
- [ ] 例外・エラーケースが設計に含まれ、想定されるケースが網羅されているか。

### 責務分担
- [ ] クラス・関数・モジュールの責務が適切に分割されているか（1部品に複数責務が詰め込まれていないか）。
- [ ] 共通化すべき処理と個別処理が整理され、将来変更が想定される箇所が局所化されているか。

### 境界条件
- [ ] 0件・NULL/空文字・最大件数/桁・日付境界・重複データの動作が過不足なく定義されているか。

### 状態管理・排他制御
- [ ] ステータス遷移、二重実行・同時更新の排他制御、再実行・処理途中失敗時の状態が適切に定義されているか。

### 非機能要件
- [ ] 性能・セキュリティ要件を満たせる設計になっているか（トレードオフを含む設計判断）。
- [ ] 保守しやすい構造になっているか。

### テスト容易性
- [ ] 依存オブジェクトがDI（依存性注入）でき、外部依存（DB・外部API・時刻）が抽象化されているか。

### 新規技術の実現可否 ※新規技術を採用する場合
- [ ] 事前検証（PoC・技術検証）で実現可否を確認し、代替案・撤退条件を提示してチームで採用に合意できているか。
