# 生成AIレビュー活用方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) と [レビュー分類方針総覧 §3（軸0）](レビュー分類方針総覧.md) で定義する **軸0の②生成AIレビュー** について、その位置づけ・向き不向きの判断基準・運用フロー・限界と責任分界をまとめた方針である。3工程（詳細設計／コード／テスト）に共通して適用する。

## 1. 位置づけ — なぜ生成AIをレビューに使うのか

レビュー時間の肥大化は、「ルールとして明確なのに守られていない」「設計書を読めば分かる」類の議論価値の低い指摘が人間のレビューに集まることで起きる（課題の詳細は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)）。軸0はこうした定型的な確認を機械（①ツール）と生成AI（②）へ寄せる（4層の定義は [総覧 §3](レビュー分類方針総覧.md)）。

```
①ツール自動化 → ②生成AIレビュー → ③人間・事前自己チェック → ④人間・チームレビュー
  （機械的）       （半定型・意味理解）     （本人の最終確認）        （判断・合意）
└─────── 自動・低コスト ───────┘ └─────── 人間・高コスト（絞る対象） ───────┘
```

**生成AIの役割**: ツールのルールでは表現しにくいが答えが概ね一意な確認を、**提出前の自動一次レビュー**として肩代わりする。これにより、人間が見る項目（③＋④）を絞り、④チームレビューを設計判断に集中させる。

## 2. 何を生成AIに任せ、何を人間が見るか（判断基準）

軸0の判断フロー（①ツールで検出可→①／不可だが答えが概ね一意→②候補／一意でない・合意が要る→④／②候補のうち本人確認が要る→③）は [総覧 §3](レビュー分類方針総覧.md) に従う。本節は、その **②生成AIと④人間を見分ける勘所** を定義する。

### 生成AI向き（②）— 判断が「正解に収束する」もの

- ルール／パターンの照合（VOの不変性、空のcatch以外の握り潰し、命名規約 等）
- 存在チェック（正常系/異常系/境界値テストの有無、バリデーションの有無 等）
- 整合性の機械的突合（設計書と実装の処理フロー、基本設計と詳細設計の記述整合 等）
- 命名の一次評価（ユビキタス言語・用語集との照合）
- 観点の網羅漏れ検出（チェックリスト項目の機械的な充足確認）
- 定型の指摘（マジックナンバー、AAA構造の崩れ、AWS CDKのデフォルト値の明示漏れ 等）

### 人間向き（④）— 判断が「合意・文脈に依存する」もの

- トレードオフ（エラー伝播戦略、モック戦略、性能と可読性の両立 等）
- ビジネス価値・リスクの判断（網羅の「十分性」、重点的にテストすべき箇所 等）
- 粒度・境界の設計判断（Aggregate境界、トランザクション境界、責務分割 等）
- 命名・設計方針の最終合意（チームでの統一）
- 新規ライブラリ・FWの採否（実現可否・代替案・撤退条件の評価）

> ④の項目でも、生成AIは「設計書との差分」「観点の抜け」「たたき台コメント」を出す一次レビューとして併用してよい。**ただし最終判断と責任は人間にある。**

## 3. 運用フロー（提出前ゲート）

```
レビューイーが成果物を用意
   ↓
①ツール自動化（CI: Lint・型・テスト・静的解析）を通す ── 落ちたら修正
   ↓
②生成AIレビュー（各工程の「②生成AIレビュー対象」項目をプロンプトで渡す）
   ↓ AIの指摘
レビューイーが指摘を解消／誤検知を棄却（棄却は理由を添える）
   ↓
③人間・事前自己チェック（AIに任せきれない項目を本人が確認）
   ↓ ここまでクリアしないとチームレビューに出さない
④人間・チームレビュー（設計判断・合意。AIは論点出しの補助）
```

### AIへの入力の与え方

- 各工程の「観点仕分け方針」「ガイドライン」の **②生成AIレビュー対象の項目を、そのままプロンプト（チェック指示）として渡す**。観点チェックリストはAIへの指示書としても機能する。
- 対象の成果物（差分・該当ファイル・設計書）と、判定に必要な前提（ユビキタス言語の用語集、レイヤー構成、コーディング規約）をコンテキストとして与える。
- 出力は **指摘ごとに「観点・該当箇所・理由・重大度（[必須]/[推奨]/[提案]）の案」** を求める形式にすると、レビューイーが解消しやすく、④チームレビューにも引き継ぎやすい。

## 4. AIの限界と責任分界（必読）

- 生成AIは **誤検知・文脈欠落・幻覚（hallucination）** を起こす。指摘の正しさは保証されない。
- **AIは「検出と一次仕分け」を担い、最終責任は人間にある。** AIが「問題なし」と返しても、それは人間の確認責任を免除しない。
- **[必須]（Must）級になりうる指摘**（バグ・セキュリティ・ルール違反・整合性破壊）は、**AI単独で完結させず人間が確認する**。
- 機密情報・顧客データを含む成果物をAIへ渡す際は、利用するAIサービスの取り扱い規約・社内ポリシーに従う。
- AIの指摘を棄却する場合は理由を残し、繰り返し誤検知する項目はプロンプト／対象から見直す。

## 5. チームで決めるべき論点

- **②生成AIレビューを提出前ゲートで必須化するか**（任意併用にとどめるか）
- 利用するAIツール・モデルと、**機密情報の取り扱いルール**
- AIレビュー結果の **記録の要否**（記録する場合の保存先・様式）
- ① と ② の線引き（ツールでルール化する範囲）、③ に残す「本人確認必須」項目の範囲
