# レビュー分類方針 総覧

## 1. このドキュメントの位置づけ

既存の仕分け方針（[詳細設計](../詳細設計レビュー/詳細設計レビュー観点仕分け方針.md) / [コード](../コードレビュー/コードレビュー観点仕分け方針.md) / [テスト](../テストレビュー/テストレビュー観点仕分け方針.md)）は、各チェック項目を **「誰が・何が確認するか」=軸0** で仕分けている。本書は、その軸0を **コストの低い順に4層（①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）へ拡張定義** し、あわせて軸0に乗らない横断的な分類軸（軸1・軸2）を束ねる **総覧** である。3工程（詳細設計・コード・テスト）すべてに共通して適用する。

軸0の **ねらいは「人間が見る項目（③＋④）を絞り、機械的・定型的な確認をツール（①）と生成AIレビュー（②）へ寄せる」** こと。これにより、限られたレビュー時間を設計判断に集中させる。

**使い方**:  
本書は分類軸ごとの「判断方針（決め方の基準）」を定義する。実際に「どの観点・どの指摘がどの分類に入るか」は、この方針をもとにメンバーと討議して決める。本書はその **討議のたたき台／アジェンダ** として用いる。各軸の末尾に「チームで決めるべき論点」を明記しているので、討議ではそこを起点にする。

---

## 2. 分類軸の全体像（洗い出し）

レビュー対象は、以下の独立した軸で多面的に分類できる。各軸は直交しており、合成して使う（**WHO → HOW → 指摘の重み** の順に重ねる）。

| 軸 | 問い（何を分けるか） | 分類 | 状態 |
|---|---|---|---|
| **軸0: 確認主体・確認手段**（誰が・何が確認するか） | 機械的に検出できるか／答えが一意か／本人確認や合意が要るか | ①ツール自動化 / ②生成AIレビュー / ③人間・事前自己チェック / ④人間・チームレビュー | 本書§3で定義（4層） |
| **軸1: レビュー実施方式**（どう確認するか） | リアルタイムの双方向対話が必要か | 同期（ミーティング/モブ） / パスアラウンド（非同期回覧） | 本書§4で定義 |
| **軸2: 指摘の重大度**（指摘をどう扱うか） | マージをブロックすべきか | [必須] / [推奨] / [提案]（Must / Should / Nits） | 本書§5で定義 |

```
レビュー対象の1項目
  ├─ 軸0: 誰が・何が確認？ → ①ツール自動化 / ②生成AIレビュー / ③人間・事前自己チェック / ④人間・チームレビュー
  │                                          ↓ ④人間・チームレビューなら
  ├─ 軸1: どう確認？  → 同期で議論 / パスアラウンドで回覧
  │                                          ↓ 指摘が出たら
  └─ 軸2: その指摘の重み？ → [必須] / [推奨] / [提案]
```

- **軸0** を本書§3で4層に定義する（従来の「事前自己チェック / チームレビュー」を、①ツール自動化・②生成AIレビューを加えた4層へ拡張）。判断基準の出発点は従来どおり「問い①: 答えが一意に決まるか」「問い②: 提出者本人が単独で確認できるか」。
- **軸1・軸2** を本書§4・§5で定義する。

## 3. 軸0: 確認主体・確認手段（4層）

### 背景・目的

レビュー指摘が多くなる原因の多くは「ルールとして明確なのに守られていない」類の指摘であり、人間が議論する価値がない。そこで確認の担い手を、コストの低い順に4層へ分け、**機械的・定型的な確認をツール（①）と生成AIレビュー（②）へ寄せ、人間（③④）は判断・合意の要る項目に集中する。**

### 4つの層

| 層 | 担い手 | 性質 | 例 |
|---|---|---|---|
| **①ツール自動化** | CI/CD・静的解析（Lint・型・テスト・cdk-nag 等） | 機械的・決定論的に検出できる。人間工数ゼロ | Lint/型エラー、import依存方向違反、フォーマット |
| **②生成AIレビュー** | 生成AI（提出前の自動一次レビュー） | ツールのルールでは表現しにくいが、答えは概ね一意で、コード/設計書の意味理解で判定でき、ビジネス文脈・合意を要さない | VOの不変性の意味的確認、命名の一次評価、設計書と実装の処理フロー整合、観点の網羅漏れ検出 |
| **③人間・事前自己チェック** | 提出者本人 | 答えは一意寄りだが、AIだけでは不安定／責任上 本人の最終確認が要る | 設計意図の最終確認、AIの誤検知の取捨、機微な前提の確認 |
| **④人間・チームレビュー** | チーム（AIは論点出しの補助） | 答えが一意に決まらない設計判断・トレードオフ・合意形成 | Aggregate粒度、エラー伝播戦略、命名の最終合意、網羅の十分性 |

### 判断基準（上の層から順に当てはめる）

各観点・各チェック項目を、上の層から順に判定する。

#### 問い①: ツールで決定論的に検出できるか？

> 「既存のルール、または容易に書けるルールで、真偽を機械的に確定できるか？」

- **Yes** → **①ツール自動化**（チェックリストにも載せず、CI/CDで強制する）
- **No** → 問い②へ

#### 問い②: 答えが概ね一意で、成果物だけで判定できるか？

> 「コード・設計書という与えられた情報だけを見て、ビジネス文脈やチーム合意なしに、答えが概ね一つに収束するか？」

- **Yes** → **②生成AIレビュー** 候補（問い③へ）
- **No（一意に決まらない／文脈・合意が要る）** → **④人間・チームレビュー**

#### 問い③: （②候補のうち）AIに任せきれるか？

> 「AIの判定が不安定になりやすいか、または責任上 本人の最終確認が要るか？」

- **No（任せきれる）** → **②生成AIレビュー**（提出前の自動一次レビューで検出）
- **Yes（本人確認が要る）** → **③人間・事前自己チェック**（AIの一次結果を本人が確認する）

### 生成AI向き／人間向きの見分け（仕分けの勘所）

- **生成AI向き（②）** ＝ 判断が「正解に収束する」もの。ルール/パターン照合、存在チェック、整合性の機械的突合、命名の一次評価、観点の網羅漏れ検出、定型の指摘。
- **人間向き（④）** ＝ 判断が「合意・文脈に依存する」もの。トレードオフ、ビジネス価値判断、粒度・境界の設計判断、命名の最終合意、網羅の「十分性」判断、リスクの重み付け。

> ④人間・チームレビューの項目でも、生成AIは「設計書との差分」「観点の抜け」「たたき台のコメント」を提示する一次レビューとして併用できる。ただし **最終判断と責任は人間** にある。

### AIの限界と責任分界

生成AIは誤検知・文脈欠落・幻覚（hallucination）を起こす。**AIは「検出と一次仕分け」を担い、最終責任は人間にある。** [必須]（Must）級になりうる指摘は、AI単独で完結させず人間が確認する。詳細は [生成AIレビュー活用方針](生成AIレビュー活用方針.md) を参照。

### チームで決めるべき論点

- **① と ② の線引き**（どこまでをツールのルール化対象とし、どこからを生成AIに委ねるか）
- **②生成AIレビューを提出前ゲートに組み込むか**（必須化の可否）、結果の記録方法
- **③に残す「本人確認必須」項目の範囲**（AIに任せきれない領域の合意）

## 4. 軸1: レビュー実施方式（同期 vs パスアラウンド）

### 背景・目的

チームレビュー対象であっても、すべてを集まって議論する必要はない。一方向のフィードバックで足りるものを非同期の回覧（パスアラウンド）に回すことで、会議のための拘束時間を削減し、設計判断の議論にミーティングの時間を集中させる。

※ この軸は **軸0で④人間・チームレビューに分類された項目** に対して適用する（①〜③で完結する項目は対象外）。

### 判断基準

各項目・各レビュー対象を以下の問いで判定する。

#### 問い①: リアルタイムの双方向対話・その場での合意形成が必要か？

> 「指摘とその応答を往復しながら、その場で結論を出す必要があるか？」

- **Yes（双方向の議論が要る）** → 同期（ミーティング/モブレビュー）
- **No（一方向のコメントで足りる）** → パスアラウンド（非同期回覧）

#### 問い②: 参加者間で前提認識がずれており、背景説明や認識合わせが要るか？

> 「文章コメントだけでは前提が伝わらず、口頭での補足が必要か？」

- **Yes（認識合わせが必要）** → 同期
- **No（書けば伝わる）** → パスアラウンド

#### 補助基準: 影響範囲の広さ・手戻りコスト

影響範囲が広く、後戻りのコストが大きいものほど同期寄り。局所的で手戻りが軽いものはパスアラウンドで足りる。

### 分類カテゴリ

| 分類 | 性質 | 該当しそうな例 |
|---|---|---|
| **同期**（ミーティング/モブ） | 双方向の議論・合意形成・前提共有が必要 | Aggregate粒度やトランザクション境界の設計判断、エラーハンドリング戦略、影響範囲の広いインターフェース設計 |
| **パスアラウンド**（非同期回覧） | 一方向のフィードバックで足り、書けば伝わる | 命名の妥当性確認、軽微な設計の追認、限定的な範囲のレビュー |

### チームで決めるべき論点

- パスアラウンドの **回覧人数**（1人で足りるか、複数の承認が要るか）
- 回覧の **SLA（期限）**（何営業日以内に返すか）
- **承認方法**（ツール上のApprove運用、コメントなしの黙示承認を認めるか 等）

## 5. 軸2: 指摘の重大度・対応強制度（[必須] / [推奨] / [提案]）

### 背景・目的

指摘の重みが人によって揃っていないと、「品質のばらつき」と「軽微な指摘による修正サイクルの肥大」が起きる。指摘に重大度ラベルを付け、それぞれの対応強制度を標準化することで、提出者は何を必ず直すべきかが一目で分かり、レビュアー間の判断もそろう。

※ この軸は **レビューで出た個々の指摘** に対して付与する（軸0・軸1が「項目をどう扱うか」なのに対し、本軸は「出てきた指摘をどう扱うか」）。

### 判断基準

各指摘を以下の問いで判定する。

#### 問い①: 修正しないとマージをブロックすべきか？

> 「これを直さないままマージしてよいか？」

- **No（マージしてはいけない）** → [必須]（Must）
- **Yes（マージはできるが直したほうがよい/任意）** → [推奨]（Should）または [提案]（Nits）（問い②で切り分け）

#### 問い②: 修正しないと何が起きるか（リスクの大きさ）

> 「放置した場合の影響はどの程度か？」

- **バグ・ルール違反・設計の致命的欠陥** → [必須]（Must）
- **改善の余地はあるが、放置しても直ちに害はない** → [推奨]（Should）
- **好み・スタイルの範囲で、品質への実害がない** → [提案]（Nits）

### 分類カテゴリ

ラベルの定義は [ソフトウェアレビュー標準 §1.7](../ソフトウェアレビュー標準.md) に従う（本書は付与の判断基準を補足する）。

| ラベル | 英語 | 意味 | 対応強制度 |
|---|---|---|---|
| **[必須]** | Must（ブロッカー） | ルール違反・バグ・設計の致命的欠陥 | マージ前に必須修正 |
| **[推奨]** | Should | 改善すべきだが、理由があれば見送り可 | 対応する、または見送る理由を記録 |
| **[提案]** | Nits（任意/好み） | 軽微・スタイル・好みの範囲 | 提出者の裁量 |

### チームで決めるべき論点

- **[必須] と [推奨] の線引き**（どこからを「マージブロック」とするかの具体例の合意）
- ラベルの **記法**（例: コメント先頭に `[必須]` / `[推奨]` / `[提案]` を付ける）
- [推奨] を見送る場合の **記録ルール**（理由をどこに残すか）

## 6. 軸の合成と運用イメージ

1つの観点・指摘は、複数の軸が重なってラベリングされる。

**例1: 「1トランザクションの粒度が適切か」（詳細設計）**
- 軸0 → ④人間・チームレビュー（正解が一意に決まらない設計判断。AIは設計書との差分提示で補助）
- 軸1 → 同期（双方向の議論・合意形成が必要）
- 軸2 → 指摘が出れば多くは [必須]〜[推奨]（境界の誤りは手戻りが大きい）

**例2: 「メソッド名がユビキタス言語として適切か」（詳細設計/コード）**
- 軸0 → ②生成AIレビューで一次評価（用語集との照合）→ ④人間・チームレビューで最終合意（命名の妥当性は主観が入る）
- 軸1 → パスアラウンド（一方向のコメントで足りることが多い）
- 軸2 → 多くは [推奨]〜[提案]（合意があれば改善、好みの範囲なら任意）

**例3: 「domain層にORMのimportがない」（コード）**
- 軸0 → ①ツール自動化（eslint-plugin-boundaries 等で機械的に検出。チェックリストにも載せない）
- 軸1・軸2 → 適用外（人間レビューに上げない）

これらの軸はいずれも **詳細設計・コード・テストの3工程に共通** して適用する。各工程の観点を仕分ける際は、まず軸0（§3の4層）で確認の担い手を決め、④人間・チームレビューとなったものに軸1・軸2を重ねて運用方法を定める。
