# レビュー観点チェックリスト

本資料は [ソフトウェアレビュー標準](../ソフトウェアレビュー標準.md) から参照される、工程別のレビュー観点チェックリストである。
各レビューで「重点的に確認する観点」を洗い出すための汎用リストとして用いる。

> DDDに特化した実務上のチェック項目（事前自己チェック／チームレビュー）は、各工程のガイドラインを参照する。
> - [詳細設計レビューガイドライン](../詳細設計レビュー/詳細設計レビューガイドライン.md)
> - [コードレビューガイドライン](../コードレビュー/コードレビューガイドライン.md)
> - [テストレビューガイドライン](../テストレビュー/テストレビューガイドライン.md)

---

## 1. 詳細設計レビュー チェックリスト

### 要件・基本設計との整合性

- [ ] 要件定義の内容と矛盾していない。
- [ ] 基本設計の内容と矛盾していない。
- [ ] 画面仕様、API仕様、DB設計と整合している。
- [ ] 業務ルールが詳細設計に反映されている。
- [ ] 未決事項や前提条件が明記されている。

### 処理仕様

- [ ] 正常系の処理フローが明確である。
- [ ] 異常系の処理フローが明確である。
- [ ] 分岐条件が明確である。
- [ ] 入力値、出力値、戻り値が明確である。
- [ ] 外部システム連携の仕様が明確である。
- [ ] バッチ処理や非同期処理の開始条件、終了条件が明確である。

### 責務分担

- [ ] クラス、関数、モジュールの責務が明確である。
- [ ] 1つの部品に複数の責務が詰め込まれていない。
- [ ] 共通化すべき処理と個別処理が整理されている。
- [ ] 呼び出し関係が分かりやすい。
- [ ] 将来変更が想定される箇所が局所化されている。

### データ設計

- [ ] 使用するテーブル、カラムが明確である。
- [ ] 型、桁数、NULL可否、初期値が明確である。
- [ ] コード値やステータス値の意味が明確である。
- [ ] 登録、更新、削除、参照の条件が明確である。
- [ ] データの整合性を保つ仕組みが考慮されている。

### 例外処理・エラー処理

- [ ] 入力エラー時の処理が明確である。
- [ ] 外部システムエラー時の処理が明確である。
- [ ] DBエラー時の処理が明確である。
- [ ] タイムアウト時の処理が明確である。
- [ ] エラー時の戻り値、画面表示、ログ出力が明確である。
- [ ] リトライやロールバックの方針が明確である。

### 境界条件

- [ ] 0件の場合の動作が定義されている。
- [ ] NULL、空文字、未設定の場合の動作が定義されている。
- [ ] 最大件数、最大文字数、最大桁数の場合の動作が定義されている。
- [ ] 日付境界、月末、年度末などの動作が定義されている。
- [ ] 重複データが存在する場合の動作が定義されている。

### 状態管理・排他制御

- [ ] ステータス遷移が明確である。
- [ ] 二重実行時の動作が定義されている。
- [ ] 同時更新時の排他制御が考慮されている。
- [ ] 再実行時の動作が定義されている。
- [ ] 処理途中で失敗した場合の状態が定義されている。

### 非機能要件

- [ ] 性能要件を満たせる設計になっている。
- [ ] セキュリティ要件を満たせる設計になっている。
- [ ] ログ出力方針が明確である。
- [ ] 運用監視に必要な情報が出力される。
- [ ] 保守しやすい構造になっている。

### テスト容易性

- [ ] 詳細設計から単体テストケースを作成できる。
- [ ] 詳細設計から結合テストケースを作成できる。
- [ ] 正常系、異常系、境界値のテスト観点が読み取れる。
- [ ] テストデータを準備できる粒度で仕様が記載されている。

## 2. コードレビュー チェックリスト

### 設計との整合性

- [ ] 詳細設計どおりに実装されている。
- [ ] 設計との差分がある場合、理由が説明されている。
- [ ] 処理フローが設計と一致している。
- [ ] 入出力仕様が設計と一致している。
- [ ] DB操作や外部連携が設計と一致している。

### 正確性

- [ ] 正常系が正しく動作する。
- [ ] 異常系が正しく動作する。
- [ ] 分岐条件に誤りがない。
- [ ] 境界値で誤動作しない。
- [ ] NULL、空文字、0件の場合に誤動作しない。
- [ ] 型変換、桁数、丸め処理に誤りがない。

### 可読性

- [ ] 処理の流れが読みやすい。
- [ ] 変数名、関数名、クラス名が分かりやすい。
- [ ] 複雑すぎる条件分岐がない。
- [ ] 不要なコメントや不足しているコメントがない。
- [ ] マジックナンバーや意味不明な定数がない。

### 保守性

- [ ] 重複コードが少ない。
- [ ] 変更が局所化されている。
- [ ] 責務が適切に分割されている。
- [ ] 共通処理と個別処理が整理されている。
- [ ] 将来の仕様変更に対応しやすい。

### 例外処理・ログ

- [ ] 例外を握りつぶしていない。
- [ ] エラー時の戻り値やレスポンスが適切である。
- [ ] 必要なログが出力されている。
- [ ] 機密情報をログに出力していない。
- [ ] 障害調査に必要な情報がログに含まれている。

### トランザクション・整合性

- [ ] トランザクション範囲が適切である。
- [ ] エラー時にロールバックされる。
- [ ] 部分更新による不整合が発生しない。
- [ ] 同時実行時の不整合が考慮されている。
- [ ] 再実行時に重複登録や二重処理が発生しない。

### 性能

- [ ] ループ内で不要なDBアクセスをしていない。
- [ ] N+1問題が発生していない。
- [ ] 大量データ処理時の性能が考慮されている。
- [ ] 不要なメモリ使用がない。
- [ ] タイムアウトやリトライが考慮されている。

### セキュリティ

> 静的解析ツール（例: Coverity）で検出できる項目は自動化を優先する。

- [ ] 入力値チェックが適切である。
- [ ] 認証・認可のチェックが適切である。
- [ ] SQLインジェクション対策がされている。
- [ ] XSSなどの脆弱性対策がされている。
- [ ] 秘密情報をソースコードに直接記載していない。

### テスト

- [ ] 単体テストが実施されている。
- [ ] 主要な正常系がテストされている。
- [ ] 主要な異常系がテストされている。
- [ ] 境界値がテストされている。
- [ ] 修正箇所に対する確認テストが実施されている。

## 3. テストレビュー チェックリスト

### 要件・設計との整合性

- [ ] 要件を確認できるテストケースになっている。
- [ ] 基本設計を確認できるテストケースになっている。
- [ ] 詳細設計を確認できるテストケースになっている。
- [ ] 重要な業務ルールを確認できる。
- [ ] テスト対象外の範囲が明記されている。

### テスト観点の網羅性

- [ ] 正常系が網羅されている。
- [ ] 異常系が網羅されている。
- [ ] 境界値が網羅されている。
- [ ] 0件、NULL、空文字のケースが含まれている。
- [ ] 権限違い、入力不備、外部連携エラーが考慮されている。
- [ ] 状態遷移や再実行のケースが含まれている。

### テストケースの明確性

- [ ] テスト目的が明確である。
- [ ] 前提条件が明確である。
- [ ] 操作手順が明確である。
- [ ] 入力データが明確である。
- [ ] 期待結果が明確である。
- [ ] 合否判定が可能な内容になっている。

### テストデータ

- [ ] 正常系のデータが用意されている。
- [ ] 異常系のデータが用意されている。
- [ ] 境界値のデータが用意されている。
- [ ] データの作成方法が明確である。
- [ ] テスト後のデータ状態が考慮されている。

### テスト実行性

- [ ] 実行手順が具体的である。
- [ ] 実行環境が明確である。
- [ ] 必要な事前準備が明確である。
- [ ] テスト順序に依存がある場合、その依存関係が明確である。
- [ ] 再実行できる内容になっている。

### 品質保証の妥当性

- [ ] 重要機能に対して十分なテストがある。
- [ ] 障害が発生しやすい箇所が重点的に確認されている。
- [ ] 過去不具合や類似不具合が考慮されている。
- [ ] リスクの高い箇所に対してテストが厚くなっている。
- [ ] テスト完了後に品質を説明できる内容になっている。
