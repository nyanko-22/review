# レビュー分類方針 総覧

## 1. このドキュメントの位置づけ

既存の仕分け方針（[詳細設計](レビュー観点仕分け方針.md) / [コード](コードレビュー観点仕分け方針.md) / [テスト](テストレビュー観点仕分け方針.md)）は、各チェック項目を **「誰が確認するか」=事前自己チェック vs チームレビュー** という1つの軸で仕分けている。

しかし、レビューを効率化・標準化するための分類軸はこの1本だけではない。本書は、その軸に乗らない **横断的な分類軸を束ねる総覧** であり、3工程（詳細設計・コード・テスト）すべてに共通して適用する。

**使い方**:  
本書は分類軸ごとの「判断方針（決め方の基準）」を定義する。実際に「どの観点・どの指摘がどの分類に入るか」は、この方針をもとにメンバーと討議して決める。本書はその **討議のたたき台／アジェンダ** として用いる。各軸の末尾に「チームで決めるべき論点」を明記しているので、討議ではそこを起点にする。

---

## 2. 分類軸の全体像（洗い出し）

レビュー対象は、以下の独立した軸で多面的に分類できる。各軸は直交しており、合成して使う（**WHO → HOW → 指摘の重み** の順に重ねる）。

| 軸 | 問い（何を分けるか） | 分類 | 状態 |
|---|---|---|---|
| **軸0: 確認主体**（誰が確認するか） | 答えが一意か／単独で確認できるか | 事前自己チェック / チームレビュー（/ CI自動化） | 既存・参照 |
| **軸1: レビュー実施方式**（どう確認するか） | リアルタイムの双方向対話が必要か | 同期（ミーティング/モブ） / パスアラウンド（非同期回覧） | 本書で定義 |
| **軸2: 指摘の重大度**（指摘をどう扱うか） | マージをブロックすべきか | Must / Should / Nits | 本書で定義 |

```
レビュー対象の1項目
  ├─ 軸0: 誰が確認？  → 自己チェック で完結 / チームレビュー へ
  │                                          ↓ チームレビューなら
  ├─ 軸1: どう確認？  → 同期で議論 / パスアラウンドで回覧
  │                                          ↓ 指摘が出たら
  └─ 軸2: その指摘の重み？ → Must / Should / Nits
```

- **軸0** は既存の仕分け方針で定義済み。本書では再定義せず、出発点として参照する（判断基準は「問い①: 答えが一意に決まるか」「問い②: 提出者本人が単独で確認できるか」）。
- **軸1・軸2** を本書で新たに定義する。

---

## 3. 軸1: レビュー実施方式（同期 vs パスアラウンド）

### 背景・目的

チームレビュー対象であっても、すべてを集まって議論する必要はない。一方向のフィードバックで足りるものを非同期の回覧（パスアラウンド）に回すことで、会議のための拘束時間を削減し、設計判断の議論にミーティングの時間を集中させる。

※ この軸は **軸0でチームレビューに分類された項目** に対して適用する（自己チェックで完結する項目は対象外）。

### 判断基準

各項目・各レビュー対象を以下の問いで判定する。

#### 問い①: リアルタイムの双方向対話・その場での合意形成が必要か？

> 「指摘とその応答を往復しながら、その場で結論を出す必要があるか？」

- **Yes（双方向の議論が要る）** → 同期（ミーティング/モブレビュー）
- **No（一方向のコメントで足りる）** → パスアラウンド（非同期回覧）

#### 問い②: 参加者間で前提認識がずれており、背景説明や認識合わせが要るか？

> 「文章コメントだけでは前提が伝わらず、口頭での補足が必要か？」

- **Yes（認識合わせが必要）** → 同期
- **No（書けば伝わる）** → パスアラウンド

#### 補助基準: 影響範囲の広さ・手戻りコスト

影響範囲が広く、後戻りのコストが大きいものほど同期寄り。局所的で手戻りが軽いものはパスアラウンドで足りる。

### 分類カテゴリ

| 分類 | 性質 | 該当しそうな例 |
|---|---|---|
| **同期**（ミーティング/モブ） | 双方向の議論・合意形成・前提共有が必要 | Aggregate粒度やトランザクション境界の設計判断、エラーハンドリング戦略、影響範囲の広いインターフェース設計 |
| **パスアラウンド**（非同期回覧） | 一方向のフィードバックで足り、書けば伝わる | 命名の妥当性確認、軽微な設計の追認、限定的な範囲のレビュー |

### チームで決めるべき論点

- パスアラウンドの **回覧人数**（1人で足りるか、複数の承認が要るか）
- 回覧の **SLA（期限）**（何営業日以内に返すか）
- **承認方法**（ツール上のApprove運用、コメントなしの黙示承認を認めるか 等）

## 4. 軸2: 指摘の重大度・対応強制度（Must / Should / Nits）

### 背景・目的

指摘の重みが人によって揃っていないと、「品質のばらつき」と「軽微な指摘による修正サイクルの肥大」が起きる。指摘に重大度ラベルを付け、それぞれの対応強制度を標準化することで、提出者は何を必ず直すべきかが一目で分かり、レビュアー間の判断もそろう。

※ この軸は **レビューで出た個々の指摘** に対して付与する（軸0・軸1が「項目をどう扱うか」なのに対し、本軸は「出てきた指摘をどう扱うか」）。

### 判断基準

各指摘を以下の問いで判定する。

#### 問い①: 修正しないとマージをブロックすべきか？

> 「これを直さないままマージしてよいか？」

- **No（マージしてはいけない）** → Must
- **Yes（マージはできるが直したほうがよい/任意）** → Should または Nits（問い②で切り分け）

#### 問い②: 修正しないと何が起きるか（リスクの大きさ）

> 「放置した場合の影響はどの程度か？」

- **バグ・ルール違反・設計の致命的欠陥** → Must
- **改善の余地はあるが、放置しても直ちに害はない** → Should
- **好み・スタイルの範囲で、品質への実害がない** → Nits

### 分類カテゴリ

| ラベル | 意味 | 対応強制度 |
|---|---|---|
| **Must**（ブロッカー） | ルール違反・バグ・設計の致命的欠陥 | マージ前に必須修正 |
| **Should**（推奨） | 改善すべきだが、理由があれば見送り可 | 対応する、または見送る理由を記録 |
| **Nits**（任意/好み） | 軽微・スタイル・好みの範囲 | 提出者の裁量 |

### チームで決めるべき論点

- **Must と Should の線引き**（どこからを「マージブロック」とするかの具体例の合意）
- ラベルの **記法**（例: コメント先頭に `[Must]` / `[Should]` / `[Nits]` を付ける）
- Should を見送る場合の **記録ルール**（理由をどこに残すか）

## 5. 軸の合成と運用イメージ

1つの観点・指摘は、複数の軸が重なってラベリングされる。

**例1: 「1トランザクションの粒度が適切か」（詳細設計）**
- 軸0 → チームレビュー（正解が一意に決まらない設計判断）
- 軸1 → 同期（双方向の議論・合意形成が必要）
- 軸2 → 指摘が出れば多くは Must〜Should（境界の誤りは手戻りが大きい）

**例2: 「メソッド名がユビキタス言語として適切か」（詳細設計/コード）**
- 軸0 → チームレビュー（命名の妥当性は主観が入る）
- 軸1 → パスアラウンド（一方向のコメントで足りることが多い）
- 軸2 → 多くは Should〜Nits（合意があれば改善、好みの範囲なら任意）

これらの軸はいずれも **詳細設計・コード・テストの3工程に共通** して適用する。各工程の観点を仕分ける際は、まず軸0（既存の仕分け方針）で確認主体を決め、チームレビューとなったものに軸1・軸2を重ねて運用方法を定める。


## 補足

章構成・文体: 既存の仕分け方針と同じ「## N. なぜ〜」「問い①②形式」「分類カテゴリ表」を踏襲。
軸一覧と詳細の整合: 第2章の軸0/1/2と、第3〜4章の詳細が対応。
討議のたたき台: 軸1・軸2それぞれに「チームで決めるべき論点」を明記。
新規追加のみ: 既存ファイルは未変更（新規1ファイルのみ追加）。

仕上がりの要点
軸0（確認主体） は既存方針の参照に留め、本書では 軸1（同期 vs パスアラウンド） と 軸2（Must/Should/Nits） を新規定義。
3軸は WHO → HOW → 指摘の重み の順に合成して使うものとして、第5章で具体例2件（トランザクション粒度／命名の妥当性）を示しました。
メンバー討議では、各軸末尾の「チームで決めるべき論点」（回覧人数・SLA・承認方法／Must-Should の線引き・ラベル記法）が起点になります。
次の議論で「軸1・軸2に沿って各観点を実際に仕分ける」段階に入ったら、既存3本の仕分け表にこの2軸の列を足していく形が自然です。必要になったら声をかけてください
