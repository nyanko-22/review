# テストレビューガイドライン

## 1. このガイドラインについて

### 目的
テストレビューは、**テストコードが実装の品質を正しく保証できる内容・構造になっているかを確認する**ことを目的とする。  
テストはバグを防ぐだけでなく、将来のリファクタリングを安全に行うための安全網であり、ビジネス仕様の「生きたドキュメント」でもある。このガイドラインによって、テスト自体の品質ばらつきを解消する。

### 対象工程
- **対象**: テストコード実装完了後・マージ前
- **対象読者**: レビュアー（テストを評価する側）・レビュイー（テストを提出する側）の両方

### アーキテクチャ前提
本プロジェクトは **ドメイン駆動設計（DDD）** を採用している。テストレビューでは、DDDの各層に応じた適切なテスト戦略（単体/統合/E2E）が選択されているかを重点的に確認する。

```
[DDDのテスト戦略]
Domain層（Entity / ValueObject / Aggregate / DomainService）
  → モックなし単体テスト（純粋なビジネスロジックを検証）

Repository（Infrastructure層）
  → インメモリ実装を使った単体テスト（DBモックは避ける）
  → または、テスト用DBを使った統合テスト

Application Service
  → Repositoryをインメモリ実装に差し替えたテスト
  → 外部依存（メール送信・外部API等）のみモック

Presentation層
  → APIレベルの統合テスト or E2Eテスト
```

---

## 2. レビュースコープ

「何を見るか」と同時に「何を見ないか」を明確にする。スコープ外の指摘はレビュー時間を無駄に消費する。

| 対象（見る） | 対象外（見ない） |
|---|---|
| テストケースの網羅性（正常系・異常系・境界値） | テスト対象コードの品質（コードレビュー済み） |
| テストの独立性・再現性 | テストフレームワーク・テストツールの設定 |
| テストの可読性・構造（AAA等） | CI/CDパイプラインの設定 |
| テストの粒度・モック戦略（DDDに基づく） | パフォーマンステスト・負荷テストの設計 |
| DDDとの整合性（ドメインルールの単体テスト） | テストデータの量（最小限であれば十分） |

---

## 3. 役割と責任

レビュアー・レビューイーの役割は [ソフトウェアレビュー標準 §1.5](../ソフトウェアレビュー標準.md) を正典とする。本工程（テストレビュー）での要点は次のとおり。

| 役割 | 本工程での要点 |
|---|---|
| **レビュイー** | テストコードを提出し、指摘に対して「修正」または「見送り理由」を回答する |
| **レビュアー** | テスト観点でフィードバックを行い、指摘には必ず重大度ラベル（[必須]/[推奨]/[提案]）を明示する |

> **重要**: レビュアーは重大度ラベルを明示せずに指摘してはいけない。レビューイーは重大度が明示されていない指摘への対応は不要とする（[標準 §1.7](../ソフトウェアレビュー標準.md)）。

## 4. レビュー観点

### 観点A: テストケースの網羅性

**なぜこれを見るか**  
テストの網羅性が不十分だと、テストが通っても本番でバグが出る。特に異常系・境界値・ドメインルール違反のケースは見落としやすく、これらが未テストのまま本番に出ることが多い。

**チェック内容**
- 正常系（ハッピーパス）が全ユースケースをカバーしているか
- 異常系（バリデーションエラー、存在しないID、権限エラー等）がテストされているか
- 境界値（最大値・最小値・空文字・ゼロ等）がテストされているか
- ドメインルール違反のケース（例: 在庫0の商品を注文、無効なメール形式等）がテストされているか

---

### 観点B: テストの独立性・再現性

**なぜこれを見るか**  
テストが他のテストの状態に依存していると、実行順序によって結果が変わり「CIでは落ちるのにローカルでは通る」問題が発生する。信頼性のないテストは、あるよりも有害になりうる。

**チェック内容**
- `beforeEach` / `afterEach` でテストデータが毎回初期化されているか
- テスト間でグローバル変数・モジュールレベルの状態を共有していないか
- `Date.now()` / `new Date()` / `Math.random()` 等の非決定的な処理がモックまたは固定値で制御されているか
- テストが特定の実行順序に依存していないか（ランダム順でも通るか）

---

### 観点C: テストの可読性

**なぜこれを見るか**  
テストは「仕様の生きたドキュメント」でもある。テスト名と構造から「何がどう動くべきか」が読み取れることで、仕様変更時の修正箇所が明確になり、新規メンバーのオンボーディングにも役立つ。

**チェック内容**
- テスト名が仕様・振る舞いを表しているか
  - 良い例: `「在庫が0の場合、注文は失敗する」` / `should throw InsufficientStockError when stock is zero`
  - 悪い例: `test1` / `注文テスト`
- Arrange（テストデータの準備）/ Act（テスト対象の実行）/ Assert（結果の検証）の3ブロックが明確に分かれているか
- 1つのテストメソッドで検証する振る舞いが1つのみか（複数のassertはテスト分割の候補）
- テストデータが最小限か（意図を理解するために不要なフィールドが多すぎないか）

---

### 観点D: テストの粒度・モック戦略

**なぜこれを見るか**  
DDDでは層ごとに適切なテスト戦略が異なる。誤ったモック戦略は「テストが通るが本番で動かない」状況を作り出す。特に「Repositoryをモックでテストする」のはDDDのアンチパターンで、実際のデータアクセスロジックが未検証になる。

**チェック内容**
- ドメイン層（Entity / ValueObject / DomainService）のテストがモックなしで動いているか（外部依存なしに単体で動くはず）
- Repositoryのテストがインメモリ実装（またはテスト用DB）を使っているか（DBをモックしていないか）
- Application Serviceのテストで、メール送信・外部API等の副作用のある外部依存のみがモックされているか
- E2Eテストの範囲が適切か（全ケースをE2Eで検証しようとしていないか。正常系の主要フローのみをE2Eに限定しているか）

---

### 観点E: DDDとの整合性

**なぜこれを見るか**  
DDDで最も重要なのはドメインロジックの正確性。テストがインフラ層・Application層に偏っていて、ドメイン層の単体テストが手薄だと、DDDとして設計した価値が半減する。

**チェック内容**
- ドメインルール（Aggregateの整合性制約、Value Objectのバリデーション等）が単体テストで検証されているか
- テスト名がユビキタス言語を使っており、ビジネス仕様のドキュメントとして読めるか
- Value Objectのファクトリメソッドに対して、正常系・異常系のテストがあるか

---

## 5. 指摘の重要度分類

レビュアーは全ての指摘に、標準で定義する重大度ラベル（[ソフトウェアレビュー標準 §1.7](../ソフトウェアレビュー標準.md)）を必ず付ける。本工程での具体的な目安は次のとおり。

| ラベル | 英語 | 本工程での目安 | レビューイーの対応 |
|---|---|---|---|
| **[必須]** | Must | テストの信頼性を損なう（テスト間依存、非決定的テスト）、重大なケースが未テスト | **必ずFix** してから次工程へ進む |
| **[推奨]** | Should | テストの可読性・網羅性・保守性の向上に有効。対応しないリスクがある | Fixするか、**見送り理由を明示**して合意を得る |
| **[提案]** | Nits | よりよいテストの可能性がある任意の改善案 | 対応は任意。返答不要 |

**記載例**
```
[必須] Date.now()が固定されていないため、実行日時によってテスト結果が変わります。
[推奨] バリデーションエラーのケース（空文字・null）のテストが不足しています。
[提案] テスト名をユビキタス言語に合わせると仕様書としての価値が上がります。
```

## 6. 完了基準（Exit Criteria）

完了の基本的な考え方は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)（レビューの進め方）に準拠する。本工程では以下の条件を全て満たした時点でレビュー完了とする。

- [ ] **[必須]** の指摘が全て解消されている
- [ ] **[推奨]** の指摘が全て「修正済み」または「見送り理由あり」の状態になっている
- [ ] レビュアーが承認コメントを記録している（例: `LGTM` / `承認`）

> **[推奨] の見送り理由の例**
> - 「今フェーズのスコープ外のため次スプリントのタスクに追加しました（チケット#XXX）」
> - 「境界値テストの追加は工数対リスクを検討し、次のイテレーションで対応します」

## 7. レビュー実施方式

レビュー指摘数を減らしてレビュー時間を圧縮するため、チェック項目を以下2トラックに分ける。

| トラック | 実施者 | タイミング | 対象項目の性質 |
|---|---|---|---|
| **事前自己チェック** | レビュイー（提出者）本人 | PRオープン前 | ルールが明確で答えが一意に決まる。テストコードを見れば客観的に確認できる |
| **チームレビュー** | チーム全員 | PRレビュー時 | 網羅性の妥当性・モック戦略・粒度の判断が必要。人によって意見が変わりうる |

> **運用ルール**: 事前自己チェックの全項目が ✅ になっていないテストコードはPRとして提出しない。

---

## 7-1. 事前自己チェックリスト（PR提出前に各自が確認）

PR提出前にレビュイー本人が確認する。**全項目が ✅ になってからPRを提出する。**

### A. テストケースの網羅性

- [ ] 正常系（ハッピーパス）のテストが存在する

### B. テストの独立性・再現性

- [ ] `beforeEach` / `afterEach` でテストデータが毎回初期化されている
- [ ] `Date.now()` / `new Date()` / `Math.random()` 等がモックまたは固定値で制御されている
- [ ] テスト間でグローバル状態を共有していない

### C. テストの可読性

- [ ] テスト名が「何をテストするか（振る舞い・仕様）」を表している
- [ ] Arrange / Act / Assert の3ブロックが分かれている
- [ ] 1テストメソッドで検証する振る舞いが1つのみ

### D. テストの粒度・モック戦略

- [ ] domain層のテストがモックなしで動いている（外部依存なし）
- [ ] Repositoryのテストがインメモリ実装またはテスト用DBを使っている（DBをモックしていない）

### E. DDDとの整合性

- [ ] Value Objectのファクトリメソッドに対して正常系・異常系のテストがある
- [ ] ドメインルール違反ケースのテストが1件以上存在する

---

## 7-2. チームレビューチェックリスト（チームで議論して確認）

チームレビュー時にレビュアーが確認する。判断が分かれる可能性があるため、**チームで合意を形成する**。

### A. テストケースの網羅性

- [ ] 異常系・境界値ケースが十分か（何が「十分」かはビジネス文脈の判断）  
  *（網羅の基準はリスクとコストのトレードオフ。チームで合意する）*
- [ ] ドメインルール違反ケースがビジネス的に十分なケース数か  
  *（どのケースをテストすべきかは、ビジネスロジックの理解が必要）*

### D. テストの粒度・モック戦略

- [ ] モック戦略が適切か（モックしすぎて実際の動作を検証できていないケースがないか）  
  *（何をモックするかはトレードオフ。過剰なモックはテストの価値を下げる）*
- [ ] テストの粒度（単体/統合/E2Eの使い分け）が適切か  
  *（どの層をどの粒度でテストするかは設計判断）*
- [ ] テスト自体の複雑さが適切か（テストの保守コストが見合っているか）  
  *（複雑なテストはテスト対象コードと同等の保守コストがかかる）*

### E. DDDとの整合性

- [ ] テスト全体として、ドメイン層のテストが十分に厚いか  
  *（ドメイン層のテストが薄く、Application/Infrastructure層に偏っていないか）*

---

## 付録: DDDのテスト戦略クイックリファレンス

| 層 | テスト種別 | モック方針 | 重点確認事項 |
|---|---|---|---|
| **Domain層** | 単体テスト | モックなし | ビジネスルール・バリデーション・状態遷移 |
| **Repository** | 単体 or 統合 | インメモリ実装 | CRUD操作・検索条件の正確性 |
| **Application Service** | 単体テスト | Repo=インメモリ、外部API=モック | ユースケースの正常系・異常系 |
| **Presentation** | 統合 or E2E | なし | リクエスト/レスポンス形式・認証 |
