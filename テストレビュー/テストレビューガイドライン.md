# テストレビューガイドライン

## 1. このガイドラインについて

### 目的
テストレビューは、**テストコードが実装の品質を正しく保証できる内容・構造になっているかを確認する**ことを目的とする。  
テストはバグを防ぐだけでなく、将来のリファクタリングを安全に行うための安全網であり、ビジネス仕様の「生きたドキュメント」でもある。このガイドラインによって、テスト自体の品質ばらつきを解消する。

### 対象工程
- **対象**: テストコード実装完了後・マージ前
- **対象読者**: レビュアー（テストを評価する側）・レビューイー（テストを提出する側）の両方

### アーキテクチャ前提
本プロジェクトは **ドメイン駆動設計（DDD）** を採用している。テストレビューでは、DDDの各層に応じた適切なテスト戦略（単体/統合/E2E）が選択されているかを重点的に確認する。

```
[DDDのテスト戦略]
Domain層（Entity / ValueObject / Aggregate / DomainService）
  → モックなし単体テスト（純粋なビジネスロジックを検証）

Repository（Infrastructure層）
  → インメモリ実装を使った単体テスト（DBモックは避ける）
  → または、テスト用DBを使った統合テスト

Application Service
  → Repositoryをインメモリ実装に差し替えたテスト
  → 外部依存（メール送信・外部API等）のみモック

Presentation層
  → APIレベルの統合テスト or E2Eテスト
```

---

## 2. レビュースコープ

「何を見るか」と同時に「何を見ないか」を明確にする。スコープ外の指摘はレビュー時間を無駄に消費する。

| 対象（見る） | 対象外（見ない） |
|---|---|
| テストケースの網羅性（正常系・異常系・境界値） | テスト対象コードの品質（コードレビュー済み） |
| テストの独立性・再現性 | テストフレームワーク・テストツールの設定 |
| テストの可読性・構造（AAA等） | CI/CDパイプラインの設定 |
| テストの粒度・モック戦略（DDDに基づく） | パフォーマンステスト・負荷テストの設計 |
| DDDとの整合性（ドメインルールの単体テスト） | テストデータの量（最小限であれば十分） |

---

## 3. 役割と責任

レビュアー・レビューイーの役割は [ソフトウェアレビュー標準 §1.5](../ソフトウェアレビュー標準.md) に従う（重大度ラベルの明示義務と、未明示の指摘は対応不要とする運用は [標準 §1.4](../ソフトウェアレビュー標準.md)）。本工程（テストレビュー）での要点は次のとおり。

| 役割 | 本工程での要点 |
|---|---|
| **レビューイー** | テストコードを提出し、指摘に対して「修正」または「見送り理由」を回答する |
| **レビュアー** | テスト観点でフィードバックを行い、指摘には必ず重大度ラベル（[必須]/[推奨]/[提案]）を明示する |

---

## 4. レビュー観点

本節は各観点の **「なぜ見るか」と要点** を示す。具体的な確認項目は §7-1（提出前）/§7-2（チームレビュー）のチェックリスト、各項目の軸0分類と分類理由は [テストレビュー観点仕分け方針 §3](テストレビュー観点仕分け方針.md) にある。

### 観点A: テストケースの網羅性

**なぜこれを見るか**  
テストの網羅性が不十分だと、テストが通っても本番でバグが出る。特に異常系・境界値・ドメインルール違反のケースは見落としやすく、これらが未テストのまま本番に出ることが多い。

正常系（ハッピーパス）が全ユースケースをカバーし、異常系（バリデーションエラー、存在しないID、権限エラー等）・境界値（最大/最小・空文字・ゼロ等）・ドメインルール違反のケース（例: 在庫0の商品を注文、無効なメール形式）がテストされているかを確認する。

---

### 観点B: テストの独立性・再現性

**なぜこれを見るか**  
テストが他のテストの状態に依存していると、実行順序によって結果が変わり「CIでは落ちるのにローカルでは通る」問題が発生する。信頼性のないテストは、あるよりも有害になりうる。

`beforeEach`/`afterEach` でテストデータが毎回初期化され、テスト間でグローバル変数・モジュールレベルの状態を共有せず、非決定的な処理（`Date.now()`/`new Date()`/`Math.random()` 等）がモックまたは固定値で制御され、特定の実行順序に依存しない（ランダム順でも通る）かを確認する。

---

### 観点C: テストの可読性

**なぜこれを見るか**  
テストは「仕様の生きたドキュメント」でもある。テスト名と構造から「何がどう動くべきか」が読み取れることで、仕様変更時の修正箇所が明確になり、新規メンバーのオンボーディングにも役立つ。

テスト名が仕様・振る舞いを表し（良い例: `「在庫が0の場合、注文は失敗する」` / `should throw InsufficientStockError when stock is zero`、悪い例: `test1` / `注文テスト`）、Arrange/Act/Assertの3ブロックが分かれ、1テストで検証する振る舞いが1つのみで、テストデータが最小限かを確認する。

---

### 観点D: テストの粒度・モック戦略

**なぜこれを見るか**  
DDDでは層ごとに適切なテスト戦略が異なる。誤ったモック戦略は「テストが通るが本番で動かない」状況を作り出す。特に「Repositoryをモックでテストする」のはDDDのアンチパターンで、実際のデータアクセスロジックが未検証になる。

ドメイン層（Entity/ValueObject/DomainService）のテストがモックなしで動き（外部依存なしに単体で動くはず）、Repositoryのテストがインメモリ実装（またはテスト用DB）を使い（DBをモックしない）、Application Serviceのテストで副作用のある外部依存（メール送信・外部API等）のみがモックされ、E2Eの範囲が適切（全ケースをE2Eにせず正常系の主要フローに限定）かを確認する。

---

### 観点E: DDDとの整合性

**なぜこれを見るか**  
DDDで最も重要なのはドメインロジックの正確性。テストがインフラ層・Application層に偏っていて、ドメイン層の単体テストが手薄だと、DDDとして設計した価値が半減する。

ドメインルール（Aggregateの整合性制約、Value Objectのバリデーション等）が単体テストで検証され、テスト名がユビキタス言語を使いビジネス仕様のドキュメントとして読め、Value Objectのファクトリメソッドに正常系・異常系のテストがあるかを確認する。

---

## 5. 指摘の重大度分類

レビュアーは全ての指摘に重大度ラベルを必ず付ける。ラベルの定義とレビューイーの対応は [ソフトウェアレビュー標準 §1.7](../ソフトウェアレビュー標準.md) に従う。本工程での目安は次のとおり。

| ラベル | 本工程での目安 |
|---|---|
| **[必須]** | テストの信頼性を損なう（テスト間依存、非決定的テスト）、重大なケースが未テスト |
| **[推奨]** | テストの可読性・網羅性・保守性の向上に有効 |
| **[提案]** | よりよいテストの可能性がある任意の改善案 |

**記載例**
```
[必須] Date.now()が固定されていないため、実行日時によってテスト結果が変わります。
[推奨] バリデーションエラーのケース（空文字・null）のテストが不足しています。
[提案] テスト名をユビキタス言語に合わせると仕様書としての価値が上がります。
```

完了基準（[必須]全解消／[推奨]修正または見送り理由あり／承認記録）と [推奨] の見送り理由の書き方（境界値テストの見送り例を含む）は、[ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) と [レビュー記録テンプレート §6](../共通/レビュー記録テンプレート.md) に従う。

## 7. レビュー実施方式

チェック項目を軸0の4層に分けて実施する。標準フロー（提出前ゲート①〜③→④チームレビュー→重大度ラベル→完了）は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) と [生成AIレビュー活用方針 §3](../共通/生成AIレビュー活用方針.md) を参照。**提出前チェック（①〜③）が済んでいないテストコードはPRとして提出しない。**

本工程の特徴: テストの **形式・構造の確認は②生成AIに寄せ**、人間は **「何をテストすべきか」の網羅性判断（④）に集中する**（④比率が3工程で最も高い）。提出前(②③)・チーム(④)の確認項目は §7-1／§7-2 を使用する（各項目の軸0分類と理由は [テストレビュー観点仕分け方針 §3](テストレビュー観点仕分け方針.md)）。

---

## 7-1. 提出前チェックリスト（②生成AIレビュー → ③本人確認）

PR提出前に確認する項目。多くは **②生成AIレビュー（提出前の自動一次レビュー）** に渡し、AIの指摘を解消したうえでレビューイー本人が最終確認する（③）。**全項目が ✅ になってからPRを提出する。**

### A. テストケースの網羅性

- [ ] 正常系（ハッピーパス）のテストが存在する

### B. テストの独立性・再現性

- [ ] `beforeEach` / `afterEach` でテストデータが毎回初期化されている
- [ ] `Date.now()` / `new Date()` / `Math.random()` 等がモックまたは固定値で制御されている
- [ ] テスト間でグローバル状態を共有していない

### C. テストの可読性

- [ ] テスト名が「何をテストするか（振る舞い・仕様）」を表している
- [ ] Arrange / Act / Assert の3ブロックが分かれている
- [ ] 1テストメソッドで検証する振る舞いが1つのみ

### D. テストの粒度・モック戦略

- [ ] domain層のテストがモックなしで動いている（外部依存なし）
- [ ] Repositoryのテストがインメモリ実装またはテスト用DBを使っている（DBをモックしていない）

### E. DDDとの整合性

- [ ] Value Objectのファクトリメソッドに対して正常系・異常系のテストがある
- [ ] ドメインルール違反ケースのテストが1件以上存在する

---

## 7-2. チームレビューチェックリスト（チームで議論して確認）

④人間・チームレビューの項目。判断が分かれる可能性があるため、**チームで合意を形成する**。生成AIは観点の抜けやテストケース候補を提示する一次レビューとして併用できるが、**最終判断は人間が行う**。各項目を④に分類した理由は [テストレビュー観点仕分け方針 §3](テストレビュー観点仕分け方針.md) を参照。

### A. テストケースの網羅性

- [ ] 異常系・境界値ケースが十分か（何が「十分」かはビジネス文脈の判断）
- [ ] ドメインルール違反ケースがビジネス的に十分なケース数か

### D. テストの粒度・モック戦略

- [ ] モック戦略が適切か（モックしすぎて実際の動作を検証できていないケースがないか）
- [ ] テストの粒度（単体/統合/E2Eの使い分け）が適切か
- [ ] テスト自体の複雑さが適切か（テストの保守コストが見合っているか）

### E. DDDとの整合性

- [ ] テスト全体として、ドメイン層のテストが十分に厚いか

---

## 付録: DDDのテスト戦略クイックリファレンス

| 層 | テスト種別 | モック方針 | 重点確認事項 |
|---|---|---|---|
| **Domain層** | 単体テスト | モックなし | ビジネスルール・バリデーション・状態遷移 |
| **Repository** | 単体 or 統合 | インメモリ実装 | CRUD操作・検索条件の正確性 |
| **Application Service** | 単体テスト | Repo=インメモリ、外部API=モック | ユースケースの正常系・異常系 |
| **Presentation** | 統合 or E2E | なし | リクエスト/レスポンス形式・認証 |
