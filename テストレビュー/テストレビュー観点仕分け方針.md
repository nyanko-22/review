# テストレビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体（事前自己チェック / チームレビュー）」を、テストレビューの各チェック項目へ適用した仕分け結果である。仕分け対象の観点A〜Eの詳細は [テストレビューガイドライン](テストレビューガイドライン.md) を参照。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

テストレビューでは特に「テストの構造・形式に関する指摘」（AAAパターン未使用、テスト名が不明瞭等）が多くなりがちで、これらはチームレビューで議論する必要がなく、提出前に本人が自己確認することで排除できる。

一方で、「テストケースの網羅性」や「モック戦略の妥当性」はビジネス文脈の理解と設計判断が必要で、チームで議論する価値が高い。

**目標状態**:  
チームレビューは「どのケースをテストすべきか（ビジネス的判断）」と「モック戦略・テスト粒度の設計判断」にのみ使われ、テストの形式・構造の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を以下2つの問いで判定する（詳細設計レビュー・コードレビューと同じ基準）。

### 問い①: 答えが一意に決まるか？

> 「このチェックをした人が10人いたとき、全員が同じ判断をするか？」

- **Yes（答えが一意）** → 事前自己チェック候補
- **No（人によって判断が変わる）** → チームレビュー候補

### 問い②: 提出者本人が単独で確認できるか？

> 「テストコードとDDDルールだけを見れば、他者の意見なしに確認できるか？」

- **Yes（単独確認可能）** → 事前自己チェック
- **No（他者視点や議論が必要）** → チームレビュー

### まとめ

| 分類 | 性質 | 例 |
|---|---|---|
| **事前自己チェック** | ルールベース・客観的・答えが一意 | AAAパターン、テスト間のグローバル状態共有、domain層テストのモックなし |
| **チームレビュー** | 設計判断・ビジネス判断・トレードオフあり | 異常系の網羅性、モック戦略の妥当性、単体/統合/E2Eの粒度 |

## 3. 仕分け結果サマリー

### 観点A: テストケースの網羅性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 正常系（ハッピーパス）のテストが存在する | 事前自己チェック | テストの存在自体は客観的に確認できる |
| 異常系・境界値ケースが十分か | チームレビュー | 何が「十分な網羅」かはビジネス文脈の判断が必要。リスクとコストのトレードオフがある |
| ドメインルール違反ケースが十分か | チームレビュー | どのドメインルール違反をテストすべきかはビジネスの理解が必要 |

### 観点B: テストの独立性・再現性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| beforeEach/afterEachでデータが初期化されている | 事前自己チェック | セットアップコードの有無はコードから客観的に確認できる |
| Date.now()/Math.random()等が制御されている | 事前自己チェック | モック・固定値の有無はコードから客観的に確認できる |
| テスト間でグローバル状態を共有していない | 事前自己チェック | グローバル変数のアクセスはコードから確認できる |
| テストが実行順序に依存していないか | 事前自己チェック | 各テストが単独で動くかはコードから確認できる（依存関係の明示的な記述） |

※ 観点Bは全項目がルールベースで客観的に確認できるため、全て事前自己チェック。

### 観点C: テストの可読性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| テスト名が振る舞い・仕様を表している | 事前自己チェック | 「should〜」「〜の場合、〜」等のパターンを使っているかは確認できる |
| Arrange/Act/Assertが分かれている | 事前自己チェック | 3ブロックの分離はコードの構造から確認できる |
| 1テストで1つの振る舞いのみを検証 | 事前自己チェック | assertの数・対象の数は客観的に確認できる |
| テストデータが最小限か | チームレビュー | 「最小限か」の判断は主観が入る。過度に詳細なテストデータへの指摘は議論が必要 |

### 観点D: テストの粒度・モック戦略

| チェック項目 | 分類 | 理由 |
|---|---|---|
| domain層のテストがモックなしで動いている | 事前自己チェック | モックの有無はコードから客観的に確認できる。DDDの明確なルール |
| RepositoryがインメモリまたはDB使用（DBモックなし） | 事前自己チェック | モック実装の種類はコードから確認できる。DDDの明確なルール |
| モック戦略全体の妥当性 | チームレビュー | 何をモックするかはトレードオフ。過剰モックも問題になる。チームの合意が必要 |
| テストの粒度（単体/統合/E2E）が適切か | チームレビュー | どの層をどの粒度でテストするかは設計判断 |
| テスト自体の複雑さが適切か | チームレビュー | テストの保守コストの判断はチームの価値観が反映される |

### 観点E: DDDとの整合性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| Value Objectのファクトリメソッドに正常/異常系テストがある | 事前自己チェック | テストの存在は客観的に確認できる |
| ドメインルール違反ケースのテストが1件以上存在する | 事前自己チェック | テストの存在は客観的に確認できる |
| ドメイン層のテストが全体として十分か | チームレビュー | 「十分か」の判断はビジネスロジックの複雑さへの理解が必要 |
| テスト全体がビジネス仕様のドキュメントとして読めるか | チームレビュー | ドキュメントとしての価値の評価は主観が入る |

## 4. 3工程の仕分け比較

詳細設計・コード・テストの3工程で共通する仕分け基準を使いながら、工程ごとに仕分けの比重が異なる。

| 工程 | 事前自己チェックに多い観点 | チームレビューに多い観点 |
|---|---|---|
| 詳細設計 | DDDルールの定義（ID有無・不変性等） | Aggregate粒度・命名・ドメインイベントの要否 |
| コードレビュー | 依存方向・import・マジックナンバー | 命名・ビジネスロジック配置・エラー伝播設計 |
| テストレビュー | AAAパターン・モックなし原則・独立性 | **網羅性（最もチームレビュー比率が高い）**・モック戦略・粒度 |

> テストレビューは「何をテストすべきか」というビジネス判断の比重が最も高く、チームレビューの重要性が特に高い工程。
