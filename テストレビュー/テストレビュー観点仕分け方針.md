# テストレビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体・確認手段（4層: ①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）」を、テストレビューの各チェック項目へ適用した仕分け結果である。4層の判断基準は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)、生成AIの活用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。仕分け対象の観点A〜Eの詳細は [テストレビューガイドライン](テストレビューガイドライン.md) を参照。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

テストレビューでは特に「テストの構造・形式に関する指摘」（AAAパターン未使用、テスト名が不明瞭等）が多くなりがちで、これらはチームレビューで議論する必要がなく、提出前に本人が自己確認することで排除できる。

一方で、「テストケースの網羅性」や「モック戦略の妥当性」はビジネス文脈の理解と設計判断が必要で、チームで議論する価値が高い。

**目標状態**:  
チームレビューは「どのケースをテストすべきか（ビジネス的判断）」と「モック戦略・テスト粒度の設計判断」にのみ使われ、テストの形式・構造の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を、軸0の4層（[レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)）へ、上の層から順に当てはめて仕分ける。判断の出発点は「問い①: 答えが一意に決まるか」「問い②: 本人が単独で確認できるか」。

| 層 | 判定 | この工程での主な該当 |
|---|---|---|
| **①ツール自動化** | ツールで決定論的に検出できる | テスト実行の成否、カバレッジ閾値（一部はlintルール化も可） |
| **②生成AIレビュー** | ①不可だが、答えが概ね一意で、テストコードだけで判定できる | テストの形式・構造（AAA・命名・独立性）、テストの存在チェック、domain層モックなし原則 |
| **③人間・事前自己チェック** | AIに任せきれない／本人の最終確認が要る | AIの誤検知の取捨、テスト意図の最終確認 |
| **④人間・チームレビュー** | 答えが一意でない・トレードオフ・合意が要る | 異常系の網羅性、モック戦略の妥当性、単体/統合/E2Eの粒度 |

テストレビューは成果物がテストコードのため、テストの形式・構造に関するルールベース項目は **②生成AIレビュー**（提出前のAI一次レビュー）が主に担う。一方、「何をテストすべきか」というビジネス判断は **④人間・チームレビュー** に残り、その比率が3工程で最も高い。生成AIは誤検知・幻覚を起こすため、最終責任は人間にある（[生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md)）。

## 3. 仕分け結果サマリー

分類欄は軸0の4層（①ツール自動化／②生成AIレビュー／③人間・自己チェック／④人間・チームレビュー）を表す。

### 観点A: テストケースの網羅性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 正常系（ハッピーパス）のテストが存在する | ②生成AI | テストの存在自体をAIが確認できる |
| 異常系・境界値ケースが十分か | ④チーム | 何が「十分な網羅」かはビジネス文脈の判断が必要。リスクとコストのトレードオフがある（AIは観点の抜けを補助提示） |
| ドメインルール違反ケースが十分か | ④チーム | どのドメインルール違反をテストすべきかはビジネスの理解が必要 |

### 観点B: テストの独立性・再現性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| beforeEach/afterEachでデータが初期化されている | ②生成AI | セットアップコードの有無をAIが確認できる |
| Date.now()/Math.random()等が制御されている | ②生成AI | モック・固定値の有無をAIが確認できる（一部はlintルール化も可） |
| テスト間でグローバル状態を共有していない | ②生成AI | グローバル変数のアクセスをAIが確認できる |
| テストが実行順序に依存していないか | ②生成AI | 各テストが単独で動くかをAIが確認できる |

※ 観点Bは全項目がルールベースで客観的に確認できるため、提出前の②生成AIレビューで検出する。

### 観点C: テストの可読性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| テスト名が振る舞い・仕様を表している | ②生成AI | 「should〜」「〜の場合、〜」等のパターン充足をAIが確認できる |
| Arrange/Act/Assertが分かれている | ②生成AI | 3ブロックの分離構造をAIが確認できる |
| 1テストで1つの振る舞いのみを検証 | ②生成AI | assertの対象数をAIが確認できる |
| テストデータが最小限か | ④チーム | 「最小限か」の判断は主観が入る。過度に詳細なテストデータへの指摘は議論が必要 |

### 観点D: テストの粒度・モック戦略

| チェック項目 | 分類 | 理由 |
|---|---|---|
| domain層のテストがモックなしで動いている | ②生成AI | モックの有無をAIが確認できる。DDDの明確なルール |
| RepositoryがインメモリまたはDB使用（DBモックなし） | ②生成AI | モック実装の種類をAIが確認できる。DDDの明確なルール |
| モック戦略全体の妥当性 | ④チーム | 何をモックするかはトレードオフ。過剰モックも問題になる。チームの合意が必要 |
| テストの粒度（単体/統合/E2E）が適切か | ④チーム | どの層をどの粒度でテストするかは設計判断 |
| テスト自体の複雑さが適切か | ④チーム | テストの保守コストの判断はチームの価値観が反映される |

### 観点E: DDDとの整合性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| Value Objectのファクトリメソッドに正常/異常系テストがある | ②生成AI | テストの存在をAIが確認できる |
| ドメインルール違反ケースのテストが1件以上存在する | ②生成AI | テストの存在をAIが確認できる |
| ドメイン層のテストが全体として十分か | ④チーム | 「十分か」の判断はビジネスロジックの複雑さへの理解が必要 |
| テスト全体がビジネス仕様のドキュメントとして読めるか | ④チーム | ドキュメントとしての価値の評価は主観が入る |

## 4. 3工程の仕分け比較

詳細設計・コード・テストの3工程で共通する4層の仕分け基準を使いながら、工程ごとに比重が異なる。特に「①ツールで自動化できる範囲」と「②生成AIに寄せる範囲」が工程によって変わる。

| 工程 | ①ツール自動化が効く範囲 | ②生成AIレビューに寄せる観点 | ④人間・チームレビューに残す観点 |
|---|---|---|---|
| 詳細設計 | 限定的（成果物が設計書） | DDDルールの定義（ID有無・不変性・依存方向・型定義） | Aggregate粒度・命名の最終合意・ドメインイベントの要否 |
| コードレビュー | 広い（Lint・型・import境界・空catch） | VO実装・DTO変換・機密ログ等の意味的確認 | 命名・ビジネスロジック配置・エラー伝播設計 |
| テストレビュー | テスト実行・カバレッジ閾値 | AAAパターン・モックなし原則・独立性・テストの存在チェック | **網羅性（最も④比率が高い）**・モック戦略・粒度 |

> テストレビューは「何をテストすべきか」というビジネス判断の比重が最も高く、④人間・チームレビューの重要性が特に高い工程。形式・構造の確認は②生成AIに寄せ、人間は網羅性の判断に集中する。
