# 設計別観点：HLF（Hyperledger Fabric）

本モジュールは、成果物に **Hyperledger Fabric（HLF）のチェーンコード／ネットワーク設計が含まれる場合のみ**適用する設計種別固有の観点である。HLFを含む成果物では、各工程の共通観点（[詳細設計](../共通観点/詳細設計.md)／[コード](../共通観点/コード.md)）に本モジュールを重ねて参照する。

> ⚠️ **本モジュールは一般的なHyperledger Fabricベストプラクティスに基づく「たたき台」である。** プロジェクトのネットワーク構成・組織構成・業務要件に合わせて、チームが項目を追加・削除・具体化すること。軸0分類（②③④）は暫定であり、[詳細設計レビュー観点仕分け方針 §3](../詳細設計レビュー/詳細設計レビュー観点仕分け方針.md) 側の確定を待って整合させる。

各工程のチェックリストは §A（提出前＝②③）／§B（チーム＝④）の2区分。

---

## 1. 詳細設計

### 観点（なぜ見るか）

- **チェーンコードの責務境界** — 1つのチェーンコード（スマートコントラクト）に責務を詰め込みすぎると、アップグレード影響とエンドースメント調整が重くなる。コントラクト分割の粒度を見る。
- **チャネル設計** — チャネルはデータ隔離の境界。どの組織がどのチャネルに参加し、何が見えるかが機密性要件と一致しているかを見る。
- **組織・MSP・ACL設計** — どのidentity（MSP・ロール・属性）が何を実行できるかが最小権限で定義されているか。
- **エンドースメントポリシー** — トランザクションの信頼性は「どの組織の承認を必須とするか」で決まる。業務の信頼要件（AND/OR、キーレベルエンドースメント）と一致しているか。
- **プライベートデータコレクション** — 組織間で共有すべきでないデータをチャネル全体に書いていないか。コレクションの対象データ・共有組織・`blockToLive`（purge方針）が定義されているか。
- **台帳データモデル（world state）** — キー設計（composite key 含む）が必要なクエリを表現でき、CouchDB リッチクエリの要否（性能・整合性トレードオフ）が判断されているか。

### A. 提出前チェックリスト（②→③）
- [ ] チェーンコード／コントラクトの責務範囲が明確で、1コントラクトの責務が過大でない。
- [ ] チャネル構成と各組織の参加範囲・可視範囲が定義されている。
- [ ] world state のキー設計（composite key の構成・区切り含む）が定義され、必要なクエリを表現できる。
- [ ] プライベートデータコレクションの定義（対象データ・共有組織・`blockToLive`）が明確である。

### B. チームレビューチェックリスト（④）
- [ ] エンドースメントポリシーが業務の信頼要件に合っているか（AND/OR・キーレベルエンドースメントの要否）。
- [ ] チャネル／プライベートデータによるデータ隔離が、機密性・組織間可視性の要件どおりか。
- [ ] MSP／ACL による権限設計が最小権限になっているか。
- [ ] CouchDB リッチクエリが本当に必要か（LevelDBで足りないか。性能・整合性のトレードオフ）。

## 2. コード（チェーンコード実装）

### 観点（なぜ見るか）

- **決定論性（determinism）** — チェーンコードは全エンドースピアで同一結果になる必要がある。非決定的処理（乱数・システム時刻・map反復順序への依存・外部API呼び出し・グローバル可変状態）はエンドースメント不一致を招く。時刻は `GetTxTimestamp()` を使う。
- **Read-Your-Writes の制約** — 同一トランザクション内で `PutState` した値は、同じトランザクション内の `GetState` では読めない（コミット前のため）。この前提でロジックが破綻していないか。
- **リッチクエリの整合性** — `GetQueryResult`（CouchDBリッチクエリ）の結果は phantom read 検証の対象外。クエリ結果を書き込み判断に使うと二重処理・不整合を招く。
- **キー設計とMVCC競合** — composite key の区切り文字衝突、同一キーへの高頻度更新による `MVCC_READ_CONFLICT`。
- **入力バリデーション／エラー処理** — 引数を検証し、想定外入力で不正な台帳状態を作らない。エラーを握り潰さず適切に返し、panicで異常終了しない。
- **機密情報の扱い** — world state・イベント・ログに機微データを平文で残さない。
- **アクセス制御・イベント** — client identity（cid）で MSPID／属性を検証。イベント発行（`SetEvent`）は1トランザクション1回で、利用側との契約が明確か。

### A. 提出前チェックリスト（②→③）
- [ ] 非決定的処理を使っていない（乱数・システム時刻・map反復依存・外部API呼び出し・グローバル可変状態）。時刻は `GetTxTimestamp()` を使用している。
- [ ] 同一トランザクション内で `PutState` した値を `GetState` で読み直す前提のロジックになっていない。
- [ ] 引数のバリデーションを行い、想定外入力で不正な台帳状態を作らない。
- [ ] エラーを握り潰さず適切に返し、panic で異常終了しない。
- [ ] 機密情報を world state・イベント・ログに平文で残していない。
- [ ] composite key がキー設計どおりに実装され、区切り文字の衝突がない。

### B. チームレビューチェックリスト（④）
- [ ] リッチクエリ（`GetQueryResult`）の結果を書き込み判断に使う場合、phantom read／整合性リスクが妥当に扱われているか。
- [ ] 同一キーへの高頻度更新による MVCC 競合（`MVCC_READ_CONFLICT`）への設計対処（キー分割・集約等）が適切か。
- [ ] client identity（cid）によるアクセス制御が業務ルール上十分か。
- [ ] イベント発行（`SetEvent`）の粒度と利用側との契約が適切か。
