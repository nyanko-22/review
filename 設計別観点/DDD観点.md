# 設計別観点：DDD（ドメイン駆動設計）

本モジュールは、成果物が **ドメイン駆動設計（DDD）を採用している場合のみ**適用する設計種別固有の観点である。非DDDの処理では [共通観点](../共通観点/) だけで完結する。DDDを含む成果物では、各工程の共通観点（[詳細設計](../共通観点/詳細設計.md)／[コード](../共通観点/コード.md)／[テスト](../共通観点/テスト.md)）に本モジュールを重ねて参照する。

- 参考アーキテクチャ: [DDD Hands-On（Zenn）](https://zenn.dev/yamachan0625/books/ddd-hands-on)
- 各チェック項目の軸0分類と理由: 各工程の観点仕分け方針 §3（[詳細設計](../詳細設計レビュー/詳細設計レビュー観点仕分け方針.md)／[コード](../コードレビュー/コードレビュー観点仕分け方針.md)／[テスト](../テストレビュー/テストレビュー観点仕分け方針.md)）
- **①ツール自動化で強制する項目**（import依存方向違反 等）はチェックリストに載せない。一覧は [コードレビュー観点仕分け方針 §4](../コードレビュー/コードレビュー観点仕分け方針.md)。

DDDパターン（Value Object / Entity / Aggregate / Domain Service / Repository / Application Service）が原則どおりに設計・実装・テストされているかを見る。各工程のチェックリストは §A（提出前＝②③）／§B（チーム＝④）の2区分。

---

## 1. 詳細設計

### 観点（なぜ見るか）

**B. ドメインモデルの正確性** — VO/Entity/Aggregate の誤った使い分けは後のリファクタリングコストが極めて高い。特にAggregate境界の設計ミスは整合性バグの温床。
- **Value Object**: 属性の同一性で等価判定し、不変（Immutable。値を変えるなら新インスタンス）として設計され、自身のバリデーションを持つか。プリミティブ型のままでVO化すべき概念が残っていないか。
- **Entity**: 一意識別子（ID）を持ち、ライフサイクル（生成・更新・削除）が定義され、状態変化がビジネスルールとして表現されているか。
- **Aggregate**: Aggregate Root が定義され、外部からはRoot経由でのみアクセスし、1トランザクションで更新する範囲として粒度が適切か、他AggregateはIDのみで参照しているか。
- **Domain Service**: Entity/VOに自然に配置できないロジックのみを置き、Infrastructure層に依存せず、ステートレスか。

**C. レイヤ構造・依存方向** — 依存方向が逆転するとドメイン層が特定のFW/DBに縛られ、テストが困難になり技術的負債化する。

```
[依存の方向（正しい方向）]
Presentation → Application → Domain ← Infrastructure
                                ↑ Infrastructure は Domain に依存（依存性逆転の原則）
```
ドメイン層が上位層に依存していないか、Repositoryインターフェースがドメイン層に定義されているか（実装はInfrastructure層）、Application Serviceがユースケースの調整のみか、PresentationがDTOを介しているかを確認する。

**D. ドメイン表現（ユビキタス言語・ドメインイベント）** — メソッド名がユビキタス言語で命名され、必要なドメインイベント（例: `OrderPlaced`, `UserRegistered`）が定義されているか。

**E. ドメイン層のテスタビリティ** — ドメイン層がフレームワークなしで単体テストできる構造か。

### A. 提出前チェックリスト（②→③）

**Value Object**
- [ ] 属性の同一性で等価判定している（IDでなく値で比較）。
- [ ] 不変（Immutable）として設計されている。
- [ ] 自身のバリデーションロジックを持っている。

**Entity**
- [ ] 一意識別子（ID）を持っている。

**Aggregate**
- [ ] Aggregate Rootが定義されている。
- [ ] 外部からはRootを通じてのみ内部オブジェクトにアクセスする。
- [ ] 他AggregateへはIDのみで参照している（オブジェクト参照ではなく）。

**Domain Service**
- [ ] ステートレスである（状態を持たない）。
- [ ] Infrastructure層に依存していない。

**レイヤ構造・依存方向**
- [ ] ドメイン層がApplication・Infrastructure層に依存していない。
- [ ] Repositoryインターフェースはドメイン層に定義されている。
- [ ] PresentationはDTOを介し、ドメインオブジェクトを直接公開していない。

### B. チームレビューチェックリスト（④）

**Value Object**
- [ ] プリミティブ型のまま使っている箇所で、VO化すべきドメイン概念が残っていないか。

**Entity**
- [ ] ライフサイクル（生成・更新・削除の条件と状態遷移）が過不足なく定義されているか。

**Aggregate**
- [ ] 1トランザクションの更新範囲として粒度が適切か（大きすぎ・小さすぎないか）。

**Domain Service**
- [ ] Entity・Value Objectに自然に配置できないロジックのみが置かれているか。

**レイヤ構造・依存方向**
- [ ] Application Serviceがビジネスロジックを含んでいないか（ユースケースの調整のみか）。

**ドメイン表現**
- [ ] メソッド名がユビキタス言語として適切か（ドメインの概念を正しく表現しているか）。
- [ ] ドメインイベントを定義すべき箇所が設計に含まれているか。

## 2. コード

### 観点（なぜ見るか）

**A. DDDパターンの実装正確性** — 正しく定義されたVO/Entity/Aggregate等も実装で誤るとDDDの価値が消える。設計意図がコードに忠実に反映されているかがこのレビューの核心。
- **Value Object**: `readonly`/`Object.freeze()` で不変、`equals()` が属性比較、ファクトリメソッドでバリデーションして不正値のインスタンスを作らせないか。
- **Entity**: IDがプリミティブでなくVOで実装され、状態変更メソッドがビジネスルールを表現し、同一性比較をIDで行うか。
- **Aggregate**: Root以外の内部オブジェクトが外部から直接 `public` アクセスされず、変更はRoot経由のメソッドのみか。
- **Domain Service**: privateフィールドを持たず（ステートレス）、Application/Infrastructure層に依存しないか。
- **Repository**: ドメイン層のインターフェースを実装し、Aggregate単位で `save()`/`findById()` を持ち、ORMモデルでなくドメインオブジェクトに変換して返すか。
- **Application Service**: ドメインオブジェクトを調整するのみ（ビジネスロジックを含まない）で、トランザクション管理を担うか。

**B. レイヤ構造・依存方向** — 実装の `import` 一つで依存方向が逆転する。domain層にORM/Webフレームワークのimportがないか、application層がinfrastructureの具体実装を直接importしていないか（interface経由か）、presentation層がドメインオブジェクトをそのままレスポンスに使っていないか（DTO変換）を確認する。※import依存方向違反は **①ツール自動化**（eslint-plugin-boundaries 等）で強制する。

**C. 命名（ユビキタス言語）** — 命名がドメインのユビキタス言語と一致し、ドメイン知識がコードから読み取れるか。

### A. 提出前チェックリスト（②→③）

**Value Object**
- [ ] `readonly` または `Object.freeze()` 等で不変性が保証されている。
- [ ] コンストラクタ/ファクトリメソッドでバリデーションを行っている。
- [ ] `equals()` が属性比較で実装されている。

**Entity**
- [ ] IDがValue Objectとして実装されている（プリミティブ型ではない）。

**Aggregate**
- [ ] Root以外の内部オブジェクトが外部から直接 `public` アクセスされていない。

**Domain Service**
- [ ] privateフィールドがない（ステートレス）。

**Repository**
- [ ] ドメイン層のインターフェースを実装している。
- [ ] 返却値がドメインオブジェクト（ORMモデルのまま返していない）。

**レイヤ構造・依存方向**
- [ ] presentation層がドメインオブジェクトをそのままレスポンスに使っていない（DTOに変換）。

### B. チームレビューチェックリスト（④）
- [ ] Aggregate境界の実装が設計意図と合っているか（RootがAggregateの整合性を保証しているか）。
- [ ] Application ServiceとDomain層のロジック配置が適切か。
- [ ] 命名がユビキタス言語として適切か（ドメインの概念を正しく表現しているか）。

## 3. テスト

### 観点（なぜ見るか）

**D. テストの粒度・モック戦略（DDD層別）** — DDDでは層ごとに適切なテスト戦略が異なる。誤ったモック戦略は「テストが通るが本番で動かない」を生む。特に「Repositoryをモックでテストする」のはアンチパターン。ドメイン層はモックなし、Repositoryはインメモリ実装（またはテスト用DB）、Application Serviceは外部依存のみモック、E2Eは正常系主要フローに限定されているかを確認する。

**E. DDDとの整合性** — DDDで最も重要なのはドメインロジックの正確性。テストがインフラ/Application層に偏りドメイン層の単体テストが手薄だと、DDDの価値が半減する。ドメインルール（Aggregate整合性制約、VOバリデーション等）が単体テストで検証され、テスト名がユビキタス言語でビジネス仕様のドキュメントとして読めるかを確認する。

### A. 提出前チェックリスト（②→③）
- [ ] domain層のテストがモックなしで動いている（外部依存なし）。
- [ ] Repositoryのテストがインメモリ実装またはテスト用DBを使っている（DBをモックしていない）。
- [ ] Value Objectのファクトリメソッドに対して正常系・異常系のテストがある。
- [ ] ドメインルール違反ケースのテストが1件以上存在する。

### B. チームレビューチェックリスト（④）
- [ ] モック戦略が適切か（モックしすぎて実際の動作を検証できていないケースがないか）。
- [ ] テストの粒度（単体/統合/E2Eの使い分け）が適切か。
- [ ] テスト全体として、ドメイン層のテストが十分に厚いか。

## 付録: DDDクイックリファレンス

### 概念（設計時の判断）
| 概念 | 定義 | 判断基準 |
|---|---|---|
| **Value Object** | 値で識別される不変オブジェクト | 「金額が同じ2つの金額オブジェクトは同じか？」→ Yes なら VO |
| **Entity** | IDで識別されるオブジェクト | 「同じ名前の2人のユーザーは同じか？」→ No なら Entity |
| **Aggregate** | 整合性境界を持つオブジェクト群 | 「一緒に変更され一緒に保存される」単位 |
| **Domain Service** | 複数のEntityにまたがる操作 | Entityに置くと不自然なビジネスロジック |
| **Repository** | Aggregateの永続化インターフェース | ドメイン層に定義し、Infrastructure層で実装 |
| **Application Service** | ユースケースの調整役 | ビジネスロジックを持たず、ドメインオブジェクトを呼び出す |

### コードレベルの確認ポイント
| DDDパターン | 確認ポイント |
|---|---|
| **Value Object** | `readonly`/`freeze` で不変。`equals()` が属性比較。ファクトリメソッドでバリデーション |
| **Entity** | IDがVO型。状態変更メソッドがビジネスルールを含む |
| **Aggregate** | Root以外の内部オブジェクトはprivateまたはprotected |
| **Domain Service** | privateフィールドなし。domain層のimportのみ |
| **Repository** | domain層のinterfaceを実装。Aggregate単位でsave/find |
| **Application Service** | ドメインオブジェクトを呼び出すのみ。ビジネスロジックなし |

### テスト戦略
| 層 | テスト種別 | モック方針 | 重点確認事項 |
|---|---|---|---|
| **Domain層** | 単体テスト | モックなし | ビジネスルール・バリデーション・状態遷移 |
| **Repository** | 単体 or 統合 | インメモリ実装 | CRUD操作・検索条件の正確性 |
| **Application Service** | 単体テスト | Repo=インメモリ、外部API=モック | ユースケースの正常系・異常系 |
| **Presentation** | 統合 or E2E | なし | リクエスト/レスポンス形式・認証 |
