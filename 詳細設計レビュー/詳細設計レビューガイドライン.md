# 詳細設計レビューガイドライン

## 1. このガイドラインについて

### 目的
詳細設計レビューは、**コーディング開始前に設計の誤りを検出する**ことを目的とする。  
コーディング後に発覚した設計ミスの修正コストは、設計段階の数倍以上になる。このガイドラインは、レビュアー・レビューイー双方が同じ観点でレビューに臨むことで、品質のばらつきとレビュー時間の無駄を解消するために策定した。

### 対象工程
- **対象**: 詳細設計工程（コーディング開始前）
- **対象読者**: レビュアー（設計を評価する側）・レビューイー（設計を提出する側）の両方

### アーキテクチャ前提
本プロジェクトは **ドメイン駆動設計（DDD）** を採用している。  
レビューはDDDの原則（Value Object / Entity / Aggregate / Domain Service / Repository / Application Service）に基づいて行う。

---

## 2. レビュースコープ

「何を見るか」と同時に「何を見ないか」を明確にする。スコープ外の指摘はレビュー時間を無駄に消費する。

| 対象（見る） | 対象外（見ない） |
|---|---|
| ドメインモデルの設計（Entity/ValueObject/Aggregate） | 実装コードの詳細・アルゴリズム |
| レイヤー構造と依存方向 | コーディングスタイル・命名規則の細部 |
| インターフェース（メソッド定義）の設計 | パフォーマンス最適化の具体策 |
| ユビキタス言語の一貫性 | UIデザイン・画面レイアウト |
| 要件との整合性 | デプロイ実行手順・CI/CDパイプライン構築作業の細部 |
| インフラ/IaC設計（AWS CDK 等。IaCを成果物に含む場合・観点F） | 監視アラート閾値の運用チューニング |
| 採用技術の実現可否（新規ライブラリ/FW・観点G） | 既存技術の細かな実装テクニック |

> 観点F・観点Gは、成果物に **IaC（AWS CDK 等）や新規技術の採用** が含まれる場合に適用する。該当しないプロジェクトでは省略してよい。観点F/Gの具体項目は実スタックに合わせて調整する。

---

## 3. 役割と責任

レビュアー・レビューイーの役割は [ソフトウェアレビュー標準 §1.5](../ソフトウェアレビュー標準.md) の定義に従う。本工程（詳細設計レビュー）での要点は次のとおり。

| 役割 | 本工程での要点 |
|---|---|
| **レビューイー** | 設計書を提出し、指摘に対して「修正」または「見送り理由」を回答する |
| **レビュアー** | 設計観点でフィードバックを行い、指摘には必ず重大度ラベル（[必須]/[推奨]/[提案]）を明示する |

> **重要**: レビュアーは重大度ラベルを明示せずに指摘してはいけない。レビューイーは重大度が明示されていない指摘への対応は不要とする（[標準 §1.7](../ソフトウェアレビュー標準.md)）。

## 4. レビュー観点

### 観点A: 要件との整合性

**なぜこれを見るか**  
設計がビジネス要件を正しく反映していなければ、完璧に実装しても誤ったものができあがる。設計段階での発見が最もコストが低い。

**チェック内容**
- 機能要件のユースケースが全て設計に反映されているか
- ドメインの言葉（ユビキタス言語）が設計書・コード・仕様書で一致しているか
- 例外ケース・エラーケースが設計に含まれているか

---

### 観点B: ドメインモデルの正確性

**なぜこれを見るか**  
Value Object / Entity / Aggregate の誤った使い分けは、後のリファクタリングコストが極めて高くなる。特にAggregate境界の設計ミスは、整合性バグの温床になる。

**チェック内容**

#### Value Object
- 値の同一性ではなく**属性の同一性**で等価判定しているか
- **不変（Immutable）** として設計されているか（値を変えるなら新しいインスタンスを生成する）
- ドメイン概念として意味を持ち、自身の検証ロジック（バリデーション）を持つか
- プリミティブ型をそのまま使っている箇所で、Value Objectにできるものが残っていないか

#### Entity
- **一意識別子（ID）** を持っているか
- ライフサイクル（生成・更新・削除）が明確に定義されているか
- 状態変化を表すメソッドがビジネスルールとして表現されているか

#### Aggregate
- **Aggregate Root** が明確に定義されているか
- 外部から内部のオブジェクトに直接アクセスできないように境界が引かれているか（Rootを通じてのみアクセス）
- 1つのトランザクションで更新する範囲として適切な粒度か（大きすぎ・小さすぎないか）
- 異なるAggregateの参照はIDのみで行っているか（オブジェクト参照ではなく）

#### Domain Service
- Entity / Value Object に自然に配置できないロジックのみがDomain Serviceに置かれているか
- Domain ServiceがInfrastructure層に依存していないか
- 状態を持っていないか（ステートレスであるか）

---

### 観点C: レイヤー構造・依存方向

**なぜこれを見るか**  
依存方向が逆転すると、ドメイン層が特定のフレームワークやDBに縛られる。これが起きると、テストが困難になり、技術的負債として長期間チームを苦しめる。

```
[依存の方向（正しい方向）]
Presentation → Application → Domain ← Infrastructure
                                ↑
                         Infrastructure は Domain に
                         依存する（依存性逆転の原則）
```

**チェック内容**
- ドメイン層（Entity / Value Object / Domain Service）がApplication層・Infrastructure層に依存していないか
- Repositoryのインターフェースはドメイン層に定義されているか（実装はInfrastructure層）
- Application Serviceがビジネスロジックを含んでいないか（ユースケースの**調整**のみを行っているか）
- Presentation層がドメイン層のオブジェクトを直接使っていないか（DTOを介しているか）

---

### 観点D: インターフェース設計

**なぜこれを見るか**  
不明確なインターフェース定義は、実装者の誤解を生み、後で「仕様と違う」問題を引き起こす。レビュー段階でのインターフェース合意が実装速度を上げる。

**チェック内容**
- メソッド名がドメインの言葉（ユビキタス言語）で命名されているか
- 引数と戻り値の型が明確に定義されているか
- 副作用（状態変更）を持つメソッドと持たないメソッドが区別されているか
- ドメインイベント（例: `OrderPlaced`, `UserRegistered`）が必要な箇所で設計されているか
- エラー時の型・例外が定義されているか

---

### 観点E: テスタビリティ

**なぜこれを見るか**  
テストしやすい設計は、バグの早期発見とリファクタリング時の安全性を高める。設計段階でテスト困難な構造を作ると、後からテストを追加することが難しい。

**チェック内容**
- 依存するオブジェクトが**注入（Dependency Injection）** できる構造になっているか
- 外部依存（DB・外部API・時刻など）が抽象化（インターフェース化）されているか
- ドメイン層のロジックがフレームワークなしで単体テストできるか

---

### 観点F: インフラ/IaC設計（AWS CDK 等）

> 成果物に IaC（AWS CDK 等）が含まれる場合に適用する。項目は実スタックに合わせて調整する。

**なぜこれを見るか**  
IaCのリソースは**デフォルト値**で生成されることが多く、暗号化・公開範囲・削除保護などが意図せず安全側／危険側に振れる。デフォルトのまま放置すると、セキュリティ事故やデータ消失につながる。設計段階で「デフォルト値を把握し、意図して上書きしたか」を確認するのが最も低コストである。

**チェック内容**
- 各リソースの**デフォルト値**（暗号化・公開設定／パブリックアクセス・削除保護（removalPolicy/RETAIN）・バックアップ保持期間・ログ保持期間など）を把握し、要件に合わせて**意図的に設定**しているか
- IAMが**最小権限**になっているか（ワイルドカード `*` の濫用がないか）
- 環境差分（dev/stg/prod）が**ハードコードされず**にパラメータ／コンテキストで外出しされているか
- ネットワーク（VPC・SG・公開サブネット）の公開範囲が要件どおりか
- 命名規約・タグ付け（コスト配分・棚卸し用）の方針に沿っているか
- 静的解析（cdk-nag 等）で検出できる項目は自動化を前提とし、設計書ではその適用方針を確認する

---

### 観点G: 新規ライブラリ・フレームワークの実現可否検証

> 新規にライブラリ／フレームワーク／外部サービスを採用する場合に適用する。

**なぜこれを見るか**  
未検証の新規技術をそのまま設計に組み込むと、実装フェーズで「想定した機能が実現できない」「ライセンス・性能・互換性の制約に抵触する」ことが判明し、大きな手戻りになる。採用判断の根拠を設計段階で示すことで、実装着手後の破綻を防ぐ。

**チェック内容**
- 採用する新規技術について、**事前検証（PoC・技術検証）を実施**し、要件を満たせること（実現可否）を確認したか
- **ライセンス・保守状況（メンテナンス頻度・コミュニティ）・性能・既存スタックとの互換性**を評価したか
- 採用が難しい場合の**代替案**と、**撤退条件（どうなったら採用を取りやめるか）** を提示しているか
- 学習コスト・運用負荷を含め、チームで採用に合意できているか

## 5. 指摘の重大度分類

レビュアーは全ての指摘に、標準で定義する重大度ラベル（[ソフトウェアレビュー標準 §1.7](../ソフトウェアレビュー標準.md)）を必ず付ける。本工程での具体的な目安は次のとおり。

| ラベル | 英語 | 本工程での目安 | レビューイーの対応 |
|---|---|---|---|
| **[必須]** | Must | DDD原則違反、要件未達、整合性破壊・システム破綻リスクがある | **必ずFix** してから次工程へ進む |
| **[推奨]** | Should | 品質・保守性・テスタビリティの向上に有効。対応しないリスクがある | Fixするか、**見送り理由を明示**して合意を得る |
| **[提案]** | Nits | よりよい設計の可能性がある任意の改善案 | 対応は任意。返答不要 |

**記載例**
```
[必須] OrderAggregateのRootが定義されていないため、整合性が保証できません。
[推奨] PriceはValue Objectとして切り出すと、バリデーションが集約できます。
[提案] この処理はDomain Eventで表現するとより意図が明確になります。
```

## 6. 完了基準（Exit Criteria）

完了の基本的な考え方は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md)（レビューの進め方）に準拠する。本工程では以下の条件を全て満たした時点でレビュー完了とする。

- [ ] **[必須]** の指摘が全て解消されている
- [ ] **[推奨]** の指摘が全て「修正済み」または「見送り理由あり」の状態になっている
- [ ] レビュアーが承認コメントを記録している（例: `LGTM` / `承認`）

> **[推奨] の見送り理由の例**
> - 「今フェーズのスコープ外のため次スプリントのタスクに追加しました（チケット#XXX）」
> - 「コスト対効果を検討した結果、現時点では見送ります。〇〇の理由で今後の拡張時に対応します」

## 7. レビュー実施方式

レビュー指摘数を減らしてレビュー時間を圧縮するため、チェック項目を軸0の4層（[レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)）に分けて実施する。**ねらいは人間（③④）が見る項目を絞り、機械的・定型的な確認をツール（①）と生成AI（②）に寄せること。**

| 層 | 実施者 | タイミング | 対象項目の性質 |
|---|---|---|---|
| **①ツール自動化** | CI／静的解析（cdk-nag 等） | 提出前（CI） | 機械的に検出できる（チェックリストにも載せない） |
| **②生成AIレビュー** | 生成AI（提出前の自動一次レビュー） | 提出前 | 答えが概ね一意で、設計書からAIが判定できる（§7-1の多く） |
| **③人間・事前自己チェック** | レビューイー本人 | 提出前 | AIに任せきれない／本人の最終確認が要る |
| **④人間・チームレビュー** | チーム全員 | レビュー会議・PR上 | 設計判断・トレードオフ・合意が必要（§7-2。AIは論点出しの補助） |

> **運用ルール**: 提出前チェック（①〜③）が全て済んでいない設計書はチームレビューに提出しない。  
> ④チームレビューは「判断が必要な議論」に集中することで、1件あたりの所要時間を削減する。生成AIの運用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。

## 7-1. 提出前チェックリスト（②生成AIレビュー → ③本人確認）

提出前に確認する項目。多くは **②生成AIレビュー（提出前の自動一次レビュー）** に渡し、AIの指摘を解消したうえでレビューイー本人が最終確認する（③）。**全項目が ✅ になってからチームレビューに提出する。**

### A. 要件との整合性

- [ ] 機能要件の全ユースケースが設計に反映されている（仕様書と照合）
- [ ] ユビキタス言語が設計書・仕様書・コードで一致している

### B. ドメインモデルの正確性

**Value Object**
- [ ] 属性の同一性で等価判定している（IDでなく値で比較）
- [ ] 不変（Immutable）として設計されている
- [ ] 自身のバリデーションロジックを持っている

**Entity**
- [ ] 一意識別子（ID）を持っている

**Aggregate**
- [ ] Aggregate Rootが定義されている
- [ ] 外部からはRootを通じてのみ内部オブジェクトにアクセスする
- [ ] 他AggregateへはIDのみで参照している（オブジェクト参照ではなく）

**Domain Service**
- [ ] ステートレスである（状態を持たない）
- [ ] Infrastructure層に依存していない

### C. レイヤー構造・依存方向

- [ ] ドメイン層がApplication・Infrastructure層に依存していない
- [ ] Repositoryインターフェースはドメイン層に定義されている
- [ ] PresentationはDTOを介してドメインオブジェクトを直接公開していない

### D. インターフェース設計

- [ ] 引数・戻り値の型が明確に定義されている
- [ ] 副作用（状態変更）の有無が明確になっている
- [ ] エラー型・例外が定義されている

### E. テスタビリティ

- [ ] 依存オブジェクトがDI（依存性注入）できる構造になっている
- [ ] 外部依存（DB・外部API・時刻）がインターフェースで抽象化されている
- [ ] ドメイン層がフレームワークなしで単体テストできる

### F. インフラ/IaC設計（AWS CDK 等）※IaCを成果物に含む場合

- [ ] リソースのデフォルト値（暗号化・公開設定・削除保護・保持期間）を把握し意図して設定している
- [ ] 環境差分（dev/stg/prod）がハードコードされず外出しされている
- [ ] IAMが最小権限になっている（①cdk-nag 等の自動検出も活用）
- [ ] 命名規約・タグ付けの方針に沿っている

### G. 新規技術の実現可否 ※新規技術を採用する場合

- [ ] ライセンス・保守状況・性能・互換性を整理している（②AIで一次整理）

## 7-2. チームレビューチェックリスト（チームで議論して確認）

④人間・チームレビューの項目。判断が分かれる可能性があるため、**チームで合意を形成する**。生成AIは「設計書との差分」「観点の抜け」を提示する一次レビューとして併用できるが、**最終判断は人間が行う**。

### A. 要件との整合性

- [ ] 例外・エラーケースが設計に含まれており、想定されるケースが網羅されているか  
  *（何が「十分な網羅」かはチームの判断が必要）*

### B. ドメインモデルの正確性

**Value Object**
- [ ] プリミティブ型（文字列・数値など）をそのまま使っている箇所で、VO化すべきドメイン概念が残っていないか  
  *（何をVOにするかは設計判断。過剰なVO化もアンチパターン）*

**Entity**
- [ ] ライフサイクル（生成・更新・削除の条件と状態遷移）が過不足なく定義されているか  
  *（どこまで設計書に明示するかは判断が必要）*

**Aggregate**
- [ ] 1トランザクションの更新範囲として粒度が適切か（大きすぎ・小さすぎないか）  
  *（Aggregate境界の設計はDDDで最も議論になる判断。チームの合意が必須）*

**Domain Service**
- [ ] Entity・Value Objectに自然に配置できないロジックのみがDomain Serviceに置かれているか  
  *（「自然に配置できない」の判断はDDD理解の深さによる差が出やすい）*

### C. レイヤー構造・依存方向

- [ ] Application Serviceがビジネスロジックを含んでいないか（ユースケースの調整のみか）  
  *（何が「ビジネスロジック」かの解釈が人によって変わりやすい）*

### D. インターフェース設計

- [ ] メソッド名がユビキタス言語として適切か（ドメインの概念を正しく表現しているか）  
  *（命名の妥当性は主観が入る。②AIで用語集と一次照合し、チームで最終合意する）*
- [ ] ドメインイベントを定義すべき箇所が設計に含まれているか  
  *（どこでイベントが「必要か」は設計判断。見落としが多い観点）*

### F. インフラ/IaC設計（AWS CDK 等）※IaCを成果物に含む場合

- [ ] ネットワークの公開範囲（VPC・SG・パブリックアクセス）が要件どおりか  
  *（「要件どおり」の妥当性・リスク判断が必要。②AIが公開設定を検出して提示）*

### G. 新規技術の実現可否 ※新規技術を採用する場合

- [ ] 事前検証（PoC・技術検証）を実施し、実現可否を確認したか  
  *（採否はプロジェクトのリスク判断。チームの合意が必要）*
- [ ] 代替案・撤退条件を提示しているか  
  *（トレードオフの判断であり、チームで合意する）*

## 付録: DDD概念クイックリファレンス

| 概念 | 定義 | 判断基準 |
|---|---|---|
| **Value Object** | 値で識別される不変オブジェクト | 「金額が同じ2つの金額オブジェクトは同じか？」→ Yes なら VO |
| **Entity** | IDで識別されるオブジェクト | 「同じ名前の2人のユーザーは同じか？」→ No なら Entity |
| **Aggregate** | 整合性境界を持つオブジェクト群 | 「一緒に変更され一緒に保存される」単位 |
| **Domain Service** | 複数のEntityにまたがる操作 | Entityに置くと不自然なビジネスロジック |
| **Repository** | Aggregateの永続化インターフェース | ドメイン層に定義し、Infrastructure層で実装 |
| **Application Service** | ユースケースの調整役 | ビジネスロジックを持たず、ドメインオブジェクトを呼び出す |
