# 詳細設計レビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体・確認手段（4層: ①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）」を、詳細設計レビューの各チェック項目へ適用した仕分け結果である。4層の判断基準は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)、生成AIの活用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。仕分け対象の観点A〜Gの詳細は [詳細設計レビューガイドライン](詳細設計レビューガイドライン.md) を参照。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

特に「ルールとして明確に決まっているのに守られていない」類の指摘は、チームレビューの場で議論する必要がない。これをチームレビューの前に提出者本人が自己確認することで、チームレビューに入る段階での品質を保証し、チームが本当に議論すべき設計判断に集中できるようにする。

**目標状態**:  
チームレビューは「設計上の判断・トレードオフ・合意形成」にのみ使われ、ルール違反の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を、軸0の4層（[レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)）へ、上の層から順に当てはめて仕分ける。

| 層 | 判定 | この工程での主な該当 |
|---|---|---|
| **①ツール自動化** | ツールで決定論的に検出できる | 設計工程では少ない（実装後にCI/静的解析で担保） |
| **②生成AIレビュー** | ①不可だが、答えが概ね一意で、設計書・仕様書だけで判定できる | DDDルールの充足（ID有無・不変性・依存方向・型定義 等）、仕様書との用語・処理整合 |
| **③人間・事前自己チェック** | AIに任せきれない／本人の最終確認が要る | AIの誤検知の取捨、機微な前提の確認 |
| **④人間・チームレビュー** | 答えが一意でない・トレードオフ・合意が要る | Aggregate粒度、命名の最終合意、ドメインイベント要否、網羅の十分性 |

従来「事前自己チェック」に分類していたルールベースの項目の多くは、**②生成AIレビュー**（提出前の自動一次レビュー）へ移す。これにより人間が見る項目（③＋④）を絞る。ただし生成AIは誤検知・幻覚を起こすため、**最終責任は人間**にある（[生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md)）。

> 判断の出発点は従来どおり「問い①: 答えが一意に決まるか」「問い②: 本人が単独で確認できるか」。一意（≒②候補）のうちツール化できるものを①へ、本人確認が要るものを③へ振り分ける。一意に決まらないものは④。

## 3. 仕分け結果サマリー

分類欄は軸0の4層（①ツール自動化／②生成AIレビュー／③人間・自己チェック／④人間・チームレビュー）を表す。「②→④」は、生成AIで一次評価したうえで人間が最終判断することを示す。

### 観点A: 要件との整合性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 機能要件の全ユースケースが設計に反映されている | ②生成AI | 仕様書とAIが突合できる。最終的な要件解釈は人間が確認 |
| ユビキタス言語が設計書・仕様書で一致している | ②生成AI | 用語の一致はAIが照合できる |
| 例外・エラーケースが設計に含まれており網羅されているか | ④チーム | 何が「十分な網羅」かはチームの判断が必要（AIは観点の抜けを補助提示） |

### 観点B: ドメインモデルの正確性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 属性の同一性で等価判定している | ②生成AI | VOのルールとして明確。設計書からAIが意味的に確認できる |
| 不変（Immutable）として設計されている | ②生成AI | 状態変更の有無を設計書からAIが確認できる |
| 自身のバリデーションロジックを持っている | ②生成AI | 持っているかどうかをAIが確認できる |
| プリミティブ型のVO化判断 | ④チーム | 何をVOにするかは設計判断。過剰なVO化もアンチパターン（AIはVO化候補を提示） |
| 一意識別子（ID）を持っている | ②生成AI | IDの有無をAIが確認できる |
| ライフサイクルの定義の完全性 | ④チーム | 「どこまで定義すれば十分か」は判断が必要 |
| Aggregate Rootが定義されている | ②生成AI | 定義の有無をAIが確認できる |
| Rootを通じてのみ内部オブジェクトにアクセスする | ②生成AI | アクセスパスを設計書からAIが確認できる |
| 1トランザクションの粒度が適切か | ④チーム | DDDで最も議論になる設計判断。正解が一意に決まらない |
| 他AggregateへはIDのみで参照している | ②生成AI | 参照方法を設計書からAIが確認できる |
| EntityやVOに置けないロジックのみがDomain Serviceにあるか | ④チーム | 「自然に配置できない」の判断はDDD理解の深さによって差が出る |
| Domain Serviceがステートレスである | ②生成AI | 状態を持つかどうかをAIが確認できる |
| Domain ServiceがInfrastructure層に依存していない | ②生成AI | 依存関係を設計書からAIが確認できる |

### 観点C: レイヤー構造・依存方向

| チェック項目 | 分類 | 理由 |
|---|---|---|
| ドメイン層がApplication・Infrastructure層に依存していない | ②生成AI | 設計段階はAIが確認。実装後は①ツール（eslint-plugin-boundaries 等）で自動化 |
| Repositoryインターフェースはドメイン層に定義されている | ②生成AI | 定義場所を設計書からAIが確認できる |
| Application Serviceにビジネスロジックが混入していないか | ④チーム | 「ビジネスロジック」の定義が人によって変わりやすく、議論が必要 |
| PresentationはDTOを介してドメインオブジェクトを直接公開していない | ②生成AI | DTOを介しているかをAIが確認できる |

### 観点D: インターフェース設計

| チェック項目 | 分類 | 理由 |
|---|---|---|
| メソッド名がユビキタス言語として適切か | ②→④ | AIが用語集と照合して一次評価 → 命名の妥当性はチームで最終合意 |
| 引数・戻り値の型が明確に定義されている | ②生成AI | 定義の有無をAIが確認できる |
| 副作用の有無が明確になっている | ②生成AI | 明示されているかをAIが確認できる |
| 必要なドメインイベントが定義されているか | ④チーム | どこでイベントが「必要か」は設計判断。見落としが多い観点（AIは候補を提示） |
| エラー型・例外が定義されている | ②生成AI | 定義の有無をAIが確認できる |

### 観点E: テスタビリティ

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 依存オブジェクトがDIできる構造になっている | ②生成AI | 構造の有無を設計書からAIが確認できる |
| 外部依存がインターフェースで抽象化されている | ②生成AI | 抽象化の有無を設計書からAIが確認できる |
| ドメイン層がフレームワークなしで単体テストできる | ②生成AI | フレームワーク依存の有無をAIが確認できる |

> **観点Eは全項目が②生成AIレビュー**: テスタビリティの観点は、いずれも設計書を見れば客観的に確認できるルールベースの項目であり、提出前のAI一次レビューで検出できる。

### 観点F: インフラ/IaC設計（AWS CDK 等）

> IaC（AWS CDK 等）を成果物に含む場合に適用する。

| チェック項目 | 分類 | 理由 |
|---|---|---|
| リソースのデフォルト値（暗号化・公開設定・削除保護・保持期間 等）を把握し意図的に設定している | ②生成AI | 設計書/CDKコードからAIがデフォルトとの差を指摘できる。cdk-nag 等の①ツールも併用 |
| IAMが最小権限になっている（ワイルドカードの濫用がない） | ①ツール＋②生成AI | cdk-nag 等で機械検出。残りはAIが補完 |
| 環境差分（dev/stg/prod）がハードコードされず外出しされている | ②生成AI | ハードコードの有無をAIが検出できる |
| ネットワークの公開範囲が要件どおりか | ④チーム | 「要件どおり」の妥当性・リスク判断が必要（AIは公開設定を検出して提示） |
| 命名規約・タグ付けの方針に沿っている | ①ツール／②生成AI | 規約はルール化でき、ツールまたはAIで照合できる |

### 観点G: 新規ライブラリ・FWの実現可否検証

> 新規技術を採用する場合に適用する。

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 事前検証（PoC・技術検証）を実施し実現可否を確認している | ④チーム | 採否はプロジェクトのリスク判断。チームの合意が必要 |
| ライセンス・保守状況・性能・互換性を評価している | ②→④ | AIが情報を整理・一次評価 → 採否はチームで判断 |
| 代替案・撤退条件を提示している | ④チーム | トレードオフの判断であり、チームで合意する |

## 4. 他工程への適用方針

コードレビュー・テストレビューでも同じ4層の判断基準で仕分けを行う（[コードレビュー観点仕分け方針](../コードレビュー/コードレビュー観点仕分け方針.md) / [テストレビュー観点仕分け方針](../テストレビュー/テストレビュー観点仕分け方針.md)）。工程による違いは「①ツールで自動化できる範囲」に表れる。

- **コードレビュー**: import依存方向・Lint・型などツールで強制できる項目が多く、**①ツール自動化** の比率が高い。
- **詳細設計・テスト**: 成果物が設計書・テストコードで、ツール検出は限定的なため、ルールベース項目は **②生成AIレビュー** が主に担う。

**原則**: ①ツールで自動化できるものは自己チェックにも含めずCI/CDで強制し、②生成AIで一次レビューできるものは提出前に通す。人間（③④）は判断・合意の要る項目に集中する。
