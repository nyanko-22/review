# 詳細設計レビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体・確認手段（4層: ①ツール自動化／②生成AIレビュー／③人間・事前自己チェック／④人間・チームレビュー）」を、詳細設計レビューの各チェック項目へ適用した仕分け結果である。4層の判断基準は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md)、生成AIの活用は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。仕分け対象の観点の詳細は [共通観点／詳細設計](../共通観点/詳細設計.md) と、該当する [設計別観点](../設計別観点/)（DDD／インフラ・IaC／HLF）を参照。本§3は **設計種別（共通／DDD／インフラ・IaC／HLF）ごと** にグルーピングして分類を示す。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー時間が肥大化する課題（「ルールとして明確なのに守られていない」類の指摘がチームレビューに集まること）と、それを提出前の自己確認・ツール・生成AIで潰すねらいは [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) を参照。

**目標状態**:  
チームレビューは「設計上の判断・トレードオフ・合意形成」にのみ使われ、ルール違反の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を軸0の4層へ上の層から順に当てはめて仕分ける。判断フロー（ツールで検出可か→答えが一意か→本人確認が要るか）は [レビュー分類方針総覧 §3](../共通/レビュー分類方針総覧.md) に従う。本工程での各層の主な該当は次のとおり。

| 層 | この工程での主な該当 |
|---|---|
| **①ツール自動化** | 設計工程では少ない（実装後にCI/静的解析で担保） |
| **②生成AIレビュー** | DDDルールの充足（ID有無・不変性・依存方向・型定義 等）、仕様書との用語・処理整合 |
| **③人間・事前自己チェック** | AIの誤検知の取捨、機微な前提の確認 |
| **④人間・チームレビュー** | Aggregate粒度、命名の最終合意、ドメインイベント要否、網羅の十分性 |

設計工程は成果物が設計書のため①ツールが少なく、従来「事前自己チェック」としていたルールベース項目の多くを **②生成AIレビュー**（提出前一次レビュー）へ移す。生成AIの限界と責任分界（最終責任は人間）は [生成AIレビュー活用方針](../共通/生成AIレビュー活用方針.md) を参照。

---

## 3. 仕分け結果サマリー

分類欄は軸0の4層（①ツール自動化／②生成AIレビュー／③人間・自己チェック／④人間・チームレビュー）を表す。「②→④」は、生成AIで一次評価したうえで人間が最終判断することを示す。項目は **設計種別ごと**（共通／DDD／インフラ・IaC／HLF）にグルーピングした。各項目のチェックリストは対応するモジュール（[共通観点／詳細設計](../共通観点/詳細設計.md)・[設計別観点](../設計別観点/)）に収録している。

### 3.1 共通観点（全設計種別で適用）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 機能要件の全ユースケースが設計に反映されている | ②生成AI | 仕様書とAIが突合できる。最終的な要件解釈は人間が確認 |
| 例外・エラーケースが設計に含まれており網羅されているか | ④チーム | 何が「十分な網羅」かはチームの判断が必要（AIは観点の抜けを補助提示） |
| 処理仕様（正常/異常フロー・分岐・入出力）が明確に記述されている | ②生成AI | 明確に書けているかをAIが確認できる |
| データ設計（テーブル/型/桁/CRUD条件）が明確に記述されている | ②生成AI | 記述の明確さをAIが確認できる |
| 例外処理（各エラー時処理・戻り値/ログ・リトライ方針）が明確 | ②生成AI | 記述の明確さをAIが確認できる |
| ログ出力方針が明確である | ②生成AI | 記述の有無・明確さをAIが確認できる |
| インターフェースの引数・戻り値の型が明確に定義されている | ②生成AI | 定義の有無をAIが確認できる |
| インターフェースの副作用の有無が明確になっている | ②生成AI | 明示されているかをAIが確認できる |
| インターフェースのエラー型・例外が定義されている | ②生成AI | 定義の有無をAIが確認できる |
| 依存オブジェクトがDIできる構造になっている | ②生成AI | 構造の有無を設計書からAIが確認できる |
| 外部依存がインターフェースで抽象化されている | ②生成AI | 抽象化の有無を設計書からAIが確認できる |
| 責務分担（クラス/関数/モジュールの責務分割・共通化）が適切か | ④チーム | 分割/統合は設計判断。過不足の評価が要る |
| 境界条件（0件/NULL/最大値/日付境界/重複）の網羅性 | ④チーム | 何が十分かはリスク判断 |
| 状態管理・排他制御（遷移/二重実行/同時更新）の妥当性 | ④チーム | トレードオフを含む設計判断 |
| 非機能（性能・セキュリティ・保守性）設計の妥当性 | ④チーム | 要件充足の妥当性判断が必要 |
| 新規技術の事前検証・代替案・撤退条件（採用時） | ②→④ | AIが情報を一次整理 → 採否はチームのリスク判断で合意 |

### 3.2 DDD観点（DDDを採用する場合）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| ユビキタス言語が設計書・仕様書・コードで一致している | ②生成AI | 用語の一致はAIが照合できる |
| VO: 属性の同一性で等価判定している | ②生成AI | VOのルールとして明確。設計書からAIが意味的に確認できる |
| VO: 不変（Immutable）として設計されている | ②生成AI | 状態変更の有無を設計書からAIが確認できる |
| VO: 自身のバリデーションロジックを持っている | ②生成AI | 持っているかどうかをAIが確認できる |
| VO: プリミティブ型のVO化判断 | ④チーム | 何をVOにするかは設計判断。過剰なVO化もアンチパターン（AIはVO化候補を提示） |
| Entity: 一意識別子（ID）を持っている | ②生成AI | IDの有無をAIが確認できる |
| Entity: ライフサイクルの定義の完全性 | ④チーム | 「どこまで定義すれば十分か」は判断が必要 |
| Aggregate: Rootが定義されている | ②生成AI | 定義の有無をAIが確認できる |
| Aggregate: Rootを通じてのみ内部オブジェクトにアクセスする | ②生成AI | アクセスパスを設計書からAIが確認できる |
| Aggregate: 1トランザクションの粒度が適切か | ④チーム | DDDで最も議論になる設計判断。正解が一意に決まらない |
| Aggregate: 他AggregateへはIDのみで参照している | ②生成AI | 参照方法を設計書からAIが確認できる |
| Domain Service: EntityやVOに置けないロジックのみか | ④チーム | 「自然に配置できない」の判断はDDD理解の深さによって差が出る |
| Domain Service: ステートレスである | ②生成AI | 状態を持つかどうかをAIが確認できる |
| Domain Service: Infrastructure層に依存していない | ②生成AI | 依存関係を設計書からAIが確認できる |
| レイヤ: ドメイン層がApplication・Infrastructure層に依存していない | ②生成AI | 設計段階はAIが確認。実装後は①ツール（eslint-plugin-boundaries 等）で自動化 |
| レイヤ: Repositoryインターフェースはドメイン層に定義されている | ②生成AI | 定義場所を設計書からAIが確認できる |
| レイヤ: Application Serviceにビジネスロジックが混入していないか | ④チーム | 「ビジネスロジック」の定義が人によって変わりやすく、議論が必要 |
| レイヤ: PresentationはDTOを介してドメインオブジェクトを直接公開していない | ②生成AI | DTOを介しているかをAIが確認できる |
| メソッド名がユビキタス言語として適切か | ②→④ | AIが用語集と照合して一次評価 → 命名の妥当性はチームで最終合意 |
| 必要なドメインイベントが定義されているか | ④チーム | どこでイベントが「必要か」は設計判断。見落としが多い観点（AIは候補を提示） |
| ドメイン層がフレームワークなしで単体テストできる | ②生成AI | フレームワーク依存の有無をAIが確認できる |

### 3.3 インフラ・IaC観点（AWS CDK 等を含む場合）

| チェック項目 | 分類 | 理由 |
|---|---|---|
| リソースのデフォルト値（暗号化・公開設定・削除保護・保持期間 等）を把握し意図的に設定している | ②生成AI | 設計書/CDKコードからAIがデフォルトとの差を指摘できる。cdk-nag 等の①ツールも併用 |
| IAMが最小権限になっている（ワイルドカードの濫用がない） | ①ツール＋②生成AI | cdk-nag 等で機械検出。残りはAIが補完 |
| 環境差分（dev/stg/prod）がハードコードされず外出しされている | ②生成AI | ハードコードの有無をAIが検出できる |
| ネットワークの公開範囲が要件どおりか | ④チーム | 「要件どおり」の妥当性・リスク判断が必要（AIは公開設定を検出して提示） |
| 命名規約・タグ付けの方針に沿っている | ①ツール／②生成AI | 規約はルール化でき、ツールまたはAIで照合できる |

### 3.4 HLF観点（Hyperledger Fabric を含む場合）

> HLF観点の軸0分類は、現時点では [HLF観点モジュール](../設計別観点/HLF観点.md) の各チェックリストに付した暫定区分（提出前＝②③／チーム＝④）を正とする。プロジェクトのネットワーク要件が固まった段階で本§3.4へ確定分類を転記する。

## 4. 他工程への適用方針

コードレビュー・テストレビューでも同じ4層の判断基準で仕分けを行う（[コードレビュー観点仕分け方針](../コードレビュー/コードレビュー観点仕分け方針.md) / [テストレビュー観点仕分け方針](../テストレビュー/テストレビュー観点仕分け方針.md)）。工程による違いは「①ツールで自動化できる範囲」に表れる。

- **コードレビュー**: import依存方向・Lint・型などツールで強制できる項目が多く、**①ツール自動化** の比率が高い。
- **詳細設計・テスト**: 成果物が設計書・テストコードで、ツール検出は限定的なため、ルールベース項目は **②生成AIレビュー** が主に担う。

**原則**: ①ツールで自動化できるものは自己チェックにも含めずCI/CDで強制し、②生成AIで一次レビューできるものは提出前に通す。人間（③④）は判断・合意の要る項目に集中する。
