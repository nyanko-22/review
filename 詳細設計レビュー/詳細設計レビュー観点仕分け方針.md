# 詳細設計レビュー 観点仕分け方針

> 本資料は [ソフトウェアレビュー標準 §1.4](../ソフトウェアレビュー標準.md) の「軸0: 確認主体（事前自己チェック / チームレビュー）」を、詳細設計レビューの各チェック項目へ適用した仕分け結果である。仕分け対象の観点A〜Eの詳細は [詳細設計レビューガイドライン](詳細設計レビューガイドライン.md) を参照。

## 1. なぜ仕分けをするのか（背景と目的）

レビュー指摘事項が多くなると、以下のサイクルが発生してレビュー時間が膨大になる。

```
指摘 → 修正 → レビュアーが再確認 → 追加指摘 → ...（繰り返し）
```

特に「ルールとして明確に決まっているのに守られていない」類の指摘は、チームレビューの場で議論する必要がない。これをチームレビューの前に提出者本人が自己確認することで、チームレビューに入る段階での品質を保証し、チームが本当に議論すべき設計判断に集中できるようにする。

**目標状態**:  
チームレビューは「設計上の判断・トレードオフ・合意形成」にのみ使われ、ルール違反の指摘に時間を取られない状態。

---

## 2. 仕分けの考え方（判断基準）

各チェック項目を以下2つの問いで判定する。

### 問い①: 答えが一意に決まるか？

> 「このチェックをした人が10人いたとき、全員が同じ判断をするか？」

- **Yes（答えが一意）** → 事前自己チェック候補
- **No（人によって判断が変わる）** → チームレビュー候補

### 問い②: 提出者本人が単独で確認できるか？

> 「設計書と仕様書・DDDルールだけを見れば、他者の意見なしに確認できるか？」

- **Yes（単独確認可能）** → 事前自己チェック
- **No（他者視点や議論が必要）** → チームレビュー

### まとめ

| 分類 | 性質 | 例 |
|---|---|---|
| **事前自己チェック** | ルールベース・客観的・答えが一意 | IDを持っているか、依存方向が正しいか |
| **チームレビュー** | 設計判断・解釈・トレードオフあり | Aggregate粒度が適切か、命名がユビキタス言語として妥当か |

## 3. 仕分け結果サマリー

### 観点A: 要件との整合性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 機能要件の全ユースケースが設計に反映されている | 事前自己チェック | 仕様書と照合すれば客観的に確認できる |
| ユビキタス言語が設計書・仕様書で一致している | 事前自己チェック | 用語の一致は機械的に比較できる |
| 例外・エラーケースが設計に含まれており網羅されているか | チームレビュー | 何が「十分な網羅」かはチームの判断が必要。一人では見落としが出やすい |

### 観点B: ドメインモデルの正確性

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 属性の同一性で等価判定している | 事前自己チェック | Value Objectのルールとして明確に定義されている |
| 不変（Immutable）として設計されている | 事前自己チェック | 設計書で状態変更の有無を確認できる |
| 自身のバリデーションロジックを持っている | 事前自己チェック | 持っているかどうかは客観的に確認できる |
| プリミティブ型のVO化判断 | チームレビュー | 何をVOにするかは設計判断。過剰なVO化もアンチパターンになるためトレードオフがある |
| 一意識別子（ID）を持っている | 事前自己チェック | IDの有無は客観的に確認できる |
| ライフサイクルの定義の完全性 | チームレビュー | 「どこまで定義すれば十分か」は判断が必要 |
| Aggregate Rootが定義されている | 事前自己チェック | 定義の有無は客観的に確認できる |
| Rootを通じてのみ内部オブジェクトにアクセスする | 事前自己チェック | アクセスパスは設計書から確認できる |
| 1トランザクションの粒度が適切か | チームレビュー | DDDで最も議論になる設計判断。正解が一意に決まらない |
| 他AggregateへはIDのみで参照している | 事前自己チェック | 参照方法は設計書から客観的に確認できる |
| EntityやVOに置けないロジックのみがDomain Serviceにあるか | チームレビュー | 「自然に配置できない」の判断はDDD理解の深さによって差が出る |
| Domain Serviceがステートレスである | 事前自己チェック | 状態を持つかどうかは客観的に確認できる |
| Domain ServiceがInfrastructure層に依存していない | 事前自己チェック | 依存関係は設計書から確認できる |

### 観点C: レイヤー構造・依存方向

| チェック項目 | 分類 | 理由 |
|---|---|---|
| ドメイン層がApplication・Infrastructure層に依存していない | 事前自己チェック | 依存方向のルールは明確で客観的に確認できる |
| Repositoryインターフェースはドメイン層に定義されている | 事前自己チェック | 定義場所は設計書から確認できる |
| Application Serviceにビジネスロジックが混入していないか | チームレビュー | 「ビジネスロジック」の定義が人によって変わりやすく、議論が必要 |
| PresentationはDTOを介してドメインオブジェクトを直接公開していない | 事前自己チェック | DTOを介しているかどうかは設計書から客観的に確認できる |

### 観点D: インターフェース設計

| チェック項目 | 分類 | 理由 |
|---|---|---|
| メソッド名がユビキタス言語として適切か | チームレビュー | 命名の妥当性は主観が入る。チームで合意することで統一感が生まれる |
| 引数・戻り値の型が明確に定義されている | 事前自己チェック | 定義の有無は客観的に確認できる |
| 副作用の有無が明確になっている | 事前自己チェック | 明示されているかどうかは確認できる |
| 必要なドメインイベントが定義されているか | チームレビュー | どこでイベントが「必要か」は設計判断。見落としが多い観点のため複数の目で確認する |
| エラー型・例外が定義されている | 事前自己チェック | 定義の有無は客観的に確認できる |

### 観点E: テスタビリティ

| チェック項目 | 分類 | 理由 |
|---|---|---|
| 依存オブジェクトがDIできる構造になっている | 事前自己チェック | 構造の有無は設計書から確認できる |
| 外部依存がインターフェースで抽象化されている | 事前自己チェック | 抽象化の有無は設計書から確認できる |
| ドメイン層がフレームワークなしで単体テストできる | 事前自己チェック | フレームワーク依存の有無は確認できる |

> **観点Eは全項目が事前自己チェック**: テスタビリティに関する観点は、いずれも設計書を見れば客観的に確認できるルールベースの項目であるため。

## 4. 他工程への適用方針

コードレビュー・テストレビューでも同じ判断基準（問い①②）で仕分けを行う（[コードレビュー観点仕分け方針](../コードレビュー/コードレビュー観点仕分け方針.md) / [テストレビュー観点仕分け方針](../テストレビュー/テストレビュー観点仕分け方針.md)）。

**コードレビューで仕分けの参考になる観点例**:
- 事前自己チェック候補: Lintエラー、型エラー、命名規則違反（ツールで検出できるもの）
- チームレビュー候補: 設計の意図との整合性、パフォーマンスのトレードオフ、抽象化の粒度

**原則**: ツールや静的解析で自動化できるものは自己チェックにも含めず、CI/CDで自動検出する方向を優先する。
